## Проект
Lind (ADND) — AI-Powered Tabletop RPG Engine. AI-Мастер (Экс-Ис) ведёт ролевую игру через веб-чат. Бэкенд FastAPI, фронтенд Vanilla JS, база SQLite + FTS5, DeepSeek API.

## Зачем это нужно
У проекта два постоянных направления работы, плюс сквозная верификация:
- **Архитектор** — System Prompt Экс-Ис, игровая механика, защита от взлома, дизайн мира
- **Инженер** — Код (FastAPI, SQLite, фронтенд), деплой, инфраструктура
- **Критик** — Верификация, поиск противоречий, регресс-тесты

Логическая цепочка: Архитектор → проектирует правила → Инженер → реализует в коде → Критик → проверяет согласованность.

Сейчас в проекте есть Pipeline (Phase 1-5) для одиночных задач, но нет системы переключения между треками и отслеживания зависимостей.

### 3. `project_memory/workflows/switch-track.md`
Протокол переключения между треками (5 шагов):

```markdown
# Протокол переключения трека

Исполнять при КАЖДОМ переключении между треками (Архитектор ↔ Инженер ↔ Критик).

## Шаг 1: Сохранить контекст текущего трека
- Открыть `project_memory/tracks.md`
- В секции текущего трека обновить: Статус, Последнее действие (что сделано, краткий итог)
- Если задача завершена — обновить статус на ✅ и указать результат

## Шаг 2: Проверить карту зависимостей
- Открыть `project_memory/dependency-map.md`
- Если текущий трек блокирует целевой — проверить, можно ли снять блокировку
- Если целевой трек зависит от текущего — убедиться, что все необходимые артефакты готовы
- Если текущий трек (Архитектор) изменил логику — целевой трек (Инженер) должен об этом знать

## Шаг 3: Верификация перед уходом (Quality Gate)
- Если после последнего действия не выполнялась верификация — предложить выполнить Слой 1
- Если счётчик задач в Критике ≥ 10 — предложить ручную проверку (Слой 3)
- Если были значимые изменения в Промтах — напомнить о Слое 2 (LLM-критик)

## Шаг 4: Загрузить контекст целевого трека
- Прочитать текущую задачу целевого трека из `project_memory/tracks.md`
- Если есть план (ссылка в tracks.md) — прочитать его
- Если задача новая — начать стандартный Pipeline (Phase 1: Analysis)

## Шаг 5: Обновить дашборд
- Установить `Активный трек` = целевой трек
- Записать `Последнее переключение` = текущая дата/время (UTC)
- В секции целевого трека обновить `Статус` = 🔄 в работе

## После переключения
- Продолжить работу по стандартному Pipeline (Phase 1-5) для задачи в целевом треке
```

### 4. `project_memory/workflows/verification.md`
3-слойный протокол верификации (Критик):

```markdown
# Протокол верификации (роль: Критик)

Верификация — сквозная функция. Исполняется в двух режимах:
1. **Встроенная** — как шаг в конце каждой задачи (Слой 1 обязателен)
2. **Фоновая** — как отдельный трек Критик (Слой 2 и 3 по триггерам)

---

## Слой 1: Автоматическая проверка (ОБЯЗАТЕЛЕН после каждого изменения)

### Для трека Инженер (код)
- `pytest tests/` — все тесты должны проходить
- `ruff check backend/` — линтер без ошибок (или `flake8` если ruff не настроен)
- Если тестов нет — создать минимальный smoke-тест
- **Результат**: pass/fail. Fail → возврат на доработку

### Для трека Архитектор (промты/знания)
- `scripts/validate_facts.py` — валидация фактов (если скрипт существует)
- Проверка структуры Markdown: заголовки, ссылки не битые
- **Результат**: pass/fail. Fail → возврат на доработку

---

## Слой 2: LLM-критик (после ЗНАЧИМЫХ изменений в Промтах)

### Триггеры
- Изменён `adnd-knowledge/system-prompts/exis-master.md`
- Изменён `adnd-knowledge/game-design/README.md`
- Добавлен новый раздел в `adnd-knowledge/`

### Процедура
1. Взять изменённый промт/документ
2. Подготовить 3 тестовых сценария (из game-loop или придумать edge-case)
3. Отправить LLM запрос в режиме критика:
   ```
   Ты — QA-инженер. Найди логические дыры, противоречия и уязвимости в этом System Prompt.
   Промт: [изменённый документ]
   Сценарии: [3 сценария]
   Вопросы:
   1. Есть ли логические дыры?
   2. Противоречит ли промт game-design/?
   3. Можно ли взломать через социальную инженерию?
   4. Все ли краевые случаи покрыты?
   ```
4. **Результат**: отчёт Критика → сохранить в `adnd-knowledge/{раздел}/_audit.json`

### Формат записи в _audit.json
```json
{
  "date": "2026-05-16T09:00:00Z",
  "track": "Архитектор",
  "layer": 2,
  "section": "system-prompts",
  "trigger": "изменение exis-master.md",
  "result": "pass|fail|warnings",
  "findings": ["описание проблемы 1", "описание проблемы 2"],
  "action": "что исправлено или требуется исправить"
}
```

---

## Слой 3: Ручная проверка (каждая 10-я задача)

### Триггер
- Счётчик завершённых задач (Архитектор + Инженер) достигает 10
- Счётчик ведётся в `project_memory/tracks.md` → секция Критик → `Счётчик задач`

### Процедура
1. Cline выбирает случайную завершённую задачу из последних 10
2. Формирует краткий отчёт: что было сделано, какие файлы изменены
3. Запрашивает пользователя: «Проверь результат задачи X. Вот что изменилось: [список]. Всё корректно?»
4. Пользователь отвечает: pass / fail / правки
5. **Результат**: запись в `adnd-knowledge/{раздел}/_audit.json`

### Формат записи
```json
{
  "date": "2026-05-16T09:00:00Z",
  "track": "Инженер",
  "layer": 3,
  "task": "название задачи",
  "result": "pass|fail|fixes_required",
  "reviewer": "exis (human)",
  "notes": "комментарий"
}
```
