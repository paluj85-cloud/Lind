# План: Наполнение adnd-knowledge/ из chat_deepseek/

**Дата:** 2026-05-15
**Метод:** Per-chat summarization + Iterative refinement (2+3)
**Исполнитель:** Cline (все этапы, без Python-скриптов)
**Принцип:** Точность и аккуратность > скорость. Без БД, файловый подход.
**Правило:** Перед каждым шагом проверять что сделано и что изменилось.

---

## Этап 0: Тестовый прогон

**Задача:** Проверить качество на 3 чатах перед полным запуском.

**Выбранные чаты:**
1. **Короткий** (~200 строк): `c723960f-создание-промпта-по-запросу.md`
2. **Средний** (~1000 строк): `8bd28a54-систематизация-логики-веб-игры.md` (481 строка)
3. **Длинный** (>2000 строк): `81fa1819-финальное-руководство-по-разработке-mvp.md` (1351 строка)

**Процесс:**
1. Классифицировать 3 чата
2. Извлечь факты (для длинного — с chunking'ом)
3. Консолидировать в README.md + `_audit.json` + `_source_map.json`
4. Проверить точность: сравнить source_quote с исходными чатами
5. Скорректировать промты Этапов 1-3 по результатам

**Критерий готовности:** ≥95% фактов подтверждены. Промты скорректированы. Можно запускать Этап 1.

## Этап 1: Классификация чатов

**Задача:** Для каждого из 121 чата определить, к какому(им) разделам `adnd-knowledge/` он относится.

**Разделы для классификации:**
1. `vision/` — концепция, миссия, философия проекта
2. `game-design/` — правила, механики, расы, классы, магия
3. `game-loop/` — цикл игры, flow, фазы взаимодействия игрок-мастер
4. `api-spec/` — API эндпоинты, форматы запросов/ответов
5. `frontend/` — UI/UX, вёрстка, клиентская логика
6. `architecture/` — архитектура системы, компоненты, связи
7. `tech-specs/` — технические спецификации (v0.1, v0.2) — уже заполнен
8. `system-prompts/` — системные промты — уже заполнен
9. `decisions/` — принятые решения и их обоснования
10. `status/` — текущий статус проекта
11. `source-map/` — карта исходного кода (будет позже)

**Способ:** Cline читает `chat_deepseek/_index.md`, для каждого чата берёт заголовок + поиск ключевых слов по всему файлу (НЕ только первые 200 строк — ключевые решения часто в конце). Вызывает DeepSeek API (flash, через `curl`) с промтом классификации. Результат: `chat_deepseek/_classification.json`.

**Критерий готовности:** Каждый чат привязан минимум к одному разделу. Спорные чаты помечены для ручной проверки.

## Этап 2: Извлечение фактов (per-chat, все разделы за один запрос)

**Задача:** Для каждого чата извлечь ВСЕ факты сразу по всем разделам. Один запрос к DeepSeek на чат → все разделы.

**Почему так:** Не 5 запросов на чат (по разделу), а 1. Экономия ~400 API-запросов.

**Обработка длинных чатов:** Для чатов > 1500 строк — chunking:
- Разбивка на чанки по 1500 строк с перекрытием 200 строк (чтобы не терять контекст на границах)
- Извлечение фактов из каждого чанка отдельно
- Объединение фактов от всех чанков, дедупликация

**Процесс:**
1. Взять чат
2. Если > 1500 строк: разбить на чанки
3. Подать полный текст (или чанк) + промт извлечения → LLM возвращает JSON со всеми разделами
4. Cline раскладывает факты по `_audit.json` каждого раздела
5. Параллельно формируется `_source_map.json`

**Формат ответа LLM (один чат → все разделы):**
```json
{
  "source_chat": "chat_N",
  "source_date": "2026-05-15",
  "sections": {
    "vision": [
      { "fact": "...", "confidence": "high|medium|low", "source_quote": "...", "topic": "..." }
    ],
    "game-design": [...],
    "game-loop": [...],
    "api-spec": [...],
    "frontend": [...],
    "architecture": [...],
    "decisions": [...],
    "status": [...]
  }
}
```
Если раздел пуст — пустой массив `[]`.

**Формат `_audit.json` (полный трейс):**
```json
{
  "section": "game-design",
  "generated": "2026-05-15",
  "facts": [
    {
      "id": "gd-001",
      "fact": "Магия вербальная, без слотов и маны",
      "confidence": "high",
      "source_chat": "chat_42",
      "source_date": "2026-05-11",
      "source_quote": "магия у нас будет чисто вербальной — игрок говорит что делает, и это происходит",
      "topic": "магия"
    }
  ]
}
```

**Формат `_source_map.json` (быстрая навигация):**
```json
{
  "section": "game-design",
  "map": {
    "gd-001": {"chat": "chat_42", "line": 120},
    "gd-002": {"chat": "chat_15", "line": 340}
  }
}
```

**Критерий готовности:** Для каждого раздела (кроме уже заполненных) собран `_audit.json` + `_source_map.json` со всеми фактами.

## Этап 3: Консолидация (per-section)

**Задача:** Собрать все извлечённые факты раздела в связный Markdown-документ.

**Процесс на раздел:**
1. Взять `_audit.json` раздела
2. **Отсортировать факты по дате чата** (source_date) — чтобы LLM видела эволюцию и разрешала противоречия (позднее > раннее)
3. Сгруппировать факты по подтемам (topic)
4. Устранить дубликаты (одинаковые факты из разных чатов)
5. Подать сгруппированные факты LLM с промтом: «Составь связный документ для раздела X на основе этих фактов»
6. LLM возвращает готовый Markdown
7. Сохранить: `adnd-knowledge/{раздел}/README.md`

**Критерий готовности:** README.md каждого раздела заполнен связным текстом. README.md чистый — без source trace внутри фактов. Source trace вынесен в отдельные файлы: `_audit.json` (полный трейс), `_source_map.json` (быстрая навигация).

**Структура файлов раздела после Этапа 3:**
```
adnd-knowledge/{раздел}/
├── README.md              ← Чистые знания для LLM (AI-мастер)
├── _audit.json            ← Полный аудит-трейл: факт → source_chat → source_quote → confidence
└── _source_map.json       ← Карта для Cline: факт → [чат, строка] для быстрой навигации
```

**README.md — чистый, без source trace в теле:**
```markdown
> **Статус**: 🔄 В процессе
> **Связан с**: [game-loop](../game-loop/), [api-spec](../api-spec/)
> **Источник**: 42 чата из chat_deepseek/
> **Аудит**: [_audit.json](./_audit.json) | **Карта**: [_source_map.json](./_source_map.json)

- Факт 1 — конкретно, без воды
- Факт 2

> 💡 **Ключевое решение**: магия вербальная...
```

Никаких `[→ chat_N]` в README.md. LLM читает чистые знания. Cline для валидации идёт в `_audit.json` и `_source_map.json`.

## Этап 4: Валидация

**Задача:** Выборочная проверка точности извлечённых знаний.

**Процесс:**
1. Выбрать 10% случайных фактов из каждого раздела (минимум 3)
2. Для каждого: найти исходный чат, проверить цитату
3. При расхождении: пометить, исправить
4. При системных ошибках: вернуться на Этап 2 с уточнённым промтом

**Критерий готовности:** ≥95% проверенных фактов подтверждены исходными чатами.

## LLM-Friendly формат выходных файлов (README.md)

Основан на лучших практиках Anthropic, OpenAI, RAG-систем.

**Принцип:** файл — не книга для человека, а **инструкция для AI**. Каждое предложение — факт или правило.

**Структура каждого README.md:**
```markdown
# Название раздела (H1)

> **Статус**: ✅ Стабильно | 🔄 В процессе | ⏳ TBD
> **Связан с**: [раздел1](../раздел1/), [раздел2](../раздел2/)
> **Источник**: N чатов из chat_deepseek/
> **Аудит**: [_audit.json](./_audit.json) | **Карта**: [_source_map.json](./_source_map.json)

Краткий абзац — суть раздела. Не более 3-4 предложений.

## Подтема (H2)

Абзац (2-3 предложения). Только факты.

- Факт 1 — конкретно, без воды
- Факт 2
- Факт 3

> 💡 **Ключевое решение**: цитата или вывод

### Детализация (H3 — опционально)

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| ... | ... | ... |
```

**Правила LLM-Friendly формата:**
1. Иерархия H1→H2→H3 даёт семантическую карту документа
2. Метаданные (статус, связи, источник, аудит, карта) — первое что читает LLM
3. Каждое предложение — факт или правило. **Без нарратива, без воды.**
4. Списки (`-`) для перечислений; таблицы для сравнений
5. Цитаты (`> 💡`) для ключевых решений — визуально выделены
6. Перекрёстные ссылки на связанные разделы — обязательны
7. Ссылки `Аудит/Карта` — для Cline, не для LLM

**Плотность фактов (главный критерий):**

| ❌ Плохо (нарратив) | ✅ Хорошо (инструкция для AI) |
|---------------------|-------------------------------|
| «В рамках разработки мы обсуждали различные подходы к магии и в итоге пришли к выводу, что лучше всего использовать систему...» | **Магия:** вербальная система. Игрок говорит → AI интерпретирует. Без слотов. Без маны. |

## Промт для Этапа 1: Классификация чатов

```markdown
<role>
Ты — классификатор знаний. Твоя задача: определить, к каким разделам базы знаний
относится содержимое чата. Ничего не извлекай — только классифицируй.
</role>

<instructions>
1. Прочитай заголовок чата и key_points (если есть)
2. Определи, какие разделы затрагиваются в этом чате
3. Верни JSON с массивом разделов и confidence для каждого
</instructions>

<format>
{ "chat_id": "chat_N", "sections": { "vision": "high|medium|low|none", "game-design": "...", ... } }
</format>

<constraints>
- ТОЛЬКО валидный JSON
- high: чат напрямую обсуждает раздел
- medium: чат косвенно касается раздела
- low: чат упоминает раздел вскользь
- none: раздел не затрагивается
- Не выдумывай связи, которых нет в чате
</constraints>
```

## Промт для Этапа 2: Извлечение фактов (один чат → все разделы)

```markdown
<role>
Ты — аналитик-извлекатель знаний. Твоя задача: прочитать чат и извлечь ВСЕ факты
по ВСЕМ разделам в строгом JSON-формате. Один запрос → полный охват.
</role>

<instructions>
1. Прочитай чат целиком
2. Для КАЖДОГО раздела из списка найди все утверждения, решения, идеи, правила, спецификации
3. Для каждого факта укажи:
   - fact: точная формулировка (1-2 предложения, на русском)
   - confidence: high (прямое утверждение) | medium (вывод из контекста) | low (намёк)
   - source_quote: ДОСЛОВНАЯ цитата из чата (не перефразируй)
   - topic: подтема внутри раздела (коротко, на русском)
4. Пропускай: общие рассуждения, шутки, повторения, несвязанные темы
5. Если фактов по разделу нет — верни ПУСТОЙ массив []
6. Если чат разбит на чанки — обработай каждый чанк независимо (повторы допустимы, дедупликация позже)

Разделы:
- vision — концепция, миссия, философия проекта
- game-design — правила, механики, расы, классы, магия
- game-loop — цикл игры, flow, фазы взаимодействия игрок-мастер, процесс хода
- api-spec — API эндпоинты, форматы запросов/ответов, протоколы
- frontend — UI/UX, вёрстка, клиентская логика, дизайн
- architecture — архитектура системы, компоненты, связи, модули
- decisions — принятые решения, их обоснования, отменённые варианты
- status — текущий статус проекта, прогресс, roadmap
</instructions>

<format>
{
  "source_chat": "chat_N",
  "source_date": "YYYY-MM-DD",
  "sections": {
    "vision": [
      { "fact": "...", "confidence": "high|medium|low", "source_quote": "...", "topic": "..." }
    ],
    "game-design": [],
    "game-loop": [],
    "api-spec": [],
    "frontend": [],
    "architecture": [],
    "decisions": [],
    "status": []
  }
}
</format>

<constraints>
- ТОЛЬКО валидный JSON. Никаких пояснений вне JSON.
- source_quote — дословно из чата, не перефразируй
- Не выдумывай факты, которых нет в чате
- confidence=high только если есть прямое утверждение, а не интерпретация
- Пустые разделы — пустой массив, НЕ omit
</constraints>
```

## Промт для Этапа 3: Консолидация в README.md

```markdown
<role>
Ты — технический писатель. Твоя задача: собрать разрозненные факты в связный
Markdown-документ для AI-мастера настольной RPG-игры.
</role>

<instructions>
1. Факты отсортированы по дате чата — используй это для разрешения противоречий:
   - Если факты противоречат друг другу — более поздний (по дате) побеждает
   - Если факт был отменён в более позднем чате — включи только финальное решение
2. Сгруппируй факты по подтемам (topic)
3. Для каждой подтемы напиши:
   - Краткий абзац (2-3 предложения) — суть подтемы
   - Список ключевых правил/фактов (bulleted list)
4. Устрани дубликаты (одинаковые факты из разных чатов)
5. Если фактов по подтеме мало (< 3) — объедини с соседней подтемой
6. Добавь: заголовки, метаданные в начале файла, перекрёстные ссылки
</instructions>

<format>
Markdown-файл строгой структуры (см. LLM-Friendly формат выше):
- H1 с названием раздела
- Блок метаданных (статус, связи, источник, аудит, карта)
- H2 для каждой подтемы
- Абзац + маркированный список фактов
- Цитаты (>) для ключевых решений
- H3 для таблиц и детализации (опционально)
</format>

<constraints>
- Статус всегда: 🔄 В процессе (раздел будет дополняться)
- Каждое предложение — факт или правило. Без воды и нарратива.
- Перекрёстные ссылки на связанные разделы — обязательны
- Не теряй source_quote — важные цитаты оформляй как > 💡
- Если фактов по разделу > 50 — разбей на подфайлы (H2 → отдельный .md)
- Никаких ссылок на чаты в README.md — source trace только в _audit.json
</constraints>
```

## Appendix: Лучшие практики (учтены в плане)

| Источник | Практика | Где учтено |
|----------|----------|------------|
| **Lost in the Middle** (Liu, 2024) | Середина длинных документов извлекается хуже | Chunking для чатов > 1500 строк, перекрытие 200 строк |
| **Chain-of-Density** (Adams, 2023) | Итеративное уплотнение суммаризации | Этап 2 (извлечение) → Этап 3 (консолидация) = два прохода с нарастающей плотностью |
| **Anthropic RAG** (2024) | Контекстные префиксы к чанкам | Каждый чанк включает перекрытие с предыдущим (200 строк контекста) |
| **GraphRAG** (Microsoft, 2024) | Глобальная кластеризация + локальное извлечение | Этап 1 (глобальная классификация) → Этап 2 (per-chat локальное извлечение) |
| **RAPTOR** (2024) | Рекурсивная кластеризация | Структура H1→H2→H3 в README.md = семантическая иерархия |
| **Anthropic Prompt Engineering** (2024) | Разделение роли, инструкций, формата, ограничений | Промты используют теги <role>, <instructions>, <format>, <constraints> |
| **Self-Consistency** (Wang, 2023) | Множественные проходы для сложных решений | Валидация 10% фактов на Этапе 4 |
