## Варианты выполнения (2-3, с pros/cons)

### Вариант A: Cline ручной (как Этап 0, но в 12 чекпоинтов)

**Как работает:** Cline сам читает чаты, извлекает факты, валидирует. По 10 чатов за итерацию. После каждой партии — чекпоинт (аудит 15% фактов). При падении точности — корректировка промта и повтор партии.

**Плюсы:**
- Полный контроль на каждом шаге
- Можно адаптировать промты на лету
- Никакой дополнительной инфраструктуры

**Минусы:**
- 12 итераций ≈ много шагов
- Контекст Cline может переполняться (128K токенов)
- Долго (каждый чекпоинт требует ручного участия в валидации)

**Оценка времени:** 6–8 часов чистого времени

---

### Вариант B: Python-скрипт + DeepSeek API (автоматический)

**Как работает:** Пишем Python-скрипт, который:
1. Читает все 121 чат
2. Батчами по 10 вызывает DeepSeek API (JSON mode) для классификации + извлечения
3. Автоматически валидирует 15% фактов через grep source_quote
4. При падении точности — корректирует промт и повторяет партию
5. Консолидирует факты хронологически

**Плюсы:**
- Быстро (без ручного участия между чекпоинтами)
- Надёжно (скрипт не забудет шаг)
- Можно поставить на ночь

**Минусы:**
- Нужно написать скрипт (~200 строк Python)
- Менее гибко при неожиданных ошибках формата
- Требует API-ключ в `.env`

**Оценка времени:** 2 часа на написание скрипта + 2 часа на прогон

---

### Вариант C: Cline как оркестратор + скрипт-помощник (гибрид)

**Как работает:** Cline управляет процессом (решает что делать дальше), но для тяжёлых операций вызывает Python-скрипты:
- `scripts/classify_chats.py` — классификация пачки чатов через API
- `scripts/extract_facts.py` — извлечение фактов из пачки чатов через API
- `scripts/validate_facts.py` — валидация фактов grep'ом
- `scripts/consolidate.py` — консолидация раздела

Cline: принимает решения о корректировке промтов, запускает скрипты, проверяет результаты.

**Плюсы:**
- Скорость скриптов + гибкость Cline
- Скрипты переиспользуемы для будущих обновлений базы знаний
- Чекпоинты автоматизированы

**Минусы:**
- Сложнее в настройке (3–4 скрипта)
- Больше точек отказа (скрипт + Cline + API)

**Оценка времени:** 3 часа на скрипты + 1.5 часа на прогон

## Архитектура процесса (единая для всех вариантов)

```
Этап 1: Классификация 121 чата
    ├── Скрипт: батчами по 10 → _classification.json
    └── Результат: каждый чат привязан к 8 разделам

Этап 2: Извлечение фактов (партии по 10 + чекпоинты)
    ├── Партия 1  (чаты 1-10)   → извлечь → валидация 15% ✓ → едем
    ├── Партия 2  (чаты 11-20)  → извлечь → валидация 15% ✓ → едем
    ├── ...
    └── Партия 13 (чаты 111-121) → извлечь → валидация 15% ✓

Этап 3: Глобальная консолидация
    ├── Сортировка по дате (позднее побеждает)
    ├── Устранение дубликатов
    ├── Разрешение противоречий
    └── Запись README.md в каждый раздел

Этап 4: Финальная валидация (10% фактов)
    └── ≥95% → готово. <95% → возврат на Этап 2.

Этап 5: Документирование
    ├── CHANGELOG
    ├── progress.log
    └── git push
```

## Структура выходных файлов (на раздел)

```
adnd-knowledge/{раздел}/
├── README.md          ← Чистые знания для AI-мастера (H1→H2→H3, без source trace)
├── _audit.json        ← Полный аудит-трейл (факт → source_quote → confidence)
└── _source_map.json   ← Карта: fact_id → [chat_id, line_range]
```

## Chunking-правила
- Чат ≤ 2000 строк → один запрос
- Чат > 2000 строк → чанки по 2000 строк, перекрытие 300 строк
- Чатов > 2000 строк: ~15 штук

## Чекпоинт-правила (после каждых 10 чатов)
1. Выбрать 15% случайных фактов из партии (min 5)
2. Для каждого: найти source_quote в исходном чате
3. ≥95% совпадений → партия принята
4. <95% → анализ ошибок → правка промта → переделка партии
