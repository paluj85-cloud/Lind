## 1. Как LLM читает документы

### Проблема «Lost in the Middle» (Liu, 2024)

LLM лучше всего извлекает из **начала** и **конца** документа. Середина — слепое пятно.

**Что делать:**
- Ключевую информацию — в начало (метаданные, суть)
- Для документов > 1500 строк — chunking с перекрытием 200 строк
- Важные решения дублировать в начале (summary) и в конце (итог)

### Контекстное окно ≠ внимание

Даже если всё влезает в контекст (128K токенов), LLM неравномерно распределяет внимание. Не пихай всё в один промт — разбивай на логические блоки.

## 2. LLM-Friendly формат документа

### Принцип: файл — инструкция для AI, не книга для человека

| ❌ Плохо (нарратив) | ✅ Хорошо (инструкция) |
|---------------------|------------------------|
| «В рамках разработки мы обсуждали различные подходы к магии и в итоге пришли к выводу...» | **Магия:** вербальная система. Игрок говорит → AI интерпретирует. Без слотов. Без маны. |

### Структура документа

```markdown
# Заголовок (H1)

> **Статус**: ✅ | 🔄 | ⏳
> **Связан с**: [раздел1](../), [раздел2](../)
> **Источник**: N источников

Краткий абзац — суть (2-3 предложения)

## Подтема (H2)

- Факт 1 — конкретно
- Факт 2 — конкретно

> 💡 **Ключевое решение**: цитата

### Детализация (H3 — опционально)

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| ... | ... | ... |
```

### Правила

1. **Иерархия H1→H2→H3** — даёт LLM семантическую карту
2. **Метаданные в начале** — статус, связи, источник
3. **Каждое предложение — факт.** Без воды и нарратива
4. **Списки** для перечислений, **таблицы** для сравнений
5. **Цитаты** (`>`) для ключевых решений
6. **Перекрёстные ссылки** на связанные разделы
7. **Source trace отдельно** — не засоряй тело документа ссылками на источники

## 3. Промт-инжиниринг

### Структура промта (4 блока)

```
<role>       — кто ты, твоя задача
<task>       — что конкретно сделать
<format>     — строгий формат ответа (JSON, Markdown)
<constraints> — что нельзя, что обязательно
```

### Пример (извлечение знаний)

```markdown
<role>
Ты — аналитик-извлекатель знаний. Читаешь чат и извлекаешь факты.
</role>

<task>
1. Прочитай чат
2. Найди все утверждения по разделам: vision, game-design, api-spec
3. Для каждого: fact, confidence (high/medium/low), source_quote, topic
</task>

<format>
{
  "sections": {
    "vision": [{ "fact": "...", "confidence": "high", "source_quote": "...", "topic": "..." }],
    "game-design": []
  }
}
</format>

<constraints>
- ТОЛЬКО валидный JSON
- source_quote — дословно из чата
- Не выдумывай
- Пустые разделы — []
</constraints>
```

### Почему это работает

- **Роль** активирует нужную «личность» модели
- **Формат** исключает creative writing, заставляет structured output
- **Ограничения** предотвращают галлюцинации

## 4. Chunking для длинных текстов

### Когда применять

- Документ > 1500 строк → разбить на чанки
- Размер чанка: 1500 строк
- Перекрытие: 200 строк (чтобы не терять контекст на границах)

### Пример (чат на 6000 строк)

```
Чанк 1: строки 1–1500
Чанк 2: строки 1300–2800   ← перекрытие 200 строк с чанком 1
Чанк 3: строки 2600–4100
Чанк 4: строки 3900–5400
Чанк 5: строки 5200–6000
```

Каждый чанк обрабатывается независимо → факты объединяются → дедупликация.

## 5. Эволюция знаний

### Проблема

Чат от 10 мая может содержать решение, которое отменено в чате от 14 мая. Если подать факты вперемешку — LLM запутается.

### Решение

**Сортировать факты по дате источника перед консолидацией.** Правило: более позднее побеждает.

```
Факты до сортировки:
- «Магия на слотах» (чат 5, 10 мая)
- «Магия вербальная» (чат 42, 14 мая)
- «Магия смешанная» (чат 8, 12 мая)

После сортировки по дате:
- «Магия на слотах» (10 мая)
- «Магия смешанная» (12 мая)
- «Магия вербальная» (14 мая) ← финал, побеждает
```

LLM видит хронологию и выбирает последнее.

## 6. Итеративное уплотнение (Chain-of-Density)

### Принцип

Не пытайся получить идеальный результат за один проход.

- **Проход 1 (извлечение):** собрать ВСЁ, включая шум
- **Проход 2 (консолидация):** уплотнить, убрать дубликаты, разрешить противоречия

Два прохода дают качество выше чем один «идеальный» промт.

---

## 7. Валидация

### Правило 10%

Проверь минимум 10% фактов (но не менее 3) из каждого сгенерированного документа:

1. Возьми случайный факт
2. Найди исходный чат по source_quote
3. Сравни
4. При расхождении — исправь, при системной ошибке — скорректируй промт

**Цель:** ≥95% фактов подтверждены.

## 8. Работа с API (DeepSeek)

### JSON Mode

```bash
curl https://api.deepseek.com/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $DEEPSEEK_API_KEY" \
  -d '{
    "model": "deepseek-chat",
    "response_format": {"type": "json_object"},
    "messages": [
      {"role": "system", "content": "Ты — анализатор. Возвращай только JSON."},
      {"role": "user", "content": "..."}
    ]
  }'
```

### Важно

- `response_format: json_object` — обязательно для structured output
- System prompt должен явно требовать JSON
- Не забудь `max_tokens` для длинных ответов

## 9. Чек-лист перед отправкой в LLM

- [ ] Документ начинается с метаданных (статус, связи)?
- [ ] Заголовки H1→H2→H3 создают семантическую карту?
- [ ] Каждое предложение — факт (не нарратив)?
- [ ] Ключевые решения выделены цитатами?
- [ ] Есть перекрёстные ссылки на связанные разделы?
- [ ] Source trace вынесен в отдельный файл?
- [ ] Длинный текст разбит на чанки с перекрытием?
- [ ] Промт имеет 4 блока: роль, задача, формат, ограничения?
- [ ] Запрошен строгий JSON/формат?
- [ ] Предусмотрена валидация 10%?
