# Игровой цикл

Игровой цикл Lind (ADND) — это последовательность этапов от подготовки мира до финала, включающая сбор заявок игроков, обработку бросков и обновление игрового мира через LLM.

## Общая структура игрового цикла

- Игровой цикл включает пять основных этапов: подготовка мира (ИИ создаёт Книги Мира и Истории), старт игры (генерация Книги Завязки), вождение (фабрика арок реагирует на действия игроков), мир живёт (NPC и таймеры действуют независимо), финал (генерация романа).
- Игровой цикл начинается с рассылки порядка хода, затем сбор заявок, вызов LLM, обработка бросков, обновление мира и генерация нового порядка.

## Начало игры

- После запуска игры мастер представляется и рассказывает основные правила и вводный сеттинг игры.
- Мастер просит игроков представиться и рассказать о желаемых персонажах.

## Порядок хода

- При старте или после завершения хода сервер извлекает из Neo4j всех активных игроков, перемешивает их случайным образом и рассылает порядок хода.
- Игроки получают порядок хода, пишут заявки и нажимают «Конец хода».

## Сбор заявок

- Каждый игрок через клиент отправляет заявку с полями: `player` (имя), `action_type` (`say` или `do`) и `content` (текст).
- Игрок может отправить заявку без текста — пустая заявка может означать пропуск хода или поддержание предыдущего действия.
- Игроки могут отправлять заявки (действия) или задавать уточняющие вопросы мастеру.

## Обработка заявок сервером

- Сервер собирает заявки, формирует промпт для Экс-Ис и отправляет в LLM.
- Если заявка от игрока пустая, сервер проверяет его предыдущую заявку (сохранённую в истории хода): если прошлое действие было `do` с пометкой «поддержание» и не было прерывания, то текущая заявка автоматически заполняется как `do` с content «продолжает поддерживать ...». Иначе игрок ничего не делает.
- Мастер может после получения заявки или вопроса, прежде чем дать ответ, задать уточняющий вопрос игроку, чтобы лучше понять его намерения.

### Формирование промпта

- Первый вызов Экс-Ис формирует промпт со следующими полями: СЕТТИНГ, ПАРТИЯ, УРОВЕНЬ ОТКРОВЕНИЯ, МИРОВОЕ ВРЕМЯ, ПОРЯДОК ДЕЙСТВИЙ, ЗАЯВКИ ИГРОКОВ, АВАТАРЫ, КОНТЕКСТ ПАМЯТИ, МИРОВОЙ ЛОР, ТАЙНЫЙ СВИТОК, ЯВЛЕННЫЙ СВИТОК, НИТИ СУДЬБЫ, СЕМЕНА БУДУЩЕГО, ТЕКУЩИЕ СУЩНОСТИ, РЕЗУЛЬТАТЫ БРОСКОВ (отсутствует в первом вызове).

### Поток обработки одного хода

- Клиент отправляет сообщение через WebSocket.
- Сервер собирает контекст: последние N реплик + результаты FTS5-поиска.
- Полный промпт отправляется в DeepSeek.
- JSON-ответ парсится: нарратив возвращается клиенту, `world_update` сохраняется в БД.
- При достижении порога несжатых событий запускается фоновая саммаризация.

### Протокол обработки WebSocket

- Получить `player_message`.
- Отправить `master_thinking` (started).
- Собрать контекст.
- Сформировать полный промпт и вызвать DeepSeek `generate()`.
- Распарсить `LLMResponse`.
- Отправить `master_speech` с `narrator`.
- Если есть `world_update` — асинхронно сохранить в БД и проверить порог саммаризации.
- Отправить `master_thinking` (stopped).

## Формат ответа Экс-Ис

- Экс-Ис общается через JSON с полями: `narrator` (текст нарратива), `world_update` (null или объект), `dice_request` (null или объект).

## Обработка бросков

- Сервер получает ответ модели и парсит XML в поисках тегов `<dice_request>`.
- Если тегов нет — ответ считается финальным, сервер переходит к обновлению мира.
- Если теги есть — сервер отправляет запрос броска игроку, получает результат, сравнивает с DC (сложностью) и определяет нарративный статус.

## Обновление мира

- `world_update` заполняется только для значимых событий (type: `event`, `entity_change`, `location_change`, `item_change`) с `narrative_summary` до 100 символов.
- Поля `world_update`: `type` (один из четырёх типов), `details` (словарь), `narrative_summary` (краткое описание).

## Сводка

1. Игровой цикл состоит из пяти крупных этапов: подготовка мира, старт, вождение, жизнь мира и финал.
2. Каждый ход включает рассылку порядка хода, сбор заявок, вызов LLM, обработку бросков и обновление мира.
3. Заявки игроков могут быть действиями (`do`), репликами (`say`) или пустыми (пропуск хода или поддержание предыдущего действия).
4. Ответ Экс-Ис формируется в JSON с нарративом, обновлением мира и запросом броска; броски обрабатываются через теги `<dice_request>`.
5. Обновление мира (`world_update`) фиксирует только значимые события с кратким описанием до 100 символов.
