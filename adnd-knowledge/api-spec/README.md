## Общая архитектура

- Приложение использует **FastAPI** как веб-фреймворк с поддержкой async и WebSocket.
- Для реального времени применяется **WebSocket** для стриминга ответов, а HTTP — для отправки сообщений.
- Все URL административных эндпоинтов начинаются с `/api/admin`.

## REST-эндпоинты

### Проверка здоровья

- **GET /api/health** — возвращает `{"status": "ok", "version": "0.1", "db": "connected"}`. Должен проверять доступность базы данных.

### Управление сессией

- **POST /api/session/start** — создаёт новую сессию. Возвращает `session_id` и статическое приветствие с просьбой назвать имя. Ответ содержит поле `master_says` с текстом приветствия и запросом имени.
- **POST /api/session/name** — устанавливает имя игрока и получает персональное благословение от LLM. Тело запроса: `{"session_id": "uuid", "player_name": "Аэрин"}`. Логика: проверяет, что сессия существует и `name_provided == false`, обновляет запись, вызывает DeepSeek с промтом, возвращает ответ с полем `master_says`.
- **GET /api/session/{session_id}** — возвращает метаданные сессии: `session_id`, `player_name`, `created_at`, `last_active`, `name_provided`.
- **GET /api/session/{session_id}/history** — (предполагаемый) возвращает JSON с историей сообщений диалога. В текущей версии ФТЗ отсутствует.

### Управление книгами (административные)

- **POST /books/upload** — принимает `multipart/form-data` с полем `file`. Валидирует расширение (`.txt`, `.epub`, `.fb2`, `.md`) и размер. Преобразует в plain text, создаёт запись в `arc_book`, сохраняет plain text на диске. Возвращает `{"book_id": 1}`.
- **GET /books/current** — возвращает информацию о текущей загруженной книге (последней по времени) или 404, если ни одной книги не загружено.
- **POST /books/current/start** — запускает или возобновляет анализ для текущей книги. Тело: `{"fragment_size": 8000, "split_by_chapters": true, "extraction_prompt": "..."}`.

### Чат и игровые действия

- **POST /api/chat/send** — эндпоинт для отправки сообщения в чат.
- **POST /api/chat/send** — также используется для отправки заявки в формате JSON: `{"player": "имя", "action_type": "say"|"do", "content": "текст"}`.

## WebSocket

### Подключение и аутентификация

- **WebSocket /ws/chat/{session_id}** — основной эндпоинт для чата.
- Подключение возможно **только после успешного именования** (через POST /api/session/name). Иначе сервер разрывает соединение с кодом 4000 и сообщением «Требуется имя».
- WebSocket хендлер использует `ConnectionManager` для управления соединениями.

### Форматы событий

**События от клиента:**
- `{"action": "player_message", "text": "..."}` — текст до 500 символов.
- `{"player": "имя", "action_type": "say"|"do", "content": "текст"}` — заявка на действие.

**События от сервера:**
- `{"event": "connected", "session_id": "..."}` — успешное подключение.
- `{"event": "master_thinking", "status": "started"}` — начало генерации ответа.
- `{"event": "master_speech", "text": "...", "turn_id": "uuid"}` — речь Мастера.
- `{"event": "master_thinking", "status": "stopped"}` — завершение генерации.
- `{"event": "error", ...}` — ошибка.
- `{"type": "dice_prompt", "reason": "...", "secret": true/false}` — запрос броска кубика.
- `{"type": "turn_order", "order": ["имя1", "имя2", ...]}` — порядок хода.

## Интеграция с DeepSeek API

### Режимы работы

- Интеграция использует **Chat Completions API** (`v1/chat/completions`) с моделью `deepseek-chat`. Поддерживается **JSON mode** для структурированных ответов.
- DeepSeek API поддерживает параметр `thinking` для модели `deepseek-v4-flash`, который может быть включен (`enabled`) или выключен (`disabled`).
- **Режим с размышлениями** (thinking enabled) — модель генерирует цепочку рассуждений, которая возвращается в ответе API как `reasoning_content`. Подходит для сложных логических задач, математики, программирования, многошаговых инструкций. Требует больше времени и токенов.
- **Режим без размышлений** (thinking disabled) — быстрый и экономичный режим для общих вопросов, творчества или простых поручений.

### Обработка ошибок и лимиты

- Обработка ошибок DeepSeek API включает **retry с exponential backoff** и **circuit breaker**.
- Rate limits учитываются с помощью **очереди с регулированием (token bucket)**.

## Сводка

1. **Два протокола взаимодействия**: REST (POST /api/session/start, POST /api/session/name, GET /api/session/{id}, POST /api/chat/send) и WebSocket (/ws/chat/{session_id}) для стриминга ответов в реальном времени.
2. **Обязательное именование перед чатом**: WebSocket-соединение принимается только после успешного вызова POST /api/session/name, иначе разрывается с кодом 4000.
3. **Административные эндпоинты** (/api/admin) для загрузки книг, получения текущей книги и запуска анализа — все URL начинаются с /api/admin.
4. **DeepSeek API** интегрирован с поддержкой двух режимов (thinking enabled/disabled), JSON mode, обработкой ошибок через retry и circuit breaker, а также управлением rate limits через token bucket.
5. **События WebSocket** строго регламентированы: клиент отправляет `player_message`, сервер отвечает событиями `master_thinking`, `master_speech`, `dice_prompt`, `turn_order` и др.
