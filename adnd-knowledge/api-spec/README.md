## REST-эндпоинты

### Эндпоинты администратора

Все URL административных эндпоинтов начинаются с `/api/admin`.

- **POST /books/upload** — загрузка книги. Принимает `multipart/form-data` с полем `file`. Валидирует расширение (`.txt`, `.epub`, `.fb2`, `.md`) и размер файла. Преобразует загруженный файл в plain text, создаёт запись в таблице `arc_book`, сохраняет текст на диске. Возвращает `{ "book_id": 1 }`.
- **GET /books/current** — получение информации о текущей (последней по времени загрузки) книге. Если ни одной книги не загружено, возвращает 404.
- **POST /books/current/start** — запуск или возобновление анализа текущей книги. Принимает тело запроса с параметрами: `fragment_size`, `split_by_chapters`, `extraction_prompt`.

### Эндпоинты сессии

- **POST /api/session/start** — создание новой сессии. Возвращает `session_id` и статическое приветствие с просьбой назвать имя. Ответ содержит поле `master_says` с текстом приветствия и запросом имени.
- **POST /api/session/name** — установка имени игрока и получение персонального благословения от LLM. Тело запроса должно содержать поля `session_id` и `player_name`. Логика: проверяет существование сессии и что `name_provided == false`, обновляет запись, вызывает DeepSeek с промтом, возвращает ответ. Ответ содержит поле `master_says`.
- **GET /api/session/{session_id}** — получение метаданных сессии. Возвращает: `session_id`, `player_name`, `created_at`, `last_active`, `name_provided`.
- **GET /api/session/{session_id}/history** — получение истории диалога. Возвращает JSON с сообщениями. (Примечание: данный эндпоинт отсутствует в ФТЗ, но предполагается к реализации.)

### Эндпоинт здоровья

- **GET /api/health** — проверка здоровья приложения. Должен проверять доступность базы данных. Возвращает `{"status": "ok", "version": "0.1", "db": "connected"}`.

### Эндпоинт чата

- **POST /api/chat/send** — отправка сообщения в чат. Используется для отправки сообщения, в то время как для стриминга ответов применяется WebSocket.

## WebSocket

### Подключение и управление

- **WebSocket /ws/chat/{session_id}** — основной эндпоинт для чата между игроком и ИИ-Мастером. Подключение возможно только после успешного именования сессии. Если имя не установлено, сервер разрывает соединение с кодом 4000 и сообщением «Требуется имя».
- Управление соединениями осуществляется через `ConnectionManager`.

### События от клиента

- Клиент отправляет сообщение в формате: `{"action": "player_message", "text": "..."}`. Длина текста ограничена 500 символами.
- Альтернативный формат заявки: `{"player": "имя", "action_type": "say"|"do", "content": "текст"}`.

### События от сервера

- `{"event": "connected", "session_id": "..."}` — успешное подключение.
- `{"event": "master_thinking", "status": "started"}` — начало генерации ответа.
- `{"event": "master_speech", "text": "...", "turn_id": "uuid"}` — сгенерированная речь Мастера.
- `{"event": "master_thinking", "status": "stopped"}` — завершение генерации.
- `{"event": "error"}` — ошибка.
- `{"type": "dice_prompt", "reason": "...", "secret": true/false}` — запрос броска кубика.
- `{"type": "turn_order", "order": ["имя1", "имя2", ...]}` — порядок хода.

## Интеграция с DeepSeek API

- Используется Chat Completions API (`v1/chat/completions`) с моделью `deepseek-chat` (или `deepseek-v4-flash`). Поддерживается JSON mode для структурированных ответов.
- Интеграция включает Chat Completions в JSON mode и FIM (Fill-in-the-Middle).

### Режимы работы

- **Режим с размышлениями (thinking enabled)** — модель генерирует цепочку рассуждений (Chain of Thought), которая возвращается в ответе API как `reasoning_content`. Требует больше времени и токенов. Подходит для сложных логических задач, математики, программирования, многошаговых инструкций.
- **Режим без размышлений (thinking disabled)** — быстрый и экономичный режим. Модель сразу генерирует итоговый ответ. Подходит для общих вопросов, творчества или простых поручений.
- Параметр `thinking` поддерживается для модели `deepseek-v4-flash` и может быть `enabled` или `disabled`.

### Обработка ошибок и лимиты

- Обработка ошибок включает retry с exponential backoff и circuit breaker.
- Rate limits DeepSeek API учитываются с помощью очереди с регулированием (token bucket).

## Сводка

1. **Архитектура API** — приложение построено на FastAPI с поддержкой async и WebSocket. REST-эндпоинты используются для управления сессиями, книгами и проверки здоровья, а WebSocket — для чата в реальном времени.
2. **Основные эндпоинты** — выделены три группы: административные (`/api/admin/books/*`), сессионные (`/api/session/*`) и чат (`/api/chat/send` + WebSocket `/ws/chat/{session_id}`).
3. **Протокол WebSocket** — строго регламентирован: подключение только после именования, фиксированные форматы событий от клиента и сервера, поддержка стриминга ответов.
4. **Интеграция с DeepSeek** — используется Chat Completions API с поддержкой JSON mode и двух режимов (с размышлениями и без). Реализована обработка ошибок и управление лимитами.
5. **Тестирование и устойчивость** — предусмотрены механизмы проверки сохранности сессии после перезапуска сервера (подключение по WebSocket к тому же session_id, проверка GET /api/session/{id}).
