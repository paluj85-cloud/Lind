### 📊 Лимиты GitHub API в GitHub Actions

Эти лимиты не зависят от OpenCode и касаются любых действий в рамках GitHub Actions. Они важны, потому что OpenCode, например, при создании pull request'ов или комментировании, также выполняет запросы к GitHub API:

| Тип аутентификации | Лимит запросов в час |
| :--- | :--- |
| **Без аутентификации** (не рекомендуется для автоматизации) | 60 запросов в час с одного IP-адреса[reference:0][reference:1] |
| **С автоматическим токеном `GITHUB_TOKEN`** (предоставляется GitHub Actions по умолчанию для репозитория) | 1,000 запросов в час на репозиторий[reference:2] |
| **С использованием Personal Access Token (PAT)** | 5,000 запросов в час (базовый лимит для пользователя, зависит от плана GitHub)[reference:3] |

Если OpenCode превысит лимит, его операции с репозиторием (комментирование, создание PR) могут временно блокироваться.

### 🤖 Лимиты API ИИ-моделей

Самые важные ограничения — это лимиты API того провайдера ИИ, которого вы используете в OpenCode.

*   **Собственные ключи от провайдеров (Anthropic, OpenAI и др.)**:
    *   **Вы полностью управляете расходами**, используя свой платный аккаунт.
    *   Лимиты зависят от вашего тарифного плана (Usage Tier). Они обычно включают:
        *   **RPM (Requests Per Minute)** — количество запросов в минуту.
        *   **TPM (Tokens Per Minute)** — количество токенов в минуту (часто более важный показатель).
    *   **Anthropic (Claude)**: Лимиты зависят от уровня (Tier). Например, для Tier 1 лимит может составлять **50 запросов в минуту (RPM) и 50,000 входных токенов в минуту (ITPM)**. При превышении вы получите ошибку `Rate limit exceeded`[reference:4][reference:5].
    *   **OpenAI**: Лимиты также зависят от уровня использования (Usage Tier). Превышение вызывает ошибку `429 RateLimitError`[reference:6]. Важно также следить за **денежным лимитом** (usage limit), который можно установить в настройках аккаунта OpenAI, чтобы контролировать расходы[reference:7].

*   **Подписка OpenCode Go**:
    *   Если вы используете встроенную подписку OpenCode Go, она имеет фиксированные **денежные лимиты на использование**:
        *   **$12** за каждые 5 часов.
        *   **$30** в неделю.
        *   **$60** в месяц[reference:8].
    *   Количество запросов, которое можно совершить в рамках этих лимитов, напрямую зависит от стоимости используемых моделей[reference:9]. Как только лимит Go исчерпан, потребление может переключиться на ваш баланс OpenCode Zen, если он настроен[reference:10].

### 🔧 Технические ограничения GitHub Actions

Эти лимиты накладываются самой платформой GitHub Actions:

*   **Тайм-аут выполнения задания (Job Timeout)**: Максимальное время выполнения одного job в GitHub Actions — **6 часов** (для приватных репозиториев; для публичных — 72 часа, но лимит в 6 часов часто является стандартом для платных планов)[reference:11]. Если OpenCode выполняет сложную задачу дольше этого времени, job будет принудительно завершен.
*   **Лимиты кэширования**: Существует ограничение на **200 загрузок новых записей кэша в минуту на репозиторий**[reference:12]. Это актуально, если вы кэшируете модели или зависимости OpenCode для ускорения работы.

### 💾 Безопасность и управление API-ключами

Официальная документация OpenCode и лучшие практики рекомендуют хранить все ключи (например, `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`) в **Secrets** вашего репозитория. Это самый безопасный способ, так как ключи не будут видны в логах сборок и коде[reference:13][reference:14].

### RESPONSE (ИИ)

Rate limits — это количество разрешённых **запросов к API** за единицу времени. Конкретные единицы измерения зависят от того, о чьём API идёт речь.

### Для GitHub API
Лимит измеряется в **количестве HTTP-запросов в час** (requests per hour, RPH).  
Один вызов MCP-инструмента типа `create_issue` или `project_item_list` — это, как правило, ровно **один запрос** к GitHub API. GitHub считает только число вызовов, независимо от того, создаёте ли вы простой комментарий или тяжелый Issue с мегабайтным телом.

*Пример:* Если ваш лимит — 1000 запросов/час, вы можете за час создать 1000 Issue, или 999 Issue плюс один аудит списком, и т.д. Когда лимит исчерпан, API возвращает ошибку `403 rate limit exceeded`, и приходится ждать сброса (обычно каждый час).

### Для API ИИ-моделей (Anthropic, OpenAI)
Здесь контроль сразу по нескольким осям:

- **RPM (Requests Per Minute)** — количество запросов-сообщений к модели в минуту.  
  Один ваш промпт к Claude — это 1 запрос, даже если внутри он содержит гигантский «контекст» из десяти абзацев. Превысите RPM — получите `429 rate limit exceeded`.

- **TPM (Tokens Per Minute)** — суммарное количество **токенов** (входных + выходных), обработанных в минуту.  
  Токен — это кусочек текста, слово или его часть, на которые модель «смотрит». Чем длиннее ваши промпты и ответы, тем больше токенов расходуется. Превышение TPM — тоже ошибка лимита, даже если вы сделали всего один запрос.

- **Денежный лимит** — например, в подписке OpenCode Go это **$12 / 5 часов**, **$30 / неделя**, **$60 / месяц**. Как только вы нагенерили ответов на эту сумму, дальнейшие вызовы ИИ прекращаются до следующего периода.

Таким образом:
- Для GitHub — считаются просто вызовы API.
- Для ИИ — считаются и вызовы, и объём текста (токены), и деньги.

## 3. Скрипт: вызов DeepSeek и работа с GitHub

Внутри блока `script` (выполняется в среде Node.js) можно использовать встроенный `fetch` (Node 18+). Пример ниже анализирует тело Issue и добавляет комментарий с предложением категории (label) и milestone.

```javascript
const issueBody = context.payload.issue.body;
const issueTitle = context.payload.issue.title;
const issueNumber = context.payload.issue.number;

// Контроль: не анализировать собственные Issue, созданные ботами (опционально)
if (context.payload.issue.user.type === 'Bot') {
  console.log('Пропускаем Issue от бота');
  return;
}

// 1. Формируем компактный промпт, извлекаем только суть
const prompt = `
Ты — ассистент проекта TTRPG Master. Проанализируй Issue и предложи:
- одну основную категорию (label) из списка: book-of-world, book-of-history, book-of-hooks, specification, design, magic, setting, social
- подходящий milestone из: v0.1-core, v0.2-setting, v0.3-history, v0.4-hooks
- краткий комментарий (1-2 предложения) с аргументацией.
Формат ответа строго JSON: {"label": "...", "milestone": "...", "comment": "..."}

Название Issue: "${issueTitle}"
Тело:
${issueBody.substring(0, 3000)}`;  // Ограничиваем длину, чтобы не перерасходовать токены

// 2. Вызов DeepSeek Chat API
const response = await fetch('https://api.deepseek.com/v1/chat/completions', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${process.env.DEEPSEEK_API_KEY}`,
  },
  body: JSON.stringify({
    model: 'deepseek-chat',          // актуальная модель
    messages: [{ role: 'user', content: prompt }],
    temperature: 0.3,                // низкая для предсказуемого JSON
    max_tokens: 150,
  }),
});

if (!response.ok) {
  const errorText = await response.text();
  throw new Error(`DeepSeek API error: ${response.status} ${errorText}`);
}

const data = await response.json();
const content = data.choices[0].message.content;

// 3. Парсим JSON из ответа (DeepSeek может вернуть обёрнутую строку)
let result;
try {
  // Иногда модель оборачивает JSON в ```json ... ```
  const jsonMatch = content.match(/\{[\s\S]*\}/);
  result = JSON.parse(jsonMatch ? jsonMatch[0] : content);
} catch (e) {
  console.error('Не удалось распарсить JSON, сырой ответ:', content);
  throw e;
}

// 4. Добавляем предложенные label и milestone
await github.rest.issues.addLabels({
  owner: context.repo.owner,
  repo: context.repo.repo,
  issue_number: issueNumber,
  labels: [result.label],
});

// 5. Привязываем milestone (требуется номер milestone, получаем по названию)
const milestones = await github.rest.issues.listMilestones({
  owner: context.repo.owner,
  repo: context.repo.repo,
  state: 'open',
});
const milestone = milestones.data.find(m => m.title === result.milestone);
if (milestone) {
  await github.rest.issues.update({
    owner: context.repo.owner,
    repo: context.repo.repo,
    issue_number: issueNumber,
    milestone: milestone.number,
  });
} else {
  console.warn(`Milestone "${result.milestone}" не найден`);
}

// 6. Добавляем комментарий
await github.rest.issues.createComment({
  owner: context.repo.owner,
  repo: context.repo.repo,
  issue_number: issueNumber,
  body: `🤖 **DeepSeek предложил:**\n> ${result.comment}\n\nПрименены label «${result.label}», milestone «${result.milestone}».`,
});
```

**Важно:** передача `DEEPSEEK_API_KEY` в окружение скрипта. Допишите в workflow-файл перед `steps`:

```yaml

  analyze:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - name: Анализ Issue
        uses: actions/github-script@v7
        env:
          DEEPSEEK_API_KEY: ${{ secrets.DEEPSEEK_API_KEY }}   # <-- проброс секрета
        with:
          github-token: ${{ secrets.GITHUB_TOKEN }}
          script: |
            // ... код выше
```

## 4. Контроль лимитов и расходов

- **GitHub API**: `actions/github-script` использует встроенный токен Actions с лимитом **1000 запросов/час** на репозиторий.  
  В нашем примере — 3–4 вызова (добавление label, обновление milestone, комментарий). Это совершенно безопасно.
- **DeepSeek API**: лимиты зависят от вашего тарифа. Для `deepseek-chat` они обычно:
  - 30 RPM (запросов в минуту)
  - 50 000 TPM (токенов в минуту)
  - Платный аккаунт — без жёсткого дневного лимита, но с контролем расходов.  
  Наш промпт с ограничением тела Issue до 3000 символов создаёт ~1000–1500 входных токенов, ответ с JSON — около 100 выходных токенов. Один запуск дешевле $0.001 (при ценах DeepSeek).
- Если Issues создаются редко, превышение RPM исключено. Для автоматизации массового аудита лучше ставить задержку между запросами.

Чтобы не сжигать бюджет на гигантские Issue, мы обрезали тело до 3000 символов. При необходимости можно увеличить лимит, но следите за TPM.

## 1. Workflow с триггером `issue_comment` и фильтром

Создайте или замените файл `.github/workflows/deepseek-mention.yml`:

```yaml
name: Ответ на @bot в Issues

on:
  issue_comment:
    types: [created]

jobs:
  respond:
    if: |
      github.event.comment.user.type != 'Bot' &&
      contains(github.event.comment.body, '@bot')
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - name: Обработка запроса
        uses: actions/github-script@v7
        env:
          DEEPSEEK_API_KEY: ${{ secrets.DEEPSEEK_API_KEY }}
        with:
          github-token: ${{ secrets.GITHUB_TOKEN }}
          script: |
            // 1. Получаем контекст комментария и Issue
            const commentBody = context.payload.comment.body;
            const issueNumber = context.payload.issue.number;
            
            // 2. Можно получить полное тело Issue, если нужно
            const { data: issue } = await github.rest.issues.get({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: issueNumber,
            });
            const issueTitle = issue.title;
            const issueBody = issue.body || '';
            
            // 3. Формируем промпт для DeepSeek, исключив сам триггер
            const userMessage = commentBody.replace(/@bot\s*/, '').trim();
            const prompt = `
            Ты — ассистент проекта TTRPG Master. Тебя вызвали в Issue. 
            Ответь на следующее сообщение, учитывая контекст Issue.
            
            Заголовок Issue: "${issueTitle}"
            Тело Issue (первые 2000 символов): "${issueBody.substring(0, 2000)}"
            
            Сообщение пользователя: "${userMessage}"
            
            Дай полезный, краткий ответ (2-5 предложений). Не фантазируй, если вопрос не по проекту — вежливо укажи это.
            `;
            
            // 4. Вызов DeepSeek
            const response = await fetch('https://api.deepseek.com/v1/chat/completions', {
              method: 'POST',
              headers: {
                'Content-Type': 'application/json',
                'Authorization': `Bearer ${process.env.DEEPSEEK_API_KEY}`,
              },
              body: JSON.stringify({
                model: 'deepseek-chat',
                messages: [{ role: 'user', content: prompt }],
                temperature: 0.5,
                max_tokens: 300,
              }),
            });
            
            if (!response.ok) {
              const errorText = await response.text();
              throw new Error(`DeepSeek API error: ${response.status} ${errorText}`);
            }
            
            const data = await response.json();
            const answer = data.choices[0].message.content;
            
            // 5. Публикуем ответный комментарий
            await github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: issueNumber,
              body: `🤖 **Ответ от @bot:**\n${answer}`,
            });
```

## 2. Пояснение ключевых моментов

### Триггер только на `@bot`

- `issue_comment` с типом `created` — срабатывает при любом новом комментарии.
- `if: ...` проверяет:
  - `github.event.comment.user.type != 'Bot'` — игнорирует комментарии от ботов (включая самого себя).
  - `contains(github.event.comment.body, '@bot')` — текст должен содержать `@bot`. Можно заменить на `@ttrpg-assistant`, если у вас зарегистрирован бот-аккаунт, или использовать `!ask`, `/bot` — что угодно.

### Извлечение сообщения пользователя

Строчка `commentBody.replace(/@bot\s*/, '').trim()` удаляет само обращение, чтобы не засорять промпт.

### Безопасность

- API-ключ DeepSeek передаётся через `env` из GitHub Secrets, не засвечивается.
- Лимиты GitHub API: два вызова в job (получить Issue и создать комментарий) — безопасно.

### Защита от бесконечного цикла

Бот добавляет в ответ `@bot`? Нет, он пишет «@bot» только в тексте, а триггер ищет вхождение в новом комментарии. Но на всякий случай мы уже отсекли ботов условием `user.type != 'Bot'`, так что ответ самого бота не вызовет повторный запуск.

## 3. Альтернативный триггер: упоминание GitHub-аккаунта

Если вы зарегистрируете отдельный GitHub-аккаунт для бота (например, `ttrpg-master-bot`) и добавите его в репозиторий как collaborator, можно ловить именно **настоящее упоминание** через `@ttrpg-master-bot`. Тогда условие можно записать так:

```yaml
if: |
  github.event.comment.user.type != 'Bot' &&
  contains(github.event.comment.body, '@ttrpg-master-bot')
```

GitHub подсвечивает такие упоминания и уведомляет аккаунт — это чуть более «родной» способ.

Но даже без настоящего аккаунта, просто текстовая метка `@bot` прекрасно работает.

## 4. Что делать, если Issue закрыт или PR?

Action срабатывает только на комментарии в Issue (тип `issues`). Если нужна поддержка Pull Request — добавьте `pull_request` или используйте `issue_comment` универсально (он срабатывает и для PR, потому что PR технически является Issue с кодом). В коде выше через `context.payload.issue` можно работать и с PR, но учтите, что для PR есть отдельный `pull_request` номер.

Если вам нужна именно реакция на метки/закрытия — это отдельный workflow.

В GitHub Actions лимиты делятся на **минуты выполнения** и **количество одновременно запущенных job'ов**. Это не git-команда, а информация, доступная в веб-интерфейсе GitHub или через API биллинга. Посмотреть остаток можно так:

---

## 1. 🖥️ Через веб-интерфейс GitHub

**Для личного аккаунта:**
- Нажмите на аватар в правом верхнем углу → **Settings**.
- В левом меню выберите **Billing & plans** → **Plans and usage**.
- Прокрутите до раздела **GitHub Actions**. Там будет показано:
  - Общее количество минут, включённых в ваш тарифный план (например, 2000 или 3000 минут/мес для Free/Pro).
  - **Used** / **Total** – использовано и остаток до конца расчётного периода.

**Для организации (если репозиторий принадлежит организации):**
- Перейдите в профиль организации → **Settings** → **Billing & plans** → **Plans and usage**.
- Раздел **GitHub Actions** покажет сводку по всем репозиториям организации.

---

## 2. 📊 Через GitHub REST API

Можно запросить данные биллинга напрямую. Это полезно, если вы хотите автоматизировать мониторинг.

**Для личного аккаунта** (требуется токен с правами `user`):
```
GET https://api.github.com/users/{username}/settings/billing/actions
```
Ответ вернёт JSON с полями:
- `total_minutes_used`
- `total_paid_minutes_used`
- `included_minutes`

**Для организации** (токен с правами `admin:org`):
```
GET https://api.github.com/orgs/{org}/settings/billing/actions
```

Пример curl с личным токеном:
```bash
curl -H "Authorization: token YOUR_TOKEN" \
  https://api.github.com/users/your-username/settings/billing/actions
```

---

## 3. 📝 Локальная команда git?

Git сам по себе не хранит информацию о лимитах Actions. Даже `gh` CLI не имеет прямой команды для биллинга. Единственный способ – через `gh api` с правильно сформированным запросом:

```bash
gh api user/settings/billing/actions --jq '.included_minutes, .total_minutes_used'
```

Но это по сути тот же вызов API.

---

## 4. 🔔 Предупреждения о превышении лимитов

GitHub отправляет email-уведомления, когда вы приближаетесь к порогу бесплатного лимита (например, израсходовано 75%, 90%, 100%). Проверьте почту, привязанную к аккаунту.

---

## 📌 Важно помнить

- Лимиты ежемесячные и сбрасываются в день начала платного периода.
- Бесплатные минуты для публичных репозиториев не ограничены (для стандартных машин). Для приватных лимит зависит от плана (например, 2000 минут/мес на Free).
- В вашем случае, судя по ошибкам, проблема была не в лимите минут, а в правах токена и в формате запроса к DeepSeek. Но проверить квоту всё равно полезно — особенно если job'ы перестанут запускаться с сообщением `You have exceeded your included minutes`.

### 💡 Что можно увидеть в логах

- **`console.log` из скрипта**: все ваши `console.log('✅ Прочитан ...')`, ошибки, вывод промптов.
- **Полный стек ошибок**: если была ошибка 400 или 403, вы увидите её прямо там вместе с ответом API.
- **Переменные окружения**: секреты скрыты (отображаются как `***`), но можно увидеть, что секрет был передан (или нет).

---

### 🛠 Если логи нужно скачать

Внутри открытого запуска есть кнопка **«…» (три точки) справа вверху** → **Download log archive**. Вы получите ZIP-архив со всеми логами job’ов в текстовом виде. Это полезно, если нужно поделиться с другими или отправить в саппорт.

---

Если вы имели в виду что-то другое — например, «консоль» в смысле терминала на виртуальной машине раннера — то прямого SSH-доступа к раннерам GitHub Actions не даёт. Для отладки можно использовать `tmux` или `tmate` для интерактивного доступа, но это уже более продвинутая тема.

Надеюсь, это то, что вы искали. Если нужна помощь в трактовке конкретных логов — скиньте фрагмент, разберёмся.
