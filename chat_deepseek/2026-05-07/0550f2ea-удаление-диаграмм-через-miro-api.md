### RESPONSE (ИИ)

Привет! Удаление элементов с доски Miro через API — это стандартная `DELETE`-операция. Важно понимать, что у каждого типа контента (фигуры, карточки, соединительные линии и т.д.) своя конечная точка (эндпоинт), так как Miro не считает их все одной «диаграммой».

Вот как это работает для двух основных типов: обычных фигур и фигур блок-схем.

### 🎯 Основные шаги для удаления

Чтобы удалить любой объект, нужно знать три ключевых параметра:
*   **Тип объекта:** От него зависит, какой URL использовать (например, фигура, текст, карточка).
*   **ID доски (`board_id`):** Уникальный идентификатор доски.
*   **ID элемента (`item_id`):** Уникальный идентификатор самого объекта. Его можно получить, запросив список всех элементов на доске с помощью `GET`-запроса по адресу `https://api.miro.com/v2/boards/{board_id}/items`.

Для запроса понадобится **токен доступа** (access token) с правами на запись для досок — `boards:write`. Токен необходимо передавать в заголовке `Authorization: Bearer {your_access_token}`[reference:0][reference:1].

### 📝 Варианты удаления

#### 1. Удаление простой фигуры (Shape Item)

Используйте этот метод для заранее созданных фигур из библиотеки Miro (прямоугольники, круги и т.д.)[reference:2].

*   **Метод:** `DELETE`
*   **URL:** `https://api.miro.com/v2/boards/{board_id}/shapes/{item_id}`[reference:3]

**Пример запроса с помощью cURL:**
```bash
curl --request DELETE \
  --url https://api.miro.com/v2/boards/{board_id}/shapes/{item_id} \
  --header 'Authorization: Bearer {your_access_token}' \
  --header 'accept: application/json'
```

#### 2. Удаление фигуры блок-схемы (Flowchart Shape Item)

Этот метод применяется к фигурам, созданным через экспериментальный инструмент для блок-схем (Flowchart shapes)[reference:4]. Основное отличие от первого варианта — это URL запроса.

*   **Метод:** `DELETE`
*   **URL:** `https://api.miro.com/v2-experimental/boards/{board_id}/shapes/{item_id}`[reference:5]

**Пример запроса с помощью cURL:**
```bash
curl --request DELETE \
  --url https://api.miro.com/v2-experimental/boards/{board_id}/shapes/{item_id} \
  --header 'Authorization: Bearer {your_access_token}' \
  --header 'accept: application/json'
```

> **Важно для блок-схем:** Экспериментальный API (`v2-experimental`) может изменяться без предварительного уведомления[reference:6].

#### 💡 Что делать с другими элементами?

Принцип удаления одинаков для всех типов элементов, отличается только конечная точка. Вот несколько примеров[reference:7]:
*   **Текст (Text):** `DELETE /v2/boards/{board_id}/texts/{item_id}`
*   **Карточка (Card):** `DELETE /v2/boards/{board_id}/cards/{item_id}`
*   **Изображение (Image):** `DELETE /v2/boards/{board_id}/images/{item_id}`
*   **Стикер (Sticky Note):** `DELETE /v2/boards/{board_id}/sticky_notes/{item_id}`
*   **Соединительная линия (Connector):** `DELETE /v2/boards/{board_id}/connectors/{item_id}`

Надеюсь, эта инструкция поможет. Если останутся вопросы или понадобится уточнить что-то по другим типам элементов, спрашивайте.

Да, такие инструкции есть. Они зависят от того, как именно ваш ИИ будет общаться с Miro — напрямую через REST API или через инструменты на базе Model Context Protocol (MCP), например, в Claude Code.

Вот три наиболее актуальных подхода и документация к ним.

### 🧠 Вариант 1: Промпты для Miro MCP Server (для VS Code, Cursor, Claude и др.)

Самый передовой способ — использовать официальный Miro MCP Server (Model Context Protocol). Он позволяет вашему AI-ассистенту напрямую и безопасно управлять досками Miro (создавать диаграммы и документы, искать контент) с помощью готовых инструментов.

Для этого сервера уже созданы специальные промпты. Вы можете не писать сложные инструкции с нуля, а использовать один из них:

*   **`code_create_from_board`**: Анализирует доску Miro (например, с ТЗ), и на основе контекста генерирует документацию и код[reference:0].
*   **`code_explain_on_board`**: Делает обратное — берёт ваш программный код и создаёт по нему на доске наглядные диаграммы (архитектуру, потоки данных)[reference:1].

Эти промпты используют разные **инструменты (tools)** для работы с контентом доски: `context_explore` и `context_get` для поиска контекста, `diagram_create` и `doc_create` — для создания контента и т.д[reference:2].

---
### ⚙️ Вариант 2: Инструкция для прямой работы с REST API

Если ваш ИИ-агент будет напрямую совершать REST API-запросы к Miro (например, `POST /v2/boards/{board_id}/shapes`), то главная «инструкция» для него — это спецификация OpenAPI. Многие AI-фреймворки умеют читать такой формат для понимания доступных методов.

Ознакомьтесь с этими ресурсами:

*   **REST API Reference Guide**: Содержит полное описание всех v1 и v2 эндпоинтов, параметров и схем запросов[reference:3].
*   **How-To's & Tutorials**: Собрание пошаговых руководств для типичных задач при работе с REST API Miro[reference:4].
*   **Miro Python API Docs**: Документация для бота на Python[reference:5].

---
### ✨ Вариант 3: Промпты для встроенного AI-помощника (Miro Assist)

Если задача — генерировать промпты для встроенного в саму Miro ИИ-функционала, то пригодится **фреймворк RISEN**[reference:6]:

*   **R (Role — Роль)**: Назначьте ИИ конкретную роль: «Ты — senior product designer...»[reference:7].
*   **I (Input — Входные данные)**: Укажите, что использовать на доске как контекст[reference:8].
*   **S (Steps — Шаги)**: Четко опишите последовательность действий[reference:9].
*   **E (Expectation — Ожидаемый результат)**: Точно опишите конечный результат (например, «12 слайдов» вместо просто «презентация»)[reference:10].
*   **N (Narrowing — Сужение)**: Укажите, чего в ответе быть не должно[reference:11].

### 💎 Практический пример

Вот так может выглядеть промпт для ИИ-агента, работающего с официальным Miro MCP Server:

> **Задача:** Создайте блок-схему на доске Miro на основе предоставленного текстового описания бизнес-процесса.
>
> **Промпт для агента:**
> *   **Роль:** Ты — опытный бизнес-аналитик, который визуализирует процессы с помощью блок-схем на Miro.
> *   **Входные данные:** Текст процесса находится в документе Miro с ID `[ID_документа]` на доске `[ID_или_URL_доски]`. Сначала используй инструмент `doc_get`, чтобы прочитать его содержимое.
> *   **Шаги:**
>     1.  Проанализируй текст и выдели ключевые шаги, точки принятия решений и конечный результат.
>     2.  Используй инструмент `diagram_get_dsl` для типа `flowchart`, чтобы узнать синтаксис, прежде чем создавать диаграмму[reference:12].
>     3.  Создай блок-схему с помощью инструмента `diagram_create` и полученного DSL-кода.
> *   **Ожидаемый результат:** Блок-схема в центре доски с корректными геометрическими фигурами.
> *   **Сужение:** Не добавляй цветовые акценты, если они не несут смысловой нагрузки.
