### 1. Введение и цели
Создать веб-приложение для текстовой ролевой игры, где роль ведущего (мастера) исполняет LLM (DeepSeek). Игроки (от 1 до N) описывают свои действия в общем чате, мастер-ИИ реагирует, управляет NPC, сюжетом и проверками. В первой версии все сообщения пишет один реальный человек – «игрок-мастер» (от лица всех персонажей). Приложение сохраняет состояние игры (листы персонажей, историю мира, чат), позволяет создавать новые игры и продолжать старые.

### 2. Функциональные требования

#### 2.1. Главный экран (при запуске)
- Список всех ранее созданных игр (название, дата создания, возможно последнее сообщение).
- Кнопка «Создать новую игру».
- При выборе существующей игры – загрузка полного состояния и переход в игровой чат.

#### 2.2. Создание новой игры
**Экран настройки сеттинга:**
- Поля с выбором или свободным вводом:
  - Эпоха / Время (реальное, прошлое, будущее, фэнтези и т.д.)
  - Наличие магии (да/нет/свободно)
  - Уровень технологий (каменный век – космос)
  - Дополнительные параметры (можно добавлять динамически)
- Большое текстовое поле «Дополнительные пожелания по миру» (свободный текст).
- Кнопка «Далее: добавить игроков».

**Экран добавления персонажей игроков:**
- Форма для каждого игрока: имя персонажа (обязательно), краткое текстовое описание (характер, внешность, особенности). Можно добавить до 10 игроков.
- Кнопка «Запустить игру».

#### 2.3. Генерация мира («Книга мира»)
- При нажатии «Запустить игру» отправляется запрос к LLM (DeepSeek) с:
  - Системным промптом (из админки).
  - Полем «Настройки сеттинга» (собранные выбранные значения).
  - Полем «Пожелания».
- LLM должна сгенерировать детальное описание мира (локации, история, ключевые NPC, законы магии/технологий, фракции и т.д.) объёмом **до 40000 символов** (настраивается в админке).
- Сгенерированная «Книга мира» сохраняется в БД.
- После генерации создаётся новое состояние игры (идентификатор) и открывается игровой чат.

#### 2.4. Игровой чат
**Компоненты:**
- Область вывода сообщений (история чата, включая системные сообщения).
- Поле ввода текста (многострочное).
- Кнопка «Отправить».
- Кнопка «Бросить кубик» (открывает диалог: выбрать кость, например d4, d6, d8, d10, d12, d20, d100; после подтверждения в чат отправляется системное сообщение вида: `[Бросок] Игрок-мастер: d20 → 17`).
- Кнопка «Ввести результат кубика» (всплывающее поле для вручную введённого результата, например `15`; в чат отправляется аналогичное сообщение).

**Логика чата:**
- Каждое сообщение (текст, бросок, ручной ввод) отправляется на сервер.
- Сервер формирует запрос к LLM, добавляя в начало промпта **Книгу мира** и **историю последних N сообщений** (где N вычисляется, чтобы не превысить контекст LLM; при переполнении обрезаются самые старые сообщения, а в чат выводится системное предупреждение: `[Система] Контекст переполнен, старые сообщения удалены из памяти ИИ`).
- Ответ LLM выводится в чат (как сообщение от «Мастера»).
- Все сообщения и ответы сохраняются в БД.

#### 2.5. Листы персонажей (упрощённая версия)
**В правом верхнем углу – кнопка «Персонажи» (или иконка). При клике открывается боковая панель/модальное окно со списком персонажей. Для каждого персонажа отображается:**
- Имя, раса, класс (если есть), уровень (1).
- Основные характеристики: Сила, Ловкость, Конституция, Интеллект, Воля (числа от 1 до 20, по умолчанию 10).
- HP (текущее / максимальное), MP (если есть магия).
- Деньги (золото/кредиты).
- Инвентарь (список предметов с кратким описанием).
- Профессии и способности (текст).
- Контакты и заметки (текст).

**Доступно редактирование:** игрок-мастер может изменять любые поля (числа, текст) для каждого персонажа – через inline-редактирование или отдельную форму.

**Верхняя панель (для каждого персонажа) – отображение HP/MP/денег в виде полосок/иконок. Возможно переключение отображаемого персонажа (по умолчанию выбран первый).**

#### 2.6. Административная панель
**Доступна по отдельному URL (или через кнопку для мастера). Требует простого пароля (один общий, задаётся в переменной окружения).**

**Разделы:**
- **Настройки LLM:** URL API, API-ключ, имя модели (DeepSeek), параметры (температура, max_tokens).
- **Системный промпт для мастера** (многострочное поле, редактируется).
- **Промпт для генерации Книги мира** (отдельно от промпта игрового мастера).
- **Настройки контекста:** максимальный размер «Книги мира» в символах (40000 по умолчанию), количество последних сообщений, отправляемых в каждом запросе (например, 30).
- **Логи ошибок** (только чтение).

**Все изменения сохраняются в БД и применяются без перезапуска сервера (для API-ключей – при следующем запросе).**

### 3. Нефункциональные требования
- **Стек:** Python 3.10+, FastAPI, WebSockets (или HTTP long polling – но рекомендую WebSockets для чата), MySQL, SQLAlchemy (или raw-запросы, но лучше ORM).
- **Фронтенд:** HTML5, CSS3, JavaScript (можно без фреймворков, но с адаптивной вёрсткой под ПК и смартфон). Возможно использовать Bootstrap 5 для скорости.
- **API:** REST для создания игры, получения списка игр, сохранения листов персонажа, бросков. WebSocket для чата (либо Fetch + polling, но WebSocket удобнее).
- **Производительность:** ответ LLM может занимать до 10 секунд – не блокировать другие запросы (асинхронность). Генерация Книги мира – отдельный фоновый task (можно синхронно, но с таймаутом).
- **Безопасность:** валидация ввода, защита от XSS (экранирование вывода в чате). Пароль админки – через переменную окружения.
- **Надёжность:** при ошибке LLM (таймаут, невалидный ответ) выводить в чат сообщение об ошибке и предлагать «Повторить». При повторной ошибке – сообщить и перейти в режим ожидания ручной команды мастера.
- **Данные:** полное сохранение чата и состояния игры в БД. Возможность в любой момент закрыть страницу и вернуться к игре (игра не удаляется, пока мастер не удалит её через админку или специальную кнопку).

### 4. Архитектура (общая схема)
**Компоненты:**
- **Frontend (SPA style):** одна HTML-страница с динамической подгрузкой.
- **Backend:** FastAPI приложение с роутами:
  - `/api/games` – список игр, создание новой.
  - `/api/games/{id}` – получение состояния игры (книга мира, листы персонажей, история чата).
  - `/api/games/{id}/message` – отправка сообщения (синхронно с вызовом LLM, возможно долгий ответ). Вместо WebSocket для упрощения MVP можно использовать длинный HTTP-запрос (но тогда интерфейс будет виснуть). **Лучше WebSocket** – отправляем сообщение, сервер шлёт несколько событий (начало обработки, ответ LLM, ошибка).
  - `/api/games/{id}/roll` – бросить кубик (возвращает результат и добавляет системное сообщение).
  - `/api/games/{id}/characters` – CRUD для листов персонажей.
  - `/admin/*` – админка.
- **База данных:**
  - Таблица `games`: id, title (название игры или автоматическое), created_at, book_of_world (текст), settings_json (сохранённые параметры сеттинга), system_prompt_override (опционально).
  - Таблица `characters`: id, game_id, name, description_json (все атрибуты: stats, hp, mp, gold, inventory и т.д.).
  - Таблица `chat_messages`: id, game_id, role (master, user, system), sender_name (кто написал, либо «Мастер»), content (текст), created_at, is_roll (boolean).
  - Таблица `admin_config`: id, key, value (для хранения API-ключей, промптов и т.д.).
- **Взаимодействие с LLM:** отдельный модуль `llm_client.py` с функцией `ask_llm(system_prompt, user_message, context_messages, book_of_world)`.

### 5. Сценарии использования (User Stories)

1. **Мастер создаёт новую игру:** выбирает настройки мира, добавляет трёх персонажей, запускает игру. Сервер генерирует Книгу мира (40k символов), сохраняет. Открывается чат.
2. **Мастер пишет:** «Игрок А осматривает комнату». Сервер отправляет запрос LLM (Книга мира + последние сообщения). LLM отвечает: «Вы видите старый сундук в углу». Сообщение появляется в чате.
3. **Мастер нажимает «Бросить кубик»** → выбирает d20 → выпадает 12. В чате появляется системное сообщение. Мастер дописывает: «Это проверка ловкости, 12 достаточно?» LLM отвечает с учётом результата.
4. **Мастер вручную правит HP персонажа:** открывает лист, уменьшает текущее здоровье на 5. Изменение сохраняется в БД.
5. **Мастер закрывает браузер.** Через час открывает приложение, выбирает ту же игру – всё состояние загружено, чат продолжается.

### 6. План проекта (спринты, ориентировочные сроки)

#### Спринт 0 (Настройка окружения) – 2-3 дня
- Установка Python, FastAPI, MySQL, настройка виртуального окружения.
- Заготовка проекта (структура папок, конфиги, переменные окружения).
- Простейший "Hello world" на FastAPI, статика.

#### Спринт 1 (База данных и модели) – 3 дня
- Проектирование полных схем таблиц (SQLAlchemy).
- Миграции (Alembic).
- Базовые CRUD для игр, персонажей, сообщений (синхронные тесты через pytest).

#### Спринт 2 (Фронтенд – основа и создание игр) – 4 дня
- Главная страница со списком игр (получение с API).
- Форма создания новой игры (настройки сеттинга + добавление игроков).
- Отправка формы на сервер, запуск генерации Книги мира (синхронно, с прогресс-баром).
- Переход в игровой экран (заглушка).

#### Спринт 3 (Игровой чат – основной функционал) – 5 дней
- Страница игры: отображение чата (история загружается из БД), поле ввода, кнопки броска.
- WebSocket соединение: отправка сообщения → обработка на сервере → вызов LLM → возврат ответа через WebSocket.
- Обработка системных сообщений от бросков кубиков.
- Сохранение всех сообщений в БД.
- Базовая обработка ошибок LLM (таймаут, повтор).

#### Спринт 4 (Листы персонажей) – 4 дня
- Отображение боковой панели с перечнем персонажей.
- Визуализация HP/MP/денег (полосы прогресса или числа).
- Редактирование полей персонажа (inline-формы или всплывающие окна).
- Сохранение изменений на сервер (API).
- Отображение полосок персонажа в верхнем левом углу (можно выбрать активного).

#### Спринт 5 (Админка и доработки) – 4 дня
- Страница администрирования с простой аутентификацией (HTTP Basic или свой пароль).
- Формы для редактирования промптов, API-ключа DeepSeek, параметров контекста.
- Сохранение настроек в БД и их использование в runtime.
- Страница логов ошибок (чтение из файла или таблицы).
- Кнопка «Удалить игру» (только в админке).

#### Спринт 6 (Тестирование, полировка, деплой) – 3 дня
- Сквозное тестирование (создание игры, несколько сообщений, броски, правка листов).
- Исправление багов.
- Оптимизация запросов к LLM (кеширование? пока не нужно).
- Подготовка Dockerfile + docker-compose (MySQL + приложение).
- Развёртывание на тестовом сервере.

**Общая длительность:** ~25 рабочих дней (при одном разработчике). Если два разработчика – можно сократить до 15-18 дней.

## Дополнительные рекомендации для реализации

### Работа с LLM (DeepSeek)
- Использовать **асинхронные HTTP-запросы** (httpx.AsyncClient), чтобы не блокировать event loop FastAPI.
- Таймаут на ответ – 30 секунд.
- Системный промпт должен явно запрещать LLM генерировать броски кубиков за игроков (только за NPC). Пример: «Ты мастер. Игроки сами бросают кубики. Если требуется проверка, попроси игрока бросить d20. Не пиши результаты бросков за игроков.»

### Обрезка контекста
- Подсчёт токенов можно не делать точно, а ограничивать по символам (ориентировочно 1 токен ≈ 4 символа на английском, на русском – 1 символ ≈ 1-2 токена). Для безопасности: лимит для Книги мира 40000 символов, плюс последние 10000 символов чата. Если превышает `max_tokens` модели – обрезаем чат до 5000 символов и выдаём предупреждение.

### Безопасность
- Все API-ключи хранить только в БД в зашифрованном виде (просто base64 – не безопасно, лучше использовать шифрование с ключом из окружения). Для MVP можно хранить открыто, т.к. пилотный проект.

### Интерфейс (мобильная версия)
- Использовать CSS Grid или Flexbox.
- Чат занимает 70% ширины, панель персонажей выезжает сверху/снизу на мобилке.
