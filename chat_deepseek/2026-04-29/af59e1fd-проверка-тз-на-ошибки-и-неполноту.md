Привет, ниже я предоставлю ТЗ на создание приложения. Проверь его на явные ошибки. Возможно чего то не хватает в логике. А возможно в описании, что бы программист (это будет ИИ) лучше понял что от него требуется.
1. Общее описание
Модуль предоставляет администратору интерфейс для загрузки художественной книги, её автоматического анализа ИИ и формирования набора нарративных атомов (конфликтов, твистов, дилемм, отношений, локаций, сенсорных крючков, деталей NPC). Результаты анализа отображаются в виде редактируемых карточек с возможностью ручной корректировки и последующего импорта в основную библиотеку атомов.

2. Пользовательский интерфейс (веб-страница)
Один экран-«мастер» с тремя последовательными состояниями.

2.1 Этап 1 – Загрузка и настройка
Заголовок: «Загрузка книги»
Кнопка: «Сбросить» — очищает все данные текущей сессии и возвращает к этапу 1.

2.1.1 Блок загрузки файла
Область Drag & Drop с подписью «Перетащите файл сюда или выберите на устройстве».

Поддерживаемые форматы: .txt, .epub, .md, .fb2. При выборе файла отображается его имя и размер.

При загрузке FB2 сервер автоматически конвертирует его в plain text (извлекает содержимое из XML-тегов <body>), результат используется для анализа.

2.1.2 Настройки разбивки
Числовое поле «Средний размер фрагмента (символов)», значение по умолчанию 8000. Диапазон 1000–50000.

Чекбокс «Разбивать по главам/разделам» (включён по умолчанию). Если в тексте обнаружены заголовки глав, разбиение выполняется по ним; фрагменты, превышающие лимит, дополнительно режутся по границам абзацев. Если чекбокс выключен — текст разбивается только по указанному лимиту с учётом целостности абзацев.

2.1.3 Блок промптов ИИ
Разворачивающаяся секция «Настройка промптов» (по умолчанию скрыта).

Textarea «Промпт для пофрагментной экстракции» (системный промпт, отправляемый с каждым фрагментом).

Textarea «Промпт для итогового анализа и связывания атомов» (для этапа постобработки).

Кнопка «Сбросить к стандартным» — заменяет содержимое полей на эталонные промпты из файла конфигурации.

2.1.4 Запуск
Кнопка «Загрузить и анализировать». При нажатии:

Проверяется, что файл выбран и размер не превышает максимально допустимого (задаётся в конфигурации сервера, например 10 МБ для plain text эквивалента).

Книга при необходимости конвертируется в plain text, разбивается на фрагменты согласно настройкам.

Начинается процесс анализа; интерфейс переходит к этапу 2 (прогресс).

2.2 Этап 2 – Прогресс обработки
Отображается блок прогресса вместо формы загрузки.
Состав:

Индикатор текущего этапа: «Разбивка на фрагменты», «Анализ фрагментов», «Постобработка».

Прогресс-бар с текстом «Обработано X из Y фрагментов».

Оценочное время завершения (если возможно оценить по среднему времени обработки одного фрагмента).

Кнопка «Прервать» (с подтверждением). При прерывании все уже полученные результаты сохраняются в статусе «Черновик», администратор может позже продолжить с того же места или удалить черновики.

Обновление прогресса происходит через периодический опрос сервера (например, раз в 2 секунды).

2.3 Этап 3 – Просмотр и редактирование результатов
После завершения постобработки автоматически отображается экран «Результаты извлечения».

2.3.1 Структура экрана
Заголовок «Результаты извлечения».

Кнопка «Импортировать в библиотеку» (активна, когда хотя бы один атом переведён в статус «Подтверждён»).

Навигация по типам атомов: вкладки или вертикальный список слева. Для каждого типа указано количество извлечённых атомов (например, «Конфликты (3)»).

2.3.2 Таблица атомов выбранного типа
Колонки (пример для конфликтов):

Временный ID

Название

Суть (кратко)

Маски (перечисление)

Совместимости (перечисление ID)

Теги (облачко)

Статус (выпадающий список: Черновик / Подтверждён / К удалению)

При клике на строку справа (или в модальном окне) открывается форма редактирования атома.

Массовые действия над таблицей:

Кнопки «Подтвердить все», «Отклонить все» (переводят в статус «К удалению»), «Сгенерировать теги заново» (для выбранных или всех).

2.3.3 Карточка редактирования атома
Содержит все поля согласно типу атома (как определено в архитектуре библиотеки). Для конфликта:

name (текст)

essence (textarea)

dramatic_function (textarea)

phase_plan — редактируемый список фаз: каждая фаза имеет поля name, description. Можно добавлять/удалять фазы.

emotional_peaks — список строк, редактируемый (добавить/удалить).

compatible_twists, compatible_dilemmas, compatible_relationships — выпадающие списки с мультивыбором существующих ID (из основной библиотеки и предложенных ИИ). Возможен ручной ввод ID.

Теги — облачко тегов (из справочника Tag). Крестик для удаления тега, поле ввода с автодополнением для добавления.

Кнопки «Сохранить изменения» и «Сбросить».

Аналогичные формы — для остальных типов атомов с соответствующими полями.

2.3.4 Импорт в библиотеку
При нажатии «Импортировать в библиотеку»:

Все атомы в статусе «Подтверждён» получают постоянные ID (формат: CONF-XXX, TWIST-XXX и т.д., без конфликтов с существующими).

Данные записываются в соответствующие основные таблицы, маски добавляются в Mask (если новые) и связываются через ConflictMask, теги связываются через AtomTag.

Атомы в статусе «Черновик» или «К удалению» игнорируются.

После успешного импорта выводится уведомление «Успешно импортировано N атомов», кнопка «Сбросить и начать заново» возвращает к этапу 1.

3. Алгоритм обработки
3.1 Разбивка на фрагменты
Если исходный файл в формате FB2, epub или md — конвертировать в plain text (UTF-8).

Если включена опция «Разбивать по главам», найти в тексте паттерны заголовков глав (регулярные выражения, например Глава, Chapter, строки, начинающиеся с #). Разделить по этим точкам.

Для каждого полученного блока (главы) и для блоков без структуры (если опция выключена) проверить длину. Если блок превышает заданный лимит, разбить его на части по границам абзацев (двойной перевод строки) так, чтобы каждая часть не превышала лимит, но была максимально близка к нему.

Каждый фрагмент должен иметь порядковый номер и метаданные: исходная глава (если есть), диапазон символов в исходном тексте.

3.2 Пофрагментная экстракция
Для каждого фрагмента (последовательно или с ограниченным параллелизмом):

Отправить запрос к ИИ с системным промптом (из настроек) + текст фрагмента.

Ожидаемый ответ — JSON с полями по типам атомов: conflicts, twists, dilemmas, relationships, locations, sensory_hooks, npc_details. Формат каждого вложения соответствует структуре карточки, но без ID (используются временные ключи, уникальные в рамках ответа).

Сохранить полученные атомы в черновые таблицы (arc_draft_*) с привязкой к фрагменту и временным ID.

3.3 Постобработка
После обработки всех фрагментов:

Собрать все черновые атомы (всех типов) в один массив.

Отправить запрос к ИИ со вторым промптом (итоговый анализ) и этим массивом.

Ожидаемый ответ — JSON, содержащий:

deduplicated: массив атомов после удаления дубликатов (каждый атом дополнен полем merge_of — список временных ID, которые были объединены).

weak_candidates: список временных ID, рекомендованных к удалению как невыразительные.

compatibility_suggestions: для каждого атома (где применимо) предложенные списки compatible_* с ID из основной библиотеки (если ИИ их знает) или ссылками на временные ID других черновых атомов.

tags_suggestions: для каждого атома список рекомендованных тегов (строк), по возможности соответствующих существующему справочнику Tag.

Все предложения сохраняются в черновиках (поля arc_draft_* обновляются).

3.4 Действия администратора на этапе ревью
Администратор может изменить любой атом, его статус, теги, связи.

При изменении тегов, если введён новый тег, он проверяется по справочнику Tag, при отсутствии создаётся новая запись.

Все изменения сохраняются в таблицах черновиков с пометкой is_manual_edit = TRUE, чтобы предотвратить перезапись при повторной постобработке.

3.5 Импорт в основную библиотеку
Для подтверждённых атомов генерируются постоянные ID.

Данные переносятся в соответствующие основные таблицы (arc_conflict, arc_twist и т.д.).

Маски и теги синхронизируются со справочниками и связями.

5. API-методы (основные эндпоинты)
Все эндпоинты доступны только из админки (внутренний проект).

POST /api/admin/book/upload — загрузка файла, возвращает book_id.

GET /api/admin/book/{id}/progress — текущее состояние обработки.

POST /api/admin/book/{id}/start — запуск анализа с переданными настройками (размер фрагмента, булева опция, промпты). Промпты автоматически сохраняются в arc_prompts (если изменились).

POST /api/admin/book/{id}/abort — прервать обработку.

GET /api/admin/book/{id}/drafts?type=conflict — получить черновые атомы заданного типа.

PUT /api/admin/drafts/{type}/{id} — обновить черновой атом (сохранить изменения администратора).

PUT /api/admin/drafts/{type}/batch — массовое изменение статусов.

POST /api/admin/book/{id}/regenerate-tags — перегенерировать теги для выбранных черновиков.

POST /api/admin/book/{id}/import — импорт подтверждённых атомов в основную библиотеку.

GET /api/admin/tags?category=... — справочник тегов для автодополнения.

GET /api/admin/prompts — текущие промпты (для заполнения полей).

6. Обработка FB2
На стороне сервера при загрузке файла с расширением .fb2:

Прочитать XML, извлечь все текстовые узлы из элементов <body> (включая вложенные секции).

Объединить текст, восстановив разбиение на абзацы (каждый <p> на новой строке, раздел/глава — двойной перевод строки).

Сохранить результат как plain text, с которым далее работают алгоритмы разбивки.

При необходимости можно также извлечь метаданные (автор, название) для информативности, но не обязательно.

7. Ограничения и допущения
Максимальный размер загружаемой книги — 10 МБ (в пересчёте на plain text).

ИИ-модель и её параметры (температура, максимальное число токенов ответа) настраиваются в конфигурации сервера, не в админке.

Параллельная обработка фрагментов возможна для ускорения, но последовательный режим проще для контроля ошибок.

Атомы, предложенные к удалению, физически не удаляются до явного решения админа.

API POST /api/admin/book/upload, вероятно, вызывается, когда файл выбран? Но в описании API этот метод возвращает book_id. Это намекает, что загрузка происходит отдельно до анализа. Тогда этап 1 должен сначала загрузить файл (кнопка "Загрузить"), а затем отдельно "Анализировать". Но в UI просто одна кнопка. Возможна нестыковка: файл загружается сразу после выбора? Нужна ясность. Рекомендуется: при перетаскивании/выборе файла он еще не отправляется; при нажатии "Загрузить и анализировать" выполняется POST /api/admin/book/upload с файлом и настройками? Или upload возвращает book_id, а затем start запускает с настройками. Противоречие: описан метод POST /api/admin/book/upload возвращает book_id. Потом для запуска анализа используется POST /api/admin/book/{id}/start с настройками. Значит, веб-интерфейс должен сначала вызвать upload, получить book_id, а затем start. Но этап 1 в описании не упоминает получения book_id до нажатия главной кнопки; кажется, что upload вызывается внутри "Загрузить и анализировать". Надо уточнить.

Импорт перенесет данные из JSON в столбцы arc_conflict. Это норм. Но есть вопрос: в основной таблице arc_conflict поле compatible_twists JSONB и т.д., а в черновике хранение тоже JSONB. Карточка редактирования использует выпадающие списки с мультивыбором существующих ID. Это подразумевает запрос существующих ID из arc_twist, arc_dilemma, arc_relationship. Следовательно, в arc_draft_conflict JSON должен содержать эти поля. А при импорте они переносятся в соответствующие колонки arc_conflict. Все верно.

- Импорт в библиотеку: "Атомы в статусе «Подтверждён» получают постоянные ID (формат: CONF-XXX, TWIST-XXX и т.д., без конфликтов с существующими)." Как генерируется номер? Нужен автоинкремент или поиск максимального. Это должно быть реализовано.

- "Маски добавляются в Mask (если новые) и связываются через ConflictMask". В структуре БД есть arc_conflict_mask, но в черновике конфликта маски хранятся внутри data JSONB (поле масок?). В примере таблицы для этапа 3 колонка "Маски (перечисление)" — значит в JSON есть поле masks, возможно, список строк (кодов масок). При импорте нужно проверить каждый код маски по arc_mask, и если нет — создать? Но маски добавляются в Mask (если новые) — значит, нужно создать запись в arc_mask. В arc_mask код VARCHAR PK. Если маска новая, надо вставить. Но что если маска совпадает по названию, но кода нет? Придется сгенерировать код. В ТЗ не указан формат кода маски. Нужно уточнить: откуда берётся код новой маски? - В разделе "3.2 Пофрагментная экстракция: Ожидаемый ответ — JSON с полями по типам атомов: conflicts, twists, dilemmas, relationships, locations, sensory_hooks, npc_details. Формат каждого вложения соответствует структуре карточки, но без ID (используются временные ключи...)." Но в формате карточки конфликта есть совместимости (compatible_twists и т.д.), но на этапе экстракции фрагментов они не могут знать ID других атомов из других фрагментов. Поэтому на этом этапе совместимости, вероятно, не заполняются, либо ИИ может указать временные ключи внутри своего ответа. В постановке это не уточнено. В постобработке ИИ предоставит compatibility_suggestions с ссылками на временные ID. Следовательно, в этапе экстракции атомы должны иметь какие-то временные ключи, чтобы в постобработке на них ссылаться. Это противоречит "используются временные ключи, уникальные в рамках ответа". В пределах одного ответа они уникальны, но как потом ссылаться между разными фрагментами? Постобработка собирает все черновые атомы со всех фрагментов, и у каждого есть temp_id (видимо генерируется сервером, например, фрагмент+порядок). Тогда ИИ на постобработке должен оперировать этими temp_id. Но в ответе постобработки, как написано, "compatibility_suggestions: для каждого атома (...) предложенные списки compatible_* с ID из основной библиотеки (если ИИ их знает) или ссылками на временные ID других черновых атомов." Это значит, что поле compatible_twists в итоговых атомах будет содержать список ID, включая временные ID черновиков. При импорте эти временные ID должны быть заменены на постоянные ID тех атомов, которые были импортированы (если они тоже подтверждены). Это сложность, и она не раскрыта в алгоритме импорта. Нужно уточнить, что при импорте ссылки на временные ID должны быть преобразованы в постоянные ID для уже импортированных атомов. А если целевой атом не подтверждён (остался черновиком), что делать? Возможно, пропускать ссылку или оставлять как есть, но это неясно. В ТЗ не учтено разрешение таких ссылок. Также в интерфейсе редактирования на этапе 3 администратор может выбирать ID из существующей библиотеки или вводить вручную. Поле совместимости — выпадающие списки с мультивыбором существующих ID и "предложенных ИИ". Предложенные ИИ — вероятно, это временные ID черновиков, которые отображаются как элементы в списке. Администратор может их оставить. При импорте эти временные ID должны заменяться на постоянные у подтверждённых атомов. Нужно описать этот механизм явно.

- **Фоновые задачи:** встроенный `BackgroundTasks` + периодический опрос клиента; для длительной обработки – отдельный поток/процесс, управляемый `asyncio.to_thread` или планировщик задач (например, ARQ). В данном ТЗ используется подход с `BackgroundTask` при запуске анализа, который запускает асинхронную функцию обработки, обновляющую состояние в БД.
- **Конфигурация:** pydantic-settings, переменные окружения.
- **Логирование:** structlog или стандартный logging.

## 5. API-методы

Все эндпоинты находятся под префиксом `/api/admin`.

### 5.1. Книги

- **`GET /books`** — список книг с фильтрацией по статусу (`?status=uploaded`). Возвращает `[{id, original_filename, file_size, status, created_at}]`.
- **`POST /books/upload`** — загрузка файла. Принимает `multipart/form-data` с полем `file`. Создаёт запись `arc_book` со статусом `uploaded`, сохраняет файл на диск (или в облако). Возвращает `book_id`.
- **`GET /books/{id}`** — информация о книге, включая настройки (`settings`).
- **`POST /books/{id}/start`** — запуск анализа. Тело запроса:
  ```json
  {
    "fragment_size": 8000,
    "split_by_chapters": true,
    "extraction_prompt": "...",
    "postprocessing_prompt": "..."
  }
  ```
  Сохраняет промпты в `arc_prompt`, обновляет книгу (`settings`, статус `processing`), запускает фоновую задачу анализа.
- **`POST /books/{id}/abort`** — прервать обработку (статус `aborted`).
- **`DELETE /books/{id}`** — полное удаление книги, фрагментов, черновиков, состояния.
- **`POST /books/{id}/clear-drafts`** — удалить все черновики и фрагменты, статус книги → `uploaded`.

### 5.2. Прогресс и результаты

- **`GET /books/{id}/progress`** — текущее состояние обработки. Возвращает:
  ```json
  {
    "stage": "processing",
    "total_fragments": 20,
    "processed_fragments": 5,
    "error_message": null
  }
  ```
- **`GET /books/{id}/drafts?type=conflict`** — список черновых атомов заданного типа. Возвращает массив объектов с `id`, `temp_id`, `data`, `status`, `merge_of`, `weak_candidate`, `is_manual_edit`.

### 5.3. Редактирование черновиков

- **`PUT /drafts/{type}/{id}`** — обновить поля атома (изменённый `data`, статус). Тело: полный объект атома. Устанавливает `is_manual_edit = true`.
- **`PUT /drafts/{type}/batch`** — массовое изменение статусов. Тело: `{ "ids": [1,2,3], "status": "confirmed" }`.

### 5.4. Импорт и постобработка

- **`POST /books/{id}/import`** — импорт подтверждённых атомов в основную библиотеку. Запускает синхронную функцию импорта. Возвращает количество импортированных.
- **`POST /books/{id}/regenerate-tags`** — перегенерировать теги для выбранных черновиков (тело: `{ "draft_ids": [1,2] }`). Отправляет ИИ запрос с текущими данными атомов, обновляет теги.

### 5.5. Справочники

- **`GET /tags?category=general`** — список тегов с фильтром по категории.
- **`GET /prompts`** — возвращает текущие промпты (`[ {prompt_type, content} ]`).
- **`GET /library/search?type=conflict&q=любовь&book_id=5`** — поиск по основной библиотеке атомов указанного типа. Параметр `q` ищет по `name`. Если передан `book_id`, дополнительно возвращает временные ID черновиков этой книги. Ответ:
  ```json
  [
    {"id": "CONF-001", "name": "Конфликт любви", "source": "library"},
    {"temp_id": "TEMP-5-CONF-0001", "name": "Противостояние", "source": "draft"}
  ]
  ```

## 6. Процессы обработки

### 6.1. Загрузка файла

1. Клиент отправляет файл на `POST /books/upload`.
2. Сервер:
   - Валидирует расширение (.txt, .epub, .fb2, .md) и размер (<= 10 МБ plain text эквивалента; для fb2/epub проверяется после конвертации или исходный размер до 20 МБ).
   - Если формат fb2, epub, md — конвертирует в plain text (fb2 — извлечение из `<body>` по абзацам, epub — через библиотеку `ebooklib`, md — простая очистка от разметки). Для txt — сохраняет как есть.
   - Создаёт запись в `arc_book` с `original_filename`, `file_size` (размер plain text).
   - Сохраняет plain text в файл на диске или в базу (фрагменты потом будут хранить контент).
   - Возвращает `book_id`.

### 6.2. Запуск анализа (`start`)

1. Получает настройки. Сохраняет промпты (если не совпадают с текущими) в `arc_prompt`.
2. Обновляет `arc_book` — `status='processing'`, `settings`.
3. Создаёт запись в `arc_processing_state` с `stage='fragmentation'`, `total_fragments=0`.
4. Запускает асинхронную фоновую задачу `process_book(book_id)`.

### 6.3. Фоновая обработка книги

Поскольку обработка может быть длительной, она выполняется в отдельной асинхронной функции, которая не блокирует event loop, но может запускаться через `asyncio.create_task`. Для надёжности можно использовать фоновый планировщик, но в данном ТЗ используем встроенный механизм FastAPI `BackgroundTasks` (только для старта). Функция `process_book` реализует конечный автомат.

Стадии:
1. **Разбивка на фрагменты**:
   - Если `split_by_chapters` включён, ищет заголовки глав (регулярные выражения: `Глава \d+`, `Chapter \d+`, `# ` в начале строки и т.п.).
   - Разбивает текст на блоки. Каждый блок, если больше `fragment_size`, разрезается по границам абзацев (двойной \n) так, чтобы части были ≤ fragment_size.
   - Создаёт записи в `arc_fragment` со статусом `pending`, сохраняет `content`.
   - Устанавливает `arc_processing_state.stage='processing'`, `total_fragments = количество`.
2. **Пофрагментная экстракция** (с параллелизмом):
   - Использует настраиваемый уровень параллелизма `PARALLEL_FRAGMENTS` (из конфига, по умолчанию 1).
   - Создаёт семафор или использует `asyncio.gather` с ограничением.
   - Для каждого фрагмента:
     - Устанавливает статус `in_progress`.
     - Отправляет запрос к ИИ (модель, URL, ключ из конфигурации). Системный промпт = extraction_prompt, user_prompt = текст фрагмента.
     - Если запрос успешен, парсит JSON ответа, для каждого типа атомов создаёт записи в соответствующей `arc_draft_*`. Поля `temp_id` генерируются сервером: `TEMP-{book_id}-{TYPE_PREFIX}-{порядковый номер с ведущими нулями}`, где TYPE_PREFIX: CONF, TWIS, DILE, REL, LOC, SENS, NPC. Номер – порядковый номер атома данного типа в рамках всей книги (можно глобальный счётчик). Данные сохраняются в `data` (JSONB). `fragment_id` проставляется.
     - Если ошибка (сетевая, таймаут, невалидный JSON), увеличивает `retry_count`, при первой ошибке меняет статус на `pending` и повторяет попытку. При `retry_count >= 2` фрагмент помечается `error`, обработка останавливается (статус книги `error`, заполняется `error_message`). В этом случае все успешные фрагменты остаются в черновиках, книга доступна для возобновления? Логика: при ошибке процесс прекращается полностью, но запись книги `error`. Администратор может нажать «Продолжить» (перезапустить анализ), который начнёт обработку заново? Или лучше продолжить с того же места? По исходному требованию: «Если ошибка повторится - полная остановка». Дальнейшее действие: администратор может очистить черновики и начать заново. Поэтому при остановке не сбрасываем готовые фрагменты; реализуем `resume`, которая обрабатывает только фрагменты со статусом `pending` или `error` после сброса retry_count.
   - После обработки всех фрагментов переходит к постобработке.
3. **Постобработка**:
   - Устанавливает `stage='postprocessing'`.
   - Собирает все черновые атомы всех типов (поля `data`, `temp_id`) в один массив JSON.
   - Отправляет ИИ запрос с постобработочным промптом, передавая этот массив и, возможно, справочную информацию о существующей библиотеке (список ID и названий для ссылок). Ожидаемый ответ JSON:
     - `deduplicated`: массив объектов, каждый содержит исходный `data` (возможно обогащённый), `merge_of` (список temp_id, которые объединены), `weak_candidate` (bool).
     - `compatibility_suggestions`: объект, где ключ – temp_id, значение – предложенные совместимости в виде JSONB.
     - `tags_suggestions`: объект, ключ – temp_id, значение – массив строк (тегов).
   - Обновляет черновые атомы: для каждого атома из `deduplicated` создаёт новую запись в соответствующей черновой таблице (или обновляет существующую) с `merge_of`, `weak_candidate`, очищает старые дубликаты? По логике, старые дубликаты должны быть помечены как `deleted`. Алгоритм:
     - Помечаем все текущие черновики как `deleted` (или `draft` но неактивные)? Удобнее: создаём новый набор черновиков, старые удаляем (или переносим в архив). Так как у нас есть `merge_of`, администратор должен видеть исходные, но в требованиях: «информация о слиянии дубликатов скрыта от администратора. Это поле только для визуального отображения процесса работы во время обработки книги». Значит, после постобработки мы просто заменяем черновики: удаляем старые и вставляем новые, с полем `merge_of` (список поглощённых temp_id). На UI показываем это поле.
     - Для каждого нового атома проставляем предложенные совместимости (внутри `data`) и теги.
   - После завершения книга получает статус `awaiting_review`. `stage` = `awaiting_review`.

### 6.4. Возобновление обработки

Кнопка «Продолжить» для книги в статусе `uploaded` или `aborted` или `error`. При `error` можно выбрать: очистить черновики и начать заново или попытаться продолжить. Реализуем: `POST /books/{id}/start` с существующими настройками (если они есть) запускает `process_book` заново, но перед этим проверяет, есть ли фрагменты. Если есть — обрабатываем только `pending` и `error` (со сбросом `retry_count`). Если фрагментов нет — разбиваем заново. При ошибке остановки можно также добавить эндпоинт `resume`, который просто продолжает с места остановки.

### 6.5. Импорт подтверждённых атомов

Эндпоинт `/books/{id}/import`:
1. Загружаем все подтверждённые (`status='confirmed'`) черновые атомы.
2. Строим `map_temp_to_perm`: для каждого генерируем постоянный ID с префиксом типа и порядковым номером (можно использовать автоинкремент отдельной таблицы или вычислять `MAX(id) + 1`).
3. Вставляем в основные таблицы. Для связей масок и тегов создаём записи в `arc_conflict_mask`, `arc_atom_tag` и т.д. Если маска (code) не существует — создаём (`INSERT IGNORE` или проверка). Генерация кода маски: транслитерация name, обрезка до 50 символов, замена пробелов на `_`, приведение к верхнему регистру, добавление суффикса при необходимости.

4. Обработка совместимостей:
   - Проходим по всем JSONB полям `compatible_*`. Элементы массива могут быть как постоянными ID (из основной библиотеки), так и временными ID (черновики). Если временной ID присутствует в `map_temp_to_perm`, заменяем на постоянный. Иначе исключаем.
5. Импортированные черновики можно оставить или удалить. Оставляем со сменой статуса на `imported` (добавить в CHECK) или просто удаляем. По условию: атомы в статусе «Черновик» или «К удалению» игнорируются, но остаются. После импорта они не удаляются автоматически. Администратор может позже очистить черновики.
6. Книга переводится в статус `done`.

## 7. Параллельная обработка фрагментов

Конфигурационный параметр `PARALLEL_FRAGMENTS` (целое число, по умолчанию 1). Механизм:
- Внутри `process_book` после разбивки создаётся asyncio семафор.
- Для каждого фрагмента запускается корутина, ограниченная семафором.
- Каждая корутина независимо обновляет статус фрагмента, обрабатывает ошибки.
- Общий прогресс (`processed_fragments`) обновляется атомарно с помощью `UPDATE ... SET processed_fragments = processed_fragments + 1` или через очередь. В простом варианте можно увеличивать счётчик после успеха каждого фрагмента, используя асинхронную функцию, которая обновляет `arc_processing_state`.
- При ошибке одного фрагмента (после повторной попытки) останавливаем все запущенные задачи. Для этого можно использовать `asyncio.Event` или флаг, проверяемый в цикле.

## 9. Обработка ошибок ИИ и повторные попытки

Функция отправки запроса к ИИ (например, OpenAI API) должна быть обёрнута в retry-логику:
- Количество попыток: 2.
- Отлавливаем исключения: `httpx.HTTPError`, `asyncio.TimeoutError`, ошибки парсинга JSON.
- Повтор осуществляется немедленно или с небольшой задержкой.
- Если после повтора ошибка сохраняется, фрагмент помечается `error`, и вся обработка останавливается с установкой статуса книги `error` и записью сообщения.
