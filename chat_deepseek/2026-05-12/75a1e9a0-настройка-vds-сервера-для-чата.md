### 1. Как «войти» в твой сервер с компьютера (Windows)

Сервер — это мощный компьютер, который находится в дата-центре Hostland. Ты будешь управлять им со своего домашнего компьютера через волшебную программу, которая называется **SSH** (это «защищённый разговор» между твоим ПК и сервером). В Windows уже есть всё нужное, оно называется **PowerShell** (синее окошко, как командная строка, только круче).

**1.1. Открой PowerShell**
- Нажми кнопку **«Пуск»** в левом нижнем углу экрана.
- Напечатай прямо на клавиатуре слово **`powershell`** (без кавычек).
- В появившемся списке нажми на **«Windows PowerShell»** (обычно первая строка). Откроется тёмно-синее окно.

**1.2. Подключись к серверу**
В письме от Hostland у тебя есть:
- IP-адрес (например, `192.168.1.100` — я буду использовать `ТВОЙ_IP`)
- Логин: `root`
- Пароль: его ты впишешь позже

В окне PowerShell напечатай эту строку, заменив `ТВОЙ_IP` на настоящий IP-адрес из письма, и нажми **Enter**:
```powershell
ssh root@ТВОЙ_IP
```
*Пример:* если IP `11.22.33.44`, то команда будет `ssh root@11.22.33.44`.

**1.3. Первое знакомство (fingerprint)**
При первом подключении выскочит вопрос:
```
The authenticity of host '...' can't be established.
...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```
Это компьютер проверяет, что сервер — именно тот, за кого себя выдаёт. Смело печатай слово **`yes`** (именно так, слитно) и нажимай Enter.

**1.4. Ввод пароля**
Потом попросят пароль:
```
root@ТВОЙ_IP's password:
```
Теперь самое важное: **когда ты вводишь пароль, на экране ничего не появляется — ни звёздочек, ни точек. Так и должно быть.** Просто правильно набери пароль из письма и нажми Enter.

Если пароль не подходит, появится сообщение `Permission denied`. Проверь, не нажат ли **Caps Lock**, и попробуй ещё раз. Обычно даётся три попытки.

Если всё хорошо, ты увидишь приветствие и строчку, заканчивающуюся на `root@...:~#`. Ты внутри сервера! Можешь похлопать в ладоши.

### 2. Создаём безопасного пользователя (не root)

**Почему root опасен?**  
`root` — это как «директор», который может делать с сервером что угодно, даже случайно стереть всё системы. Поэтому мы создадим себе обычного пользователя с правами «помощника директора» (это называется **sudo**). И будем работать только под ним.

**2.1. Создай пользователя с именем `exis`**
Введи команду и нажми Enter:
```bash
adduser exis
```
Тебя попросят придумать пароль для `exis`. Придумай любой (можешь тот же, что от root, но лучше новый). Запомни его! Пароль опять будет невидимым. На все остальные вопросы (Full Name и т.д.) просто нажимай **Enter** — они не обязательны. В конце напиши `Y` и Enter, если спросит подтверждение.

**2.2. Дай пользователю `exis` права «помощника директора» (sudo)**
Выполни по очереди две команды:
```bash
usermod -aG sudo exis
```
Эта команда добавила `exis` в группу `sudo` (список тех, кому разрешено отдавать важные распоряжения).

**2.3. Переключись с `root` на `exis`**
Теперь выйди из root-сессии (выйти из сервера) — просто набери:
```bash
exit
```
Ты вернёшься в PowerShell. Теперь зайди на сервер заново, но уже под новым пользователем:
```powershell
ssh exis@ТВОЙ_IP
```
Введи пароль `exis`, который ты только что придумал. Поздравляю, ты на сервере в безопасности. С этого момента **всегда** используй `exis`, а не `root`. Если сервер попросит права root для команды, мы будем добавлять слово `sudo` в начале.

### 7. Автоматический запуск приложения (systemd-сервис)

**Что такое systemd?**  
Это «диспетчер задач» в Ubuntu, который умеет запускать программы и следить, чтобы они не падали. Мы создадим «службу» (сервис), которая будет сама поднимать твой FastAPI при включении сервера и перезапускать, если что-то случится.

**7.1. Создай файл службы**
```bash
sudo nano /etc/systemd/system/exis.service
```

**7.2. Вставь содержимое**  
Замени **`/home/exis/myapp`** на реальный путь к папке твоего проекта (если ты всё делал по шагам, это он и есть). Также обрати внимание на `main:app` — это означает: в файле `main.py` ищи приложение с именем `app`. У тебя может быть иначе: например, `server:app` или `app.main:app`. Узнай в своём коде: там должно быть что-то вроде `app = FastAPI()`. Если файл называется `app.py`, а переменная `app`, то пиши `app:app`. Вот готовый блок:
```ini
[Unit]
Description=Exis FastAPI Chat
After=network.target

[Service]
User=exis
Group=www-data
WorkingDirectory=/home/exis/myapp
EnvironmentFile=/home/exis/myapp/.env
ExecStart=/home/exis/myapp/venv/bin/uvicorn main:app --host 127.0.0.1 --port 8000
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```
Пояснения:
- `User` — от имени `exis`.
- `Group` — чтобы nginx имел доступ при необходимости.
- `WorkingDirectory` — откуда запускать.
- `EnvironmentFile` — подгрузит секреты из `.env`.
- `ExecStart` — запускает uvicorn внутри твоего venv, слушая порт 8000 **только локально** (127.0.0.1). Из интернета к этому порту не достучатся, запросы будет принимать nginx.
- `Restart=always` — если приложение упадёт, systemd поднимет его снова.

Сохрани: **Ctrl+O**, Enter, **Ctrl+X**.

**7.3. Включи и запусти сервис**
```bash
sudo systemctl daemon-reload
sudo systemctl enable exis
sudo systemctl start exis
```
- `daemon-reload` — перечитать все файлы служб.
- `enable` — разрешить запускаться при загрузке сервера.
- `start` — запустить прямо сейчас.

**7.4. Проверь, что всё работает**
```bash
sudo systemctl status exis
```
Ты должен увидеть зелёную строчку `active (running)`. Если что-то красное и `failed`, не пугайся — см. раздел 10 про ошибки.
Чтобы посмотреть лог в реальном времени:
```bash
sudo journalctl -u exis -f
```
(нажать Ctrl+C для выхода из лога)

### 8. Настройка Nginx («прокси»-сервер)

Nginx будет как секретарь в приёмной: встречает посетителей из интернета на 80 порту (http) и передаёт их запросы твоему приложению на порт 8000. А также умеет обрабатывать WebSocket-соединения (нужны для чата, чтобы сообщения летали мгновенно).

**8.1. Создай конфигурационный файл для твоего сайта**
```bash
sudo nano /etc/nginx/sites-available/exis
```

**8.2. Вставь настройки**
Замени `ТВОЙ_ДОМЕН_ИЛИ_IP` — если домена ещё нет, впиши IP-адрес сервера (тот самый, по которому подключался по SSH). Потом, когда появится домен, заменишь и перезагрузишь nginx.
```nginx
server {
    listen 80;
    server_name ТВОЙ_ДОМЕН_ИЛИ_IP;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```
Строчки `Upgrade` и `Connection` как раз разрешают WebSocket’ы, чтобы чат работал без перезагрузки страницы.

Сохраняем: **Ctrl+O**, Enter, **Ctrl+X**.

**8.3. Включи этот сайт (создай символьную ссылку)**
```bash
sudo ln -s /etc/nginx/sites-available/exis /etc/nginx/sites-enabled/
```

**8.4. Убери стандартную заглушку Nginx**
```bash
sudo rm /etc/nginx/sites-enabled/default
```
(вводить `y`, если спросит)

**8.5. Проверь конфигурацию на ошибки**
```bash
sudo nginx -t
```
Если выведет `syntax is ok` и `test is successful` — всё отлично. Если ругается — проверь, не пропустил ли ты точку с запятой в файле.

**8.6. Перезагрузи Nginx**
```bash
sudo systemctl reload nginx
```

Теперь твой FastAPI доступен снаружи по порту 80. Браузер сам использует этот порт, когда ты набираешь IP-адрес.

### Где искать «app = FastAPI()»?

FastAPI-приложение всегда начинается с создания экземпляра (копии) FastAPI. Эту копию обычно называют `app`, и она живёт в каком-то `*.py` файле (например, `main.py` или `app.py`). Строка выглядит примерно так:

1. **Имя файла** (без `.py`). Например, если файл `main.py`, то модуль называется `main`.

### Почему до сих пор просит пароль?

В главном файле `/etc/sudoers` ты добавил:
```
exis    ALL=(ALL) NOPASSWD: ALL
```
Но ниже есть строка:
```
%sudo   ALL=(ALL:ALL) ALL
```
`%sudo` означает «все, кто состоит в группе `sudo`». Когда `exis` входит в эту группу, к нему применяются **оба** правила: персональное (без пароля) и групповое (с паролем). Система читает правила по порядку, и **последнее подходящее** — групповое (потому что оно написано позже) — перебивает первое. Вот почему пароль всё ещё требуется.

Решение очень простое: убрать пользователя `exis` из группы `sudo`. Персонального правила `NOPASSWD` ему достаточно, а группа ему не нужна.

### Два типа пропусков (простыми словами)

Представь, что у пользователя `exis` сейчас есть два «удостоверения»:
1.  **Личная карточка** — строка `exis ALL=(ALL) NOPASSWD: ALL`, которую мы добавили в файл `sudoers` вручную. Она говорит: «Этот конкретный человек, `exis`, может всё, и пароль не нужен».
2.  **Значок группы `sudo`** — он даёт право на вход по строке `%sudo ALL=(ALL:ALL) ALL`. Эта строка говорит: «Все, у кого есть значок группы `sudo`, могут всё, но пароль спросить надо».

Охранник (система) проверяет документы по порядку. Сейчас он видит личную карточку («Ага, `exis` может без пароля»), но потом натыкается на значок группы, который требует пароль. По правилам, последнее, что он прочитал, идёт в дело. Поэтому пароль и спрашивается.

Если мы **уберём значок группы `sudo`**, останется только личная карточка. Охранник увидит только её — и будет пускать без пароля.

---

### Почему `exis` останется администратором

Право выполнять `sudo` даёт **запись в файле `sudoers`** (или в `sudoers.d`). Сейчас у нас есть явная запись:
```
exis    ALL=(ALL) NOPASSWD: ALL
```
Эта запись никуда не денется, даже если пользователь выйдет из группы `sudo`. Она — его личное разрешение.

**Проверка после удаления из группы:**
Мы можем убедиться уже сейчас, что права не пропадут. После выхода из группы `sudo` командой `sudo deluser exis sudo`, мы попросим систему заново прочитать правила:
```bash
sudo -k && sudo -n whoami
```
- `sudo -k` сбрасывает кэш паролей (чтобы проверить с чистого листа).
- `sudo -n whoami` попытается выполнить `whoami` без запроса пароля (`-n`).

Если после удаления из группы эта команда выведет `root`, значит, личная карточка работает, и он всё ещё администратор. Пароль не потребуется.
