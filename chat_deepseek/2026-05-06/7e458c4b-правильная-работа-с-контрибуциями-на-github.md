## 🧭 Основной принцип: «Форк — ветка — Pull Request»

Вы не работаете напрямую в оригинальном репозитории. Вы создаёте его копию, вносите правки у себя и предлагаете изменения автору проекта. Это называется **Fork & Pull Request**.

### 1. Сделайте форк репозитория
- На странице проекта нажмите кнопку **Fork** (в правом верхнем углу).
- У вас появится точная копия репозитория под вашим аккаунтом: `https://github.com/ваш-логин/название-проекта`.

### 2. Клонируйте форк на компьютер
```bash
git clone https://github.com/ваш-логин/название-проекта.git
cd название-проекта
```

### 3. Настройте связь с оригинальным репозиторием (upstream)
Это нужно, чтобы потом подтягивать свежие изменения из источника и избегать конфликтов.
```bash
git remote add upstream https://github.com/ОРИГИНАЛЬНЫЙ-ВЛАДЕЛЕЦ/название-проекта.git
git remote -v   # проверяем: origin → ваш форк, upstream → оригинал
```

### 4. Обновите свою основную ветку (main/master) до актуального состояния
Прежде чем начинать новую задачу, синхронизируйтесь с апстримом:
```bash
git checkout main         # перейти на основную ветку
git pull upstream main    # забрать все изменения из оригинала
git push origin main      # отправить актуальную версию в свой форк
```

### 5. Создайте отдельную ветку под задачу
**Никогда не работайте прямо в `main`!** Ветка должна иметь осмысленное название, отражающее суть правки.
```bash
git checkout -b fix-typo-readme
# или feat/add-dark-mode
```

## ✍️ Внесение изменений и коммиты

### Правила хороших коммитов
- Один коммит — одна логическая правка.
- Заголовок коммита на английском или языке проекта, в повелительном наклонении (imperative mood):  
  `Fix broken link in README`, `Add support for dark theme`.
- Длина заголовка — до 50–72 символов. Если нужно объяснить детали, оставьте пустую строку и напишите тело коммита.

Пример:
```
git commit -m "Fix typo in installation guide"

The previous command missed a slash in the path.
Updated the docs to match version 2.0 syntax.
```

- Перед коммитом проверяйте, что вы не закоммитили лишнее: `git status`, `git diff`.
- Следите за стилем кода, используйте линтеры, если проект их предоставляет.

## 🔄 Отправка в Pull Request

### 1. Отправьте ветку в свой форк
```bash
git push origin fix-typo-readme
```

### 2. Создайте Pull Request на GitHub
- Перейдите на страницу оригинального репозитория.
- Обычно сразу появляется зелёная кнопка **Compare & pull request** — нажмите её.
- Если кнопки нет, идите во вкладку **Pull Requests** → **New pull request**.  
  Выберите: base repository — оригинал, base branch — `main`, head repository — ваш форк, head branch — `fix-typo-readme`.
- Заполните описание:
  - Что сделано.
  - Зачем это нужно.
  - Ссылки на обсуждения, баг-репорты (issues), скриншоты до/после, если требуется.
  - Отметьте, если правка частично решает Issue (ключевые слова `Closes #123` автоматически закроют задачу после мёрджа).

### 3. Пройдите код-ревью
- Поддерживайте вежливый и конструктивный диалог.
- Если просят внести правки, сделайте их **в той же ветке**:  
  ```bash
  git add .
  git commit -m "Apply review feedback: fix variable naming"
  git push origin fix-typo-readme
  ```
  Pull Request автоматически обновится. Не создавайте новый PR для правок!

- При необходимости подтяните свежий `main` из апстрима и устраните конфликты.

## ✅ После принятия PR (мёрджа)
- Переключитесь на `main`, обновите его из upstream и удалите ветку, которая больше не нужна:
```bash
git checkout main
git pull upstream main
git push origin main
git branch -d fix-typo-readme        # удалить локально
git push origin --delete fix-typo-readme   # удалить в форке
```

1. **Читайте CONTRIBUTING.md**  
   Почти у каждого серьёзного проекта есть файл с правилами контрибуции. Там указаны требования к коду, коммитам, форматированию, тестам. Это экономит всем время.

   Убедитесь, что проект имеет открытую лицензию, и ваши правки будут приняты на соответствующих условиях.

6. **Терпение и уважение**  
   Мейнтейнеры часто волонтёры. Ответ может занять несколько дней. Будьте вежливы, и вам ответят взаимностью.

### RESPONSE (ИИ)

**Issues** (с англ. «вопросы», «проблемы», «задачи») — это встроенная в GitHub система отслеживания задач, ошибок, предложений и обсуждений. По сути, это «доска заявок» проекта, где ведётся вся организационная работа, не связанная напрямую с кодом.

В контексте контрибуций Issues выполняют несколько ключевых ролей:

### 1. Баг-репорты
Если кто-то нашёл ошибку, он открывает Issue с подробным описанием: что случилось, как воспроизвести, ожидаемое и реальное поведение. Разработчики и контрибьюторы видят список известных проблем и могут брать их в работу.

### 2. Предложения по улучшению (feature requests)
Пользователи или сами контрибьюторы предлагают новые функции, изменения интерфейса, рефакторинг. Здесь происходит первичное обсуждение: нужно ли это проекту, в каком виде.

### 3. Планирование и roadmap
Мейнтейнеры создают Issue для каждой планируемой фичи, распределяют их по вехам (Milestones), присваивают метки и проекты. Это превращает Issues в публичный план развития.

### 4. Поиск задач для новичков
Проекты часто помечают простые Issues специальными тегами:
- **`good first issue`** — задача, с которой можно начать знакомство с проектом.
- **`help wanted`** — мейнтейнеры просят помощи сообщества.
- **`beginner-friendly`** — аналогично.

В предыдущем ответе как раз рекомендовалось искать такие Issue для первых контрибуций.

### 5. Привязка к Pull Request
Самый частый способ закрыть Issue — это Pull Request. В описании PR пишут специальные ключевые слова: `Fixes #42`, `Closes #101`, `Resolves #7`. Когда PR принимают (мёржат), соответствующая Issue автоматически закрывается. Это связывает обсуждение задачи и код, который её решает.

### 6. Обсуждение до написания кода
Правило хорошего тона: **сначала Issue, потом PR**. Если вы задумали крупное изменение, откройте Issue, опишите идею, дождитесь комментариев от мейнтейнеров. Это экономит время: может оказаться, что фича не нужна или уже разрабатывается, и ваш код не пропадёт зря.

---

### Из чего состоит Issue:
- **Title** — краткий заголовок, отражающий суть.
- **Description** — поле ввода с Markdown-разметкой, куда можно вставить код, скриншоты, ссылки.
- **Assignees** — ответственный, кто работает над задачей.
- **Labels** — цветные метки для категоризации (`bug`, `enhancement`, `documentation`, `help wanted` и т.д.).
- **Milestone** — привязка к конкретной версии или этапу.
- **Linked pull requests** — показываются связанные PR.

В общем, **Issues — это центр коммуникации** между мейнтейнерами и сообществом. Для контрибьютора это стартовая точка: увидел интересный Issue → написал в комментарии, что возьмёшь его в работу → сделал форк, правки, PR с привязкой. Всё прозрачно, история сохраняется, и никто не делает двойную работу.

Похоже, вы хотите перейти к обсуждениям, связанным с конкретным файлом (или со всем проектом), но термин «ветка обсуждений» может сбивать с толку — в Git ветки (branches) это одно, а на GitHub обсуждения ведутся в **Issues** (задачи) и **Discussions** (форум).

## 1. Если вы открыли файл **на GitHub** и хотите создать обсуждение об этом файле

Вы можете открыть **Issue** (задачу), прямо указав в ней на этот файл или даже на конкретную строку.

- На странице файла в GitHub нажмите на номер строки слева (появится кнопка `...` или сразу ссылка). Удерживая Shift, можно выделить диапазон строк.
- После выделения строки/строк появится маленькое меню с иконкой трёх точек `...` (или синяя ссылка `...`). Нажмите её → выберите **Reference in new issue**.
- Откроется форма создания нового Issue, где в описании уже будет ссылка на этот файл и конкретные строки кода.

Если такой кнопки нет, просто скопируйте URL страницы (он будет содержать название файла и, возможно, строки в конце, например, `#L23-L45`) и вставьте в новое Issue вручную.

---

## 2. Если вы открыли файл локально (на своём компьютере) и хотите обсудить его

Тут два пути:

- **Найти все Issues проекта**, чтобы посмотреть, не обсуждается ли уже проблема с этим файлом.  
  → Откройте репозиторий на GitHub → вкладка **Issues** (вверху). Можете ввести название файла в строку поиска, чтобы отфильтровать задачи, где он упоминается.

- **Создать новое Issue** по поводу этого файла.  
  → В том же репозитории нажмите кнопку **New issue** (зелёная кнопка справа). Заполните заголовок и описание, при желании укажите путь к файлу и суть предложения/ошибки.

---

## 3. Возможно, вы слышали про **GitHub Discussions** (обсуждения)

Это аналог форума, вкладка **Discussions** есть не во всех репозиториях. Если она присутствует, туда тоже можно выносить вопросы, идеи, неофициальные разговоры. Откройте репозиторий → вкладка **Discussions** (рядом с Issues).

Если ваша цель — внести изменения и сделать вклад (контрибуцию), работать нужно в режиме **`Code`**. Вот как это сделать:

### 🎯 Главное: кнопка для редактирования

Находясь в любом из этих режимов просмотра, найдите в правом верхнем углу, над содержимым файла, **иконку карандаша** ✏️. Обычно она подписана **`Edit this file`** (Редактировать этот файл).

1.  Нажмите на этот карандаш.
2.  Файл откроется в режиме редактирования, и вы сможете вносить правки.
3.  После этого внизу страницы появится форма для описания ваших изменений и создания коммита (commit).

> Если вы работаете в чужом репозитории, GitHub, скорее всего, предупредит, что у вас нет прав на прямую запись. Он сам предложит **автоматически создать форк** и новую ветку для ваших правок одним нажатием кнопки — это самый простой и правильный путь.

### 🛠️ Что означают сами кнопки (Preview, Code, Blame)

Это просто переключатели внешнего вида файла, чтобы вам было удобнее с ним знакомиться:

| Кнопка | Режим | Для чего это нужно |
| :--- | :--- | :--- |
| **`Preview`** | **Предпросмотр** | Показывает, как будет выглядеть файл в «человеческом» виде. Актуален для Markdown (.md) и текстовых документов. Код отображается с форматированием (заголовки, списки, жирный шрифт). |
| **`Code`** | **Просмотр кода** | Показывает чистый исходный код файла с подсветкой синтаксиса. Это основной режим для программистов, чтобы изучать логику работы. **Редактировать файл можно и из режима `Code`, и из `Preview`.** |
| **`Blame`** | **История изменений** | Показывает файл построчно с информацией о том, **кто, когда и в каком коммите** последним менял каждую строчку. Это мощный инструмент для расследования: вы можете найти автора конкретной строки и понять логику её появления.[reference:0][reference:1] |

### ✍️ Важное замечание о контрибуции

Хотя кнопка `Edit this file` — это быстрый способ внести мелкие правки прямо на сайте, для серьёзной работы над задачами мы возвращаемся к правилам хорошего тона:

*   **Сначала Issue**: обсудите крупные правки с авторами.
*   **Работайте в отдельной ветке**: никогда не вносите изменения напрямую в `main`.
*   **Используйте Pull Request**: предлагайте свои изменения через PR, даже если создали их через веб-интерфейс.

Для небольших правок (например, опечаток в документации) веб-редактор — это нормальный инструмент. GitHub создаст для вас форк и ветку автоматически, когда вы сохраните изменения, а затем проведёт к созданию Pull Request.
