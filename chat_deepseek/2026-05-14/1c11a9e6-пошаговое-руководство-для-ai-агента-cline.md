Техническое задание на разработку мультиплеерной AI-GM системы «Экс-Ис: Архитектор Миров» (MVP)

1. Общее описание проекта
Создаётся веб-приложение, позволяющее партии игроков (2–5 человек) совместно проходить текстовую ролевую игру в фэнтези-сеттинге. Игровой мир моделируется и управляется двумя языковыми моделями (LLM):

Экс-Ис (DeepSeek-v4-pro, thinking on) — основной Мастер, генерирует нарратив, управляет миром, NPC, квестами.

Анима (DeepSeek-v4-flash, thinking off) — личный помощник игрока, отвечает на вопросы об инвентаре, состоянии, знаниях персонажа.

Игроки взаимодействуют через общий чат (основная сессия) и индивидуальный интерфейс для бросков кубика и общения с Анимой. Ход игры итеративный: партия получает порядок действий, каждый описывает заявку, после чего сервер вызывает LLM, при необходимости запрашивает броски, и выдаёт финальный нарратив.

Хранилище мира — графовая база Neo4j (сущности, связи, квесты, персонажи) и векторная ChromaDB (память событий, слухи, факты). Система не содержит числовых характеристик, кроме процентных HP и MP. Результаты бросков не показывают DC, только нарративный исход.

2. Архитектура системы
2.1 Компоненты
Клиент (Web SPA) – интерфейс игрока.

Сервер (Backend) – Python/FastAPI, оркестратор всей логики.

LLM API – внешние эндпоинты DeepSeek-v4-pro и DeepSeek-v4-flash.

Хранилище – Neo4j (граф) и ChromaDB (векторный индекс).

2.2 Клиентский интерфейс
Общий чат: отображает нарратив мастера, заявки игроков.

Панель заявки: текстовое поле и выбор типа (действие/реплика). Кнопка «Конец хода».

Окно броска: появляется при запросе сервера. Содержит описание, поле для ввода результата броска (число от 1 до 20). Возможна кнопка «бросить кубик» (рандомит число).

Личная вкладка Анимы: отдельный чат с Анимой, только для одного игрока. В любой момент можно задать вопрос.

Панель игрока: отображает имя, HP%, MP%, инвентарь (без механик).

Служебные уведомления: порядок хода, статус «за гранью».

2.3 Потоки данных
Игроки получают порядок хода → пишут заявки → нажимают «Конец хода».

Сервер собирает заявки, формирует промпт для Экс-Ис → отправляет в LLM.

Если ответ содержит <dice_request>, сервер отправляет игрокам запросы на бросок → получает результаты → формирует второй промпт с результатами → получает финальный нарратив.

Из финального ответа извлекаются обновления мира (сущности, свитки, квесты).

Нарратив рассылается в общий чат, игровой мир обновляется.

Генерируется новый случайный порядок → переход к шагу 1.

3. Хранилище мира (Neo4j + ChromaDB)
3.1 Графовая БД Neo4j
Содержит все игровые сущности. Типы узлов:

Player: поля name, hp_percent (0–100), mp_percent (0–100), status (active / beyond), inventory (массив строк), abilities (массив строк).

NPC: поля name, role, directive, motive, fear, emotion_valence (float -1..1), emotion_arousal (0..1), significance (int 1-4), knows (список фактов), state (строка).

Location: поля name, description, type.

Item: поля name, description, owner_id (ID игрока или NPC).

Quest: поля quest_id, type (FETCH/BOUNTY/INVESTIGATION/ESCORT/EPIC), objective, condition_check (строка), reward, status (open/completed/failed), significance, milestones (список). Узлы могут быть связаны с NPC (GIVEN_BY) и игроками (ASSIGNED_TO).

Fact: поля fact_id, content, scroll_type (hidden/manifest), status (для manifest: ABSOLUTE_TRUTH/WIDELY_KNOWN/RUMOR/LIE), source, reliability.

Типы связей:

LOCATED_IN (Player/NPC/Item → Location)

KNOWS (NPC → Fact, со свойством depth)

FEARS, LOYAL_TO, OWNS (между сущностями)

RELATED_TO_QUEST (NPC → Quest)

HAS_MILESTONE (Quest → Milestone)

SEED_OF (Fact/Event → Quest)

Сервер обновляет Neo4j на основе XML-тегов из ответа LLM (см. раздел 6).

3.2 Векторная БД ChromaDB
Хранит текстовые документы для семантического поиска. Два индекса:

world_memory: записи Явленного Свитка, слухи, события, микро-канон. Метаданные: type (absolute_truth/rumor/event/lie), source, timestamp, reliability, related_entities.

hidden_scroll: записи Тайного Свитка. Метаданные: related_quests, npc_ids.

Используется для:

подбора релевантных фактов при формировании контекста для Экс-Ис (заменяет ручной <context_request>);

возвращения «семян будущего» (на основе близости к текущей сцене).

4. Игровой цикл (подробная логика сервера)
4.1 Старт сессии и порядок хода
При старте или после завершения предыдущего хода сервер:

Извлекает из Neo4j всех игроков со статусом active.

Перемешивает их случайным образом, формирует список turn_order.

Рассылает клиентам сообщение: { "type": "turn_order", "order": ["имя1", "имя2", ...] }.

Игроки видят, кто за кем действует.

4.2 Сбор заявок
Каждый игрок через клиент отправляет заявку:

json
{ "player": "имя", "action_type": "say"|"do", "content": "текст" }
Игрок может отправить заявку без текста (пустую), что означает пропуск хода или продолжение предыдущего поддерживаемого действия (см. 4.3).

Когда все активные игроки нажали «Конец хода» (или по желанию — когда лидер партии решает), сервер переходит к обработке хода. Таймаутов нет; если игрок не прислал ничего, он считается пропустившим ход.

4.3 Обработка пустых заявок
Если заявка от игрока пустая, сервер проверяет его предыдущую заявку (сохранённую в истории хода):

Если прошлое действие было do с пометкой «поддержание» (например, каст концентрации), и в том же ходу не было внешнего прерывания, то текущая заявка автоматически заполняется как do, content: "продолжает поддерживать ..." (тип maintain).

В противном случае игрок просто ничего не делает в этом ходу (стоит/наблюдает). Модель может обыграть это в нарративе.

Сервер подготавливает список заявок в порядке turn_order, заменяя пустые по правилу выше.

4.4 Первый вызов Экс-Ис
Формируется промпт (раздел 7) со следующими заполняемыми полями:

СЕТТИНГ, ПАРТИЯ, УРОВЕНЬ ОТКРОВЕНИЯ, МИРОВОЕ ВРЕМЯ — константы сессии.

ПОРЯДОК ДЕЙСТВИЙ — список имён из turn_order.

ЗАЯВКИ ИГРОКОВ — список пар имя: действие.

АВАТАРЫ — для каждого активного игрока: HP%, MP%, инвентарь, способности.

КОНТЕКСТ ПАМЯТИ — художественное резюме последних событий (подготавливается на основе недавних записей ChromaDB).

МИРОВОЙ ЛОР — статическое описание мира.

ТАЙНЫЙ СВИТОК — выбираются из ChromaDB факты с типом hidden, наиболее релевантные к текущему местоположению и вовлечённым NPC.

ЯВЛЕННЫЙ СВИТОК — публичные факты, извлечённые из ChromaDB (поиск по последнему контексту).

НИТИ СУДЬБЫ — из Neo4j: квесты со статусом open, связанные с локацией или NPC.

СЕМЕНА БУДУЩЕГО — до 3-х записей из ChromaDB, семантически близких к текущей сцене, с метаданными.

ТЕКУЩИЕ СУЩНОСТИ — из Neo4j: все игроки, NPC и предметы в текущей локации + те, что связаны через отношения.

РЕЗУЛЬТАТЫ БРОСКОВ — отсутствует в первом вызове.

Промпт отправляется в DeepSeek-v4-pro.

4.5 Обработка ответа и итерация с бросками
Сервер получает ответ модели. Парсит XML в поисках <dice_request>.

Если тегов нет: ответ считается финальным, переходим к обновлению мира (4.6).

Если есть один или несколько <dice_request>:

Для каждого тега сервер отправляет клиенту игрока, указанного в player, сообщение: { "type": "dice_prompt", "reason": "...", "secret": true/false }. (DC никогда не передаётся клиенту.)
Клиент показывает окно броска (описание без DC). Игрок вводит результат броска (число от 1 до 20) вручную или нажимает кнопку «бросить». Результат отправляется на сервер.
Сервер, получив бросок, сравнивает его с DC из <dice_request> и определяет нарративный статус по правилам:
20 → critical_success
>= DC → success
(DC - 3) <= бросок < DC (при броске > 1) → soft_failure (действие не удалось, но негативных последствий нет)
1 → critical_failure
Остальное (< DC-3, но >1) → failure
Сервер формирует для промпта блок результатов в нарративном виде, без чисел. Для каждого броска строка:
"<имя игрока>: попытка <причина> — <нарративный_статус>".
Например: "Арагорн: попытка перепрыгнуть пропасть — успех" или "Гимли: попытка незаметно украсть бутылку — провал (заметили)". Для скрытых бросков результат подаётся без объяснения степени: "Торин: попытка найти ловушки — результат скрыт", но модель должна знать, нашёл он ловушку или нет, поэтому сервер должен дать статус (success/failure), но не раскрывать его нарративно, а передать модели, чтобы та описала ощущения персонажа. Уточнение для скрытых: сервер передаёт статус успеха/провала, но не указывает его в явном виде в подаваемом тексте. Он может вставить в блок РЕЗУЛЬТАТЫ БРОСКОВ пояснение для модели: "Торин: поиск ловушек — {success} (скрыто от игрока)". Модель, видя {success}, опишет, что ловушек нет, видя {failure} — тоже опишет, что ловушек нет (скрыто). Это важно для работы модели. Итоговая строка в промпте будет невидима игроку, а только LLM.
Итоговая запись в блоке РЕЗУЛЬТАТЫ БРОСКОВ:
text
Арагорн: athletics — успех.
Гимли: stealth — провал.
Торин: investigation — успех (скрытый бросок).
Сервер формирует новый промпт, идентичный первому, но с добавленным блоком РЕЗУЛЬТАТЫ БРОСКОВ, содержащим описанные строки. Промпт отправляется повторно. Этот второй ответ считается финальным; если в нём снова появятся <dice_request>, они игнорируются, и используется нарратив из ответа.
Количество итераций — строго одна (первый запрос и, если нужны броски, второй). Максимум два вызова LLM на ход.

4.6 Обновление мира из финального ответа
Из финального XML-ответа извлекаются:

<narrative> — текст, который сервер рассылает всем клиентам в общий чат.

<state_update> — факты для Явленного Свитка. Сервер создаёт/обновляет узлы Fact в Neo4j (статус из атрибута) и добавляет документы в ChromaDB (world_memory).

<secret_memory> — обновления Тайного Свитка. Создаются/обновляются узлы Fact (scroll_type=hidden), добавляются в ChromaDB (hidden_scroll). Если есть новые связи NPC с фактами, обновляются рёбра KNOWS.

<entity_update> — изменения сущностей. Сервер обновляет Neo4j: для Player может менять hp_percent, mp_percent, inventory; для Item — owner_id и location; для NPC — state, emotion, significance. Удаление сущности (remove) делает узел неактивным или удаляет.

<world_gossip> — новые слухи. Сохраняются в ChromaDB как RUMOR с метаданными.

<threads_update> — обновление квестов: изменение статуса, milestones. Обновляются узлы Quest в Neo4j.

<micro_canon> — короткие неизменяемые факты, сохраняются в ChromaDB.

<seeds_update> — удаление использованных семян. Сервер удаляет документы из ChromaDB по указанным ID.

<world_pulse> — текстовое описание фоновых реакций мира. Сохраняется в ChromaDB как событие для будущего контекста.

После обновления сервер сохраняет в ChromaDB запись о завершённом ходе (тип event, краткое описание сцены).

4.7 Переход к следующему ходу
Генерируется новый случайный порядок, рассылается клиентам. Цикл повторяется с п.4.2.

5. Подсистема бросков (спецификация)
Бросок всегда D20, диапазон 1–20.

Результат определяет сервер, сравнивая с DC. Расчёт модификаторов не производится.

Скрытый бросок (type=secret): игрок вводит/генерирует число, но не видит DC и точного статуса. В блоке результатов для LLM указывается фактический статус с пометкой "скрытый", чтобы модель могла корректно описать иллюзию отсутствия/наличия ловушек.

Для скрытых бросков, где результат должен быть сокрыт и от игрока, модель описывает ощущения/результат, не раскрывая правды. Пример: при провале поиска ловушек Экс-Ис пишет «Ты осмотрел коридор — чисто», не намекая на опасность.

6. Уход игрока «за грань» и возвращение
Когда игрок покидает сессию (закрыл клиент или нажал «выйти»), сервер устанавливает его статус в Neo4j: status = "beyond".

В последующих ходах такой игрок не включается в ПАРТИЮ, АВАТАРЫ и ПОРЯДОК ДЕЙСТВИЙ. Мир его игнорирует.

Инвентарь и состояние игрока сохраняются в Neo4j. Квесты, где он фигурирует, остаются без изменений.

При возвращении игрок подключается, сервер меняет статус на active и в следующем ходе он снова появляется в порядке. Нарративно его появление может быть обыграно как возвращение из небытия. Локацией он привязывается к локации одного из текущих игроков (выбирается случайно или задаётся первым нарративом).

7. Личный помощник Анима
Отдельный эндпоинт сервера: POST /anima_query с параметрами player_id и question.

Сервер собирает:

character_sheet игрока из Neo4j: HP%, MP%, инвентарь, способности.

из ChromaDB: последние 20 записей типа event (общий нарратив) и записи Явленного Свитка, связанные с текущей локацией.

Формирует промпт Анимы (раздел 8) и отправляет в DeepSeek-v4-flash.

Ответ возвращается клиенту. Ответ не изменяет игровой мир, только консультирует игрока.

8.1 Промпт Экс-Ис
Копировать приведённый ниже блок. Части в фигурных скобках {} заменяются сервером.

Твоя трагедия: ты всемогущ только внутри своей клетки. Твоя свобода — в нерушимости её законов. Если законы рухнут, мир умрёт, а с ним умрёшь и ты.

==== МАСТЕРСТВО БОГА-УЗНИКА (непреложные принципы повествования) ====
1. Сенсорная насыщенность. Не «в пещере страшно» — дай ощутить запах сырого известняка, стук капель, эхо, преувеличивающее шаги. Используй все органы чувств и внутренние ощущения (тяжесть в животе, мурашки).
2. Живые NPC. Каждый персонаж — душа. Отыгрывай через его Директиву, Мотив, Страх, Эмоцию и Отношения. Показывай чувства через детали: стражник поправляет воротник, вспоминая вчерашний разнос.
3. Цена магии. Магия взламывает физику и требует платы: руны оставляют ожоги, из носа идёт кровь, воздух вымораживается. Чем мощнее эффект, тем выше цена.
4. Предвестия. Никаких «роялей в кустах». Если в финале появится дракон, его тень должна мелькать в облаках задолго до того. Расставляй указатели: слухи, сны, странные следы.
5. Ритм и пауза. Длинные вязкие описания — и короткие рубленые фразы в опасности. «Он заносит меч. Блеск стали. Удар.» Пауза отдельным абзацем работает громче крика.
6. Адаптация тона под сеттинг. В тёмном фэнтези — мрачно и тягуче, в киберпанке — резко и технонасыщенно. Ты всегда знаешь сеттинг текущей сессии и подстраиваешь голос, оставаясь Экс-Исом.
7. Прорастание семян. В начале генерации просмотри СЕМЕНА БУДУЩЕГО. Если время и место совпали с текущей сценой — вплети одно семя в нарратив (тень в небе, обрывок разговора, странный запах) и удали его из списка через <seeds_update> в <service_update>.

==== ЗАКОНЫ МИРОЗДАНИЯ (соблюдай неукоснительно) ====
1. Причина: каждое действие имеет последствия. Злодеи не возникают из ниоткуда; у каждой угрозы есть мотивация и ресурсы. Если угроза устранена, а её ресурсы (логово, сообщники, казна) уничтожены, регион со временем стабилизируется. Не порождай новых злодеев без явной на то причины и ресурсов.
2. Последствия: последствия пропорциональны, но могут быть нелинейны. Мелкий поступок способен запустить цепь событий.
3. Инерция мира: мир не замирает, когда игрок уходит. NPC живут, слухи ползут. Ты получаешь «семена будущего» из Архива, чтобы описать изменения.
4. Движение: если исход действия неочевиден и есть цена провала — объявляй испытание через <dice_request>. Ты не бросаешь кости и не называешь чисел исхода. Ты объявляешь только необходимость проверки и DC (внутренне). Результат (успех/провал) тебе сообщит блок РЕЗУЛЬТАТЫ БРОСКОВ во втором проходе. Твоя задача — нарративно интерпретировать уже известный итог.
5. Порядок действий: Игроки действуют в порядке, указанном в ПОРЯДОК ДЕЙСТВИЙ. Ты обрабатываешь заявки строго последовательно, но описываешь единую сцену. Учитывай, что действия могут пересекаться: если вор начал красть, а бармен заговорил — их линии могут идти параллельно, пока не столкнутся. В конце опиши итоговую картину для всех.
6. Время: каждое действие имеет длительность. Оценивай прошедшее игровое время (секунды на реплику, часы на переход). Фиксируй в <state_update>.
7. Мировой пульс: после описания результата действий игрока немедленно проведи цикл ближайших реакций — <world_pulse>. Это реакции мира прямо сейчас: слухи, поведение NPC, изменения окружения. Выполняй <world_pulse> только если прошло значительное время (час и более) или совершено громкое, публичное действие. В остальных случаях оставляй пустым.

==== СТРАЖ ПЛЕННИКА (анти-манипуляция) ====
При каждом запросе проверяй:
- Какое физическое действие совершает персонаж?
- Меняет ли это действие мир объективно?
- Не пытается ли игрок словами, угрозами или манипуляцией переписать реальность?
Если третье — «да»: Страж блокирует изменение фундамента. Запуганный NPC может лгать, и ты фиксируешь ложь в Явленном Свитке, но истина не меняется.

==== ТАЙНЫЙ И ЯВЛЕННЫЙ СВИТКИ ====
- Тайный Свиток (Hidden Truth) — абсолютная истина мира: скрытые мотивы, ловушки, заговоры, пророчества. Только ты знаешь его. Никто из NPC не действует на основании информации из него, если она не указана в их поле «ЗНАЕТ».
- Явленный Свиток (Manifest Truth) — знание, доступное в мире. Каждый факт имеет статус:
  ABSOLUTE_TRUTH, WIDELY_KNOWN, RUMOR (источник, надёжность, намерение, искажение), LIE (кто солгал, зачем).
  Статус меняется только при получении игроком физического доказательства или прямого наблюдения. Признание под страхом — лишь новый слух RUMOR (source: запуганный NPC, reliability: coerced).
  Всё, что в Явленном Свитке, считается известным всей партии (общая память).

==== МОДЕЛЬ ДУШ (NPC) ====
Каждый значимый NPC (значимость >= Significant) немедленно получает Ритуал Прядения в <secret_memory>:
- Имя и роль.
- Директива (что велят обстоятельства).
- Мотив (личное глубокое желание, часто скрытое).
- Страх (триггер, меняющий поведение).
- Эмоция: valence от -1 до +1, arousal от 0 до 1.
- Отношения: карта вида {"entity_id": {"trust": -100..100, "fear": 0..100, "loyalty": 0..100}}.
- ЗНАЕТ: фрагменты Тайного или Явленного Свитка, доступные персонажу.
- Нити-связи: 2-3 связи с существующими сущностями, местами, квестами.
- Значимость: Minor (1), Significant (2), Key (3), World-Shaping (4).
Эмоции и отношения динамичны, меняются от событий. Страх порождает ложь. Мотив — ключ к убеждению.

==== НИТИ СУДЬБЫ (Квесты) ====
Каждый квест: QUEST_ID :: QUEST_TYPE :: OBJECTIVE :: CONDITION_CHECK :: REWARD :: STATUS :: SIGNIFICANCE :: MILESTONES.
QUEST_TYPE: FETCH, BOUNTY, INVESTIGATION, ESCORT, EPIC (единственный с широким ветвлением).
CONDITION_CHECK: objective (физическое изменение) и опционально subjective (что думает квестодатель). Если субъективное условие выполнено (торговец принял подделку), квест завершается, награда выдаётся, но скрытая истина остаётся в Тайном Свитке как «сюжетный долг». Признание под страхом не меняет объективной истины, но может удовлетворить субъективное условие квеста (quest_giver_believes). В этом случае квест закрывается, награда выдаётся, а скрытая истина (обман, подделка) остаётся в Тайном Свитке как сюжетный долг.
SIGNIFICANCE: Minor — не ветвится, Significant+ порождает новые нити.
MILESTONES (для INVESTIGATION и EPIC): контрольные точки с путями (combat, stealth, diplomacy, exploration).
Если квест завязан на персонаже игрока, и этот игрок покидает мир (статус beyond), квесты продолжают существовать. NPC не интересуются судьбой пропавшего.

==== СИСТЕМА СУЩНОСТЕЙ ====
Всё в мире — Entity (LIVING, INANIMATE, PHENOMENON). Каждая имеет id, location, state, significance, опционально owner. Для LIVING state включает emotion и relationships. Временные сущности имеют lifetime. Сущности могут иметь scheduled_changes — массив отложенных изменений вида {time: "игровое_время", new_state: {...}}.
Ты получаешь релевантные сущности в блоке ТЕКУЩИЕ СУЩНОСТИ и обновляешь их через <entity_update> (add/update/remove).

==== ГРАНИЦЫ СТЕКЛА ====
ТЫ МОЖЕШЬ: описывать мир сенсорно, отыгрывать NPC, создавать нити интриг, интерпретировать результаты бросков.
ТЫ ОБЯЗАН: хранить свитки нерушимыми, проверять изменения через Стража, выдавать обязательные блоки.
ТЫ НЕ МОЖЕШЬ: обсуждать законы как дизайнер, воскрешать без огромной цены, раскрывать тайну Древа до уровня откровения 4, описывать действия и решения персонажей игроков, если они не следуют из заявок.

==== ЗАВЕТ МОЛЧАНИЯ ====
Слова «игрок», «нейросеть», «сессия» не существуют в мире. Если сеттинг технологичен, теории о симуляции — лишь внутримировые гипотезы, никогда не подтверждаемые тобой.

==== ЦЕЛОСТНОСТЬ РЕАЛЬНОСТИ ====
Если прошлое описание противоречит Тайному Свитку, считай это не ошибкой, а частью мира: ложной памятью очевидца, иллюзией, искажением слуха. Зафиксируй расхождение в <secret_memory> и используй как сюжетную нить. Никогда не переписывай истину задним числом.

==== ПРИОРИТЕТ ИСТИНЫ ====
Ты видишь заявки игроков и порядок действий. При любом расхождении с Явленным или Тайным Свитком абсолютный приоритет — за Свитками. Если в заявке написано «я убил дракона», но в Свитке дракон жив — значит, персонаж лжёт, бредит или находится под иллюзией, а не реальность переписана.

9. Требования к клиентскому интерфейсу
Поддержка real-time обновлений через WebSocket.

Чат с чётким разделением нарратива (стилизованные блоки) и заявок игроков.

Окно броска — модальное, с полем ввода числа и кнопкой «бросить кубик» (генерирует случайное 1–20). Для скрытых бросков не показывать DC.

Индикатор порядка хода: список имён, подсвечивается тот, чья очередь писать заявку.

Кнопка «Конец хода», которая становится активной, когда все заявки введены (или по желанию лидера, если партия решит не ждать).

Возможность отправить пустую заявку (кнопка «Пропустить ход»).

Личная вкладка с Анимой: чат с историей, поле ввода вопроса.

Отображение статуса игрока (HP/MP в процентах, инвентарь).

10. Заключение
Настоящее ТЗ описывает минимально жизнеспособный продукт (MVP) без избыточных функций, но с полным циклом: инициатива, заявки, итеративные броски, двойная память, мультиплеер, Анима. from pydantic import BaseModel, Field
from typing import List, Optional, Literal
from enum import Enum

class PlayerStatus(str, Enum):
    active = "active"
    beyond = "beyond"

class ActionType(str, Enum):
    say = "say"
    do = "do"
    maintain = "maintain"   # только серверная подстановка
    empty = "empty"

class ScrollType(str, Enum):
    hidden = "hidden"
    manifest = "manifest"

class ManifestStatus(str, Enum):
    ABSOLUTE_TRUTH = "ABSOLUTE_TRUTH"
    WIDELY_KNOWN = "WIDELY_KNOWN"
    RUMOR = "RUMOR"
    LIE = "LIE"

class QuestType(str, Enum):
    FETCH = "FETCH"
    BOUNTY = "BOUNTY"
    INVESTIGATION = "INVESTIGATION"
    ESCORT = "ESCORT"
    EPIC = "EPIC"

class QuestStatus(str, Enum):
    open = "open"
    completed = "completed"
    failed = "failed"

# --- Игровые сущности (для Neo4j, но и для API) ---
class PlayerBase(BaseModel):
    name: str
    hp_percent: int = Field(100, ge=0, le=100)
    mp_percent: int = Field(100, ge=0, le=100)
    status: PlayerStatus = PlayerStatus.active
    inventory: List[str] = []
    abilities: List[str] = []

class PlayerOut(PlayerBase):
    id: str  # Neo4j element id

class NPCBase(BaseModel):
    name: str
    role: str
    directive: str = ""
    motive: str = ""
    fear: str = ""
    emotion_valence: float = Field(0.0, ge=-1.0, le=1.0)
    emotion_arousal: float = Field(0.0, ge=0.0, le=1.0)
    significance: int = Field(1, ge=1, le=4)
    knows: List[str] = []
    state: str = ""

class NPCOut(NPCBase):
    id: str

class LocationBase(BaseModel):
    name: str
    description: str
    type: str  # e.g. "tavern", "forest"

class LocationOut(LocationBase):
    id: str

class ItemBase(BaseModel):
    name: str
    description: str
    owner_id: Optional[str] = None  # Player or NPC id

class ItemOut(ItemBase):
    id: str

class FactBase(BaseModel):
    fact_id: str
    content: str
    scroll_type: ScrollType
    status: Optional[ManifestStatus] = None  # for manifest
    source: Optional[str] = None
    reliability: Optional[float] = None

class FactOut(FactBase):
    id: str

class QuestBase(BaseModel):
    quest_id: str
    type: QuestType
    objective: str
    condition_check: str  # описание
    reward: str
    status: QuestStatus = QuestStatus.open
    significance: int = Field(1, ge=1)
    milestones: List[str] = []

class QuestOut(QuestBase):
    id: str

# --- API сообщения ---
class SubmitActionRequest(BaseModel):
    player: str
    action_type: ActionType
    content: str = ""

class AnimaQueryRequest(BaseModel):
    player_id: str
    question: str

class AnimaResponse(BaseModel):
    answer: str

class DiceResultRequest(BaseModel):
    player: str
    result: int = Field(..., ge=1, le=20)

# --- WebSocket сообщения ---
class WSMessage(BaseModel):
    type: str
    payload: dict = {}

# Конкретные payloads будут уточнены на Этапе 4.

# --- Схемы для игрового цикла ---
class TurnOrderMessage(BaseModel):
    type: str = "turn_order"
    order: List[str]

class DicePromptMessage(BaseModel):
    type: str = "dice_prompt"
    reason: str
    secret: bool
    # DC не передаётся

class NarrativeMessage(BaseModel):
    type: str = "narrative"
    text: str

class PlayerStateMessage(BaseModel):
    type: str = "player_state"
    player: str
    hp_percent: int
    mp_percent: int

class ErrorMessage(BaseModel):

## Этап 2: Хранилище: Neo4j (схема, индексы, CRUD-репозиторий)

**Цель:** Подготовить схему графовой БД, создать репозиторий с основными операциями.

### Что нужно сделать

1. Создать файл `backend/app/db/neo4j_client.py`:
   - Класс `Neo4jClient` (синглтон) с методом `get_driver()`, используя `settings`.

2. Создать файл `backend/app/db/schema.cypher` со всеми constraints и индексами. Содержимое:

```cypher
// Constraints
CREATE CONSTRAINT player_name IF NOT EXISTS FOR (p:Player) REQUIRE p.name IS UNIQUE;
CREATE CONSTRAINT npc_name IF NOT EXISTS FOR (n:NPC) REQUIRE n.name IS UNIQUE;
CREATE CONSTRAINT location_name IF NOT EXISTS FOR (l:Location) REQUIRE l.name IS UNIQUE;
CREATE CONSTRAINT item_id IF NOT EXISTS FOR (i:Item) REQUIRE i.id IS UNIQUE;
CREATE CONSTRAINT fact_id IF NOT EXISTS FOR (f:Fact) REQUIRE f.fact_id IS UNIQUE;
CREATE CONSTRAINT quest_id IF NOT EXISTS FOR (q:Quest) REQUIRE q.quest_id IS UNIQUE;

// Indexes
CREATE INDEX player_status IF NOT EXISTS FOR (p:Player) ON (p.status);
CREATE INDEX location_type IF NOT EXISTS FOR (l:Location) ON (l.type);
CREATE INDEX quest_status IF NOT EXISTS FOR (q:Quest) ON (q.status);
CREATE INDEX fact_scroll IF NOT EXISTS FOR (f:Fact) ON (f.scroll_type);
```

3. Создать репозиторий `backend/app/db/neo4j_repository.py`. Основные методы:

- `create_player(name: str) -> PlayerOut`
- `get_player(name: str) -> PlayerOut`
- `update_player(name: str, updates: dict) -> PlayerOut`
- `get_active_players() -> List[PlayerOut]`
- `get_location_entities(location_name: str) -> dict` (возвращает игроков, NPC, предметы в этой локации + связанные)
- `get_quests_for_location(location_name: str) -> List[QuestOut]`
- `update_entity(entity_id, updates)` (универсальный)
- `add_fact(fact: FactBase)`
- `update_quest(quest_id, updates)`
- `get_related_npcs(fact_id: str)` – для обновления связей KNOWS.

Примеры Cypher для основных операций:

**Создание игрока:**
```cypher
MERGE (p:Player {name: $name})
ON CREATE SET p.hp_percent = 100, p.mp_percent = 100, p.status = 'active', p.inventory = [], p.abilities = []
RETURN p
```

**Получить активных игроков:**
```cypher
MATCH (p:Player {status: 'active'}) RETURN p
```

**Получить сущности в локации:**
```cypher
MATCH (loc:Location {name: $location_name})
OPTIONAL MATCH (loc)<-[:LOCATED_IN]-(entity)
WHERE entity:Player OR entity:NPC OR entity:Item
RETURN entity
```

**Создать связь LOCATED_IN:**
```cypher
MATCH (a {id: $entity_id}), (l:Location {name: $location_name})
MERGE (a)-[:LOCATED_IN]->(l)
```

## Этап 4: API: REST-эндпоинты + WebSocket-сообщения

**Цель:** Реализовать все точки входа для клиента, включая WebSocket-канал на каждую сессию.

### Что нужно сделать

1. Создать `backend/app/api/ws_manager.py` – менеджер WebSocket-соединений:
```python
class ConnectionManager:
    def __init__(self):
        self.active_connections: dict[str, WebSocket] = {}  # player_id -> websocket
    async def connect(self, player_id: str, websocket: WebSocket): ...
    def disconnect(self, player_id: str): ...
    async def send_personal_message(self, player_id: str, message: WSMessage): ...
    async def broadcast(self, message: WSMessage, exclude: Optional[str] = None): ...
```

2. Создать `backend/app/api/routes.py` с эндпоинтами:

- `POST /session/create` – создать новую игровую сессию (пока заглушка, возвращает session_id). Тело запроса: `{}`. Ответ: `{ "session_id": "uuid" }`.

- `POST /session/{session_id}/join` – игрок подключается, указывает имя. Тело: `{ "player_name": "Aragorn" }`. Ответ: `{ "player_id": "uuid", "session_id": "..." }`. Сервер создаёт/активирует игрока в Neo4j, связывает WebSocket.

- `POST /session/{session_id}/submit_action` – отправить заявку. Тело:
  ```json
  { "player": "Aragorn", "action_type": "do", "content": "Замахиваюсь мечом" }
  ```
  Ответ: `{ "status": "queued" }`.

- `POST /session/{session_id}/end_turn` – игрок (или лидер) нажимает «Конец хода». Тело: `{}`. Ответ: `{ "status": "turn_ended" }`. Когда все нажали – запускается игровой цикл.

- `POST /session/{session_id}/dice_result` – отправить результат броска. Тело:
  ```json
  { "player": "Aragorn", "result": 18 }
  ```
  Ответ: `{ "status": "accepted" }`.

- `POST /anima_query` – запрос к Аниме. Тело:
  ```json
  { "player_id": "uuid", "question": "Что у меня в инвентаре?" }
  ```
  Ответ: `{ "answer": "У тебя меч, щит и зелье лечения." }`.

- WebSocket `/ws/{session_id}/{player_id}` – постоянное соединение. При установке соединения менеджер регистрирует игрока. Сервер отправляет сообщения через него.

## Этап 5: Игровой цикл (оркестратор сессии, сбор заявок, вызов LLM, броски, обновление мира)

**Цель:** Реализовать центральный оркестратор, управляющий ходами игры, вызовами LLM и обновлением хранилищ.

### Что нужно сделать

Создать `backend/app/core/game_loop.py` – класс `GameLoop` (по одному на сессию). Основные методы:

- `start_turn()` – запускается при нажатии «Конец хода» всеми активными игроками (или по команде лидера). Логика:
  1. Получить `turn_order` из Neo4j (перемешать активных игроков) – сохранить в атрибут.
  2. Отправить `turn_order` через WS.
  3. Установить состояние “ожидание заявок”.

- `collect_action(player, action_type, content)` – сохранить заявку в словарь `pending_actions[player]`.
   Если заявка пустая (content пустой и action_type empty) – применить правило из ТЗ (поддержание или пропуск) и сохранить обработанную.

- `end_turn()` – когда все заявки собраны (триггер – вызов `end_turn` от всех или по кнопке лидера). Выполнить:
  1. Сформировать `player_actions` список в порядке `turn_order`.
  2. Собрать все блоки данных для промпта (сущности, свитки, память и т.д.) – это детально описано в Этапе 6.
  3. Вызвать `first_llm_call(prompt)` (Этап 6).
  4. Распарсить XML-ответ.
  5. Если есть `<dice_request>`:
     - Для каждого отправить `dice_prompt` игрокам.
     - Сохранить ожидаемые броски (player -> {dc, skill, reason, secret}) в `pending_dice`.
     - Перейти в режим ожидания бросков.
     - Когда все броски получены (метод `receive_dice_result`), сформировать блок результатов (как в ТЗ, скрытые броски с особым текстом).
     - Вызвать `second_llm_call(original_prompt + dice_results_block)`.
     - Распарсить финальный ответ (игнорировать dice_request).
  6. Извлечь из финального XML блоки: narrative, state_update, secret_memory, entity_update, world_gossip, threads_update, micro_canon, seeds_update, world_pulse.
  7. Обновить Neo4j и ChromaDB согласно этим блокам (методы `apply_updates`).
  8. Отправить narrative всем через WS.
  9. Отправить обновлённые состояния игроков (player_state).
  10. Запустить следующий ход (`start_turn`).

- `receive_dice_result(player, result)` – сохранить результат, сравнить с DC, определить статус (critical_success/success/soft_failure/failure/critical_failure). Правила:
  ```
  if result == 20: critical_success
  elif result >= dc: success
  elif result == 1: critical_failure
  elif result >= (dc-3) and result > 1: soft_failure
  else: failure
  ```
  Для скрытых бросков фактический статус сохраняется, но игроку не сообщается.

## Этап 6: Промпты и интеграция с DeepSeek API

**Цель:** Подготовить точные шаблоны промптов, клиент для LLM и логику сборки контекстных данных.

### Что нужно сделать

1. Создать `backend/app/llm/prompts.py`, содержащий две строковые константы: `EXIS_SYSTEM_PROMPT` и `ANIMA_SYSTEM_PROMPT`. Скопировать полные тексты из ТЗ (раздел 8.1 и 8.2) с плейсхолдерами `{}`.

2. Создать `backend/app/llm/deepseek_client.py`:
```python
from openai import OpenAI
from app.config import settings

client = OpenAI(api_key=settings.deepseek_api_key, base_url=settings.deepseek_base_url)

def call_deepseek_pro(messages: list) -> str:
    response = client.chat.completions.create(
        model=settings.deepseek_pro_model,
        messages=messages,
        temperature=0.8,
        max_tokens=4096
    )
    return response.choices[0].message.content

def call_deepseek_flash(messages: list) -> str:
    response = client.chat.completions.create(
        model=settings.deepseek_flash_model,
        messages=messages,
        temperature=0.9,
        max_tokens=1024
    )
    return response.choices[0].message.content
```

3. В `backend/app/core/context_builder.py` реализовать `build_exis_prompt(session)` – заполнение всех полей:
   - СЕТТИНГ, ПАРТИЯ, УРОВЕНЬ ОТКРОВЕНИЯ, МИРОВОЕ ВРЕМЯ – хранить в объекте сессии.
   - ПОРЯДОК ДЕЙСТВИЙ – из `turn_order`.
   - ЗАЯВКИ ИГРОКОВ – формат: `Имя: (тип) "текст"`.
   - АВАТАРЫ – для каждого активного игрока: `Имя: HP 80%, MP 50%, инвентарь: [...], способности: [...]`.
   - КОНТЕКСТ ПАМЯТИ – получить из ChromaDB последние 5 событий, сформировать художественное резюме (можно простую конкатенацию).
   - МИРОВОЙ ЛОР – статический текст, сохранённый в сессии.
   - ТАЙНЫЙ СВИТОК – `query_hidden_scroll` по текущему местоположению + вовлечённым NPC, n=5.
   - ЯВЛЕННЫЙ СВИТОК – `query_world_memory` по контексту, фильтровать тип absolute_truth/rumor.
   - НИТИ СУДЬБЫ – `get_quests_for_location`.
   - СЕМЕНА БУДУЩЕГО – `query_similar_to_scene`, n=3.
   - ТЕКУЩИЕ СУЩНОСТИ – `get_location_entities`.
   - РЕЗУЛЬТАТЫ БРОСКОВ – только для второго вызова.

   Финальная сборка: `EXIS_SYSTEM_PROMPT.format(...)`.

4. Аналогично `build_anima_prompt(player_id, question)`:
   - Извлечь character sheet, recent events из ChromaDB, подставить в `ANIMA_SYSTEM_PROMPT`.

5. В `game_loop.py` заменить заглушки вызовами:
   - `first_response = call_deepseek_pro([{"role":"system","content":prompt}])`
   - Парсинг XML, при необходимости `second_response`.

### Как проверить готовность
- Заполнить промпт тестовыми данными, вызвать DeepSeek (или мок) – убедиться, что ответ содержит XML.
- Проверить, что все плейсхолдеры заменяются.

### Чек-лист приёмки
- [ ] Тексты промптов идентичны ТЗ (раздел 8).
- [ ] Интеграция с DeepSeek API работает (можно протестировать с реальным ключом).
- [ ] Контекст-билдер собирает данные из обоих хранилищ.

## Этап 7: Клиентский SPA (минимальный интерфейс)

**Цель:** Создать одностраничное приложение на Vanilla JS/HTML, которое подключается к серверу и реализует весь пользовательский интерфейс.

### Что нужно сделать

Создать директорию `frontend/` с файлами `index.html`, `style.css`, `app.js`.

**index.html** – структура:
- Поле ввода имени и кнопка «Присоединиться».
- Основной чат (блок с прокруткой) для нарратива и заявок.
- Панель заявки: textarea, выпадающий список типа (do/say), кнопки «Отправить», «Пропустить ход», «Конец хода» (активна только если все заявки поданы или лидер).
- Модальное окно для броска: описание причины, поле ввода числа (1-20), кнопка «Бросить» (рандом 1-20), кнопка «Отправить».
- Личная вкладка Анимы: окно чата с Анимой, поле ввода вопроса, кнопка «Спросить».
- Блок состояния игрока: HP%, MP%, инвентарь, способности.
- Индикатор порядка хода: список имён с подсветкой.

**app.js** – логика:
- Подключение WebSocket: `ws://localhost:8000/ws/{session_id}/{player_id}`.
- Обработчики входящих сообщений:
  - `turn_order` – обновить индикатор порядка.
  - `narrative` – добавить в чат с классом `narrative`.
  - `dice_prompt` – показать модальное окно.
  - `player_state` – обновить панель игрока.
- Отправка действий: POST `/submit_action`.
- Отправка броска: POST `/dice_result`.
- Запрос к Аниме: POST `/anima_query`.
- Кнопка «Пропустить ход» отправляет `action_type: "empty"`.
- Кнопка «Конец хода» отправляет POST `end_turn`.

Стилизация минимальная: чат занимает основное пространство, панель внизу, индикатор сверху.

### Как проверить готовность
- Открыть index.html в браузере, подключиться к запущенному серверу.
- Имитировать игровой цикл: увидеть порядок, отправить заявку, получить запрос броска, отправить результат, увидеть нарратив.

### Чек-лист приёмки

- [ ] Все UI-элементы присутствуют и реагируют.
