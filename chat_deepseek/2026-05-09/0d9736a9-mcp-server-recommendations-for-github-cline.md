### RESPONSE (ИИ)

При использовании VSCode и Cline для работы с GitHub особенно рекомендуют несколько MCP-серверов. Они отличаются по своим возможностям: от управления репозиториями до интеллектуальной работы с кодом.

Вот их краткое сравнение, чтобы ты мог выбрать подходящий:

| MCP-сервер | Ключевая особенность | Для каких задач подходит |
| :--- | :--- | :--- |
| **github-repos-manager-mcp** | Самый функциональный, 89 инструментов для полного цикла разработки[reference:0] | Управление репозиториями, issues, PR, коммитами, ветками и файлами[reference:1] |
| **github-fetcher-mcp** | Лёгкий и быстрый, для чтения содержимого репозиториев[reference:2] | Получение содержимого файлов и структуры директорий с GitHub[reference:3] |
| **codebase-mcp** | Локальный сервер с поддержкой Git и "интеллектом" для анализа кода[reference:4] | Навигация по кодовой базе, поиск, анализ зависимостей и многое другое[reference:5] |
| **cline-community** | Официальный сервер для связи с сообществом Cline[reference:6] | Автоматизирует создание баг-репортов и обратной связи на GitHub[reference:7] |

### 🛠️ Детали и рекомендации по выбору

*   **`github-repos-manager-mcp`**: Вероятно, это лучший выбор для комплексного управления. Этот сервер на 89 инструментов[reference:8] работает напрямую через GitHub API, поэтому лёгок и не требует Docker[reference:9][reference:10]. Настроить его просто — достаточно токена, и ты сразу можешь управлять задачами, кодом и репозиториями[reference:11].
*   **`github-fetcher-mcp`**: Идеален, если твоя основная потребность — быстро "доставать" файлы. Сервер очень прост в установке: достаточно одной команды `npx`, чтобы ИИ-ассистент смог читать код из публичных репозиториев прямо во время работы[reference:12][reference:13].
*   **`codebase-mcp`**: Если ты работаешь с крупным проектом и ценишь "умные" подсказки, присмотрись к этому серверу. Он не просит доступа к GitHub, а запускается локально и анализирует твой код: показывает связи, историю изменений и многое другое[reference:14]. Учти, для его работы нужен установленный в системе Rust[reference:15].
*   **`cline-community`**: Этот сервер нужен для связи с разработчиками Cline и помогает быстро и правильно оформить баг-репорт, автоматически прикрепляя всю техническую информацию[reference:16]. Для работы потребуется установленный и настроенный GitHub CLI (`gh`)[reference:17].

Надеюсь, это сравнение поможет тебе с выбором. Если понадобится детальная инструкция по установке какого-либо из этих серверов, просто дай знать

### RESPONSE (ИИ)

Чтобы взаимодействовать с GitHub через API (например, с MCP-серверами, о которых мы говорили), нужен **персональный токен доступа (Personal Access Token, PAT)**. По сути это и есть твой API-ключ.

Вот пошаговая инструкция, как его создать.

### 1. Заходим в настройки токенов
- Открой [github.com](https://github.com) и войди в аккаунт.
- Нажми на свой аватар в правом верхнем углу → **Settings**.
- В левом меню пролистай вниз и выбери **Developer settings**.
- Затем слева выбери **Personal access tokens**.

Здесь ты увидишь два варианта:
- **Fine-grained tokens** (гранулярные) — более новые, с точным контролем разрешений для конкретных репозиториев.
- **Tokens (classic)** — классические, с широкими скоупами.

Для MCP-серверов чаще всего подходят оба типа, но в документации многих инструментов проще всего использовать **Classic**, если не указано иное.

### 2. Создаём токен

#### Вариант A: Classic token (простой и совместимый)
1. Нажми **Tokens (classic)** → **Generate new token (classic)**.
2. Введи понятное название (например `cline-mcp`).
3. В поле **Expiration** выбери срок действия (рекомендую 90 дней или кастомный, чтобы не забыть).
4. **Самый важный шаг** — отметь нужные разрешения (scopes):
   - Для `github-repos-manager-mcp` и работы с репозиториями, issues, PR, кодом понадобятся:
     - `repo` (все права на репозиторий, включая приватные)
     - `workflow` (если нужно управлять Actions)
     - `read:org` (если работаешь с организациями)
     - `user` (базовое, часто требуется)
     - Если будешь работать с проектами (Projects), поставь `project`.
   - Для простого `github-fetcher-mcp` обычно хватает только `repo`.
5. Нажми **Generate token**.
6. **Скопируй токен немедленно!** GitHub покажет его только один раз.

#### Вариант B: Fine-grained token (более безопасный)
1. Нажми **Fine-grained tokens** → **Generate new token**.
2. Выбери владельца ресурса (твой аккаунт или организацию).
3. Укажи **Repository access** — только выбранные репозитории или все.
4. В разделе **Permissions** раскрой и установи:
   - **Contents**: Read & write
   - **Issues**: Read & write
   - **Pull requests**: Read & write
   - **Metadata**: Read-only (обязательно)
   - И другие, если нужны Actions, Projects, Webhooks и т.д.
5. Нажми **Generate token** и скопируй.

### 3. Использование токена в MCP
Скопированный токен нужно вставить в конфигурацию MCP-сервера. Например, в настройках Cline для `github-repos-manager-mcp` указывается переменная `GITHUB_PERSONAL_ACCESS_TOKEN`, куда ты вставляешь сам токен.

Важно:
- Никогда не коммить токен в репозиторий.
- Если токен скомпрометирован, немедленно отзови его в том же разделе **Personal access tokens**.
