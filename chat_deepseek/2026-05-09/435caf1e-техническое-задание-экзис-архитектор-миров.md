### 2.3 Эндпоинты
| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/health` | Проверка работоспособности. Возвращает `{"status": "ok", "version": "0.1"}` |
| POST | `/api/session/start` | Создаёт новую игровую сессию. Возвращает `session_id` (UUID). |
| GET | `/api/session/{session_id}` | Получить метаданные сессии (время создания, последнее взаимодействие). |
| WebSocket | `/ws/chat/{session_id}` | Открывает двустороннее соединение для обмена сообщениями чата (основной игровой канал). |
| POST | `/api/chat/send` | (Альтернатива WebSocket) Отправить сообщение и получить ответ без стриминга. Требует `session_id`, payload: `{ "message": "..." }`. |

**Протокол WebSocket (`/ws/chat/{session_id}`):**
- Клиент подключается, передавая идентификатор сессии в URL.
- Сервер отправляет первое системное событие: `{"event": "connected", "session_id": "..."}`.
- Если сессия только что создана, Мастер автоматически генерирует приветственное сообщение и отправляет его как `{"event": "narrator", "text": "..."}`.
- Клиент отправляет сообщения в формате: `{"action": "player_message", "text": "Осматриваю таверну"}`.
- Сервер реагирует:
  1. Отправляет `{"event": "master_thinking", "status": "started"}`.
  2. Генерирует ответ (потоково или целиком) и отправляет `{"event": "master_speech", "text": "...", "turn_id": "..."}`.
  3. Завершает индикатор: `{"event": "master_thinking", "status": "stopped"}`.

**Обработка ошибок:** Сервер отправляет `{"event": "error", "message": "..."}`.

## 3. Модуль Памяти (Memory Core)
### 3.1 Двухконтурная схема
```
                   +---------------------+
                   |   Игровое сообщение |
                   +----------+----------+
                              |
            +-----------------v------------------+
            |  1. Поиск контекста (Vector + SQL)|
            +-----------------+------------------+
            |                                     |
   +--------v--------+              +-------------v-------------+
   | ChromaDB        |              | SQLite (Chronicles)        |
   | (эмбеддинги)    |              | - id, session_id, type    |
   | - collection:   |              | - timestamp, content      |
   |   "world_memory"|              | - embedding_id (ссылка)   |
   | - metadata:     |              | - summary                 |
   |   type, session |              | - parent_links (связи)     |
   +-----------------+              +---------------------------+
```
**Хроника (SQLite)** — основная структурная память.  
**Векторный индекс (ChromaDB)** — быстрый семантический поиск по содержимому хроник.

### 3.2 JSON-схема записи хроники (Chronicle Record)
```json
{
  "id": "chr_<uuid>",
  "session_id": "uuid сессии",
  "type": "event",  // event | npc | location | item | note
  "timestamp": "2025-01-01T12:00:00Z",
  "content": "Полный текстовый фрагмент, описывающий событие или сущность.",
  "summary": "Краткое резюме (генерируется при саммаризации).",
  "embedding_id": "id в ChromaDB (или null, пока не проиндексирован)",
  "parent_ids": ["chr_<uuid>", ...],  // связи для хронологической/логической цепочки
  "importance": 0.5  // float 0..1, для приоритизации при извлечении
}
```

### 3.3 Алгоритм сохранения
При поступлении релевантного действия (из поля `world_update` ответа ИИ):
1. Сформировать объект `Chronicle` без `embedding_id` и `summary`.
2. Сохранить в SQLite, получить `id`.
3. Асинхронно (через фоновую задачу FastAPI BackgroundTasks):
   - Вызвать embedding-функцию (локальную модель или DeepSeek embedding, настраивается) для `content`.
   - Добавить эмбеддинг в коллекцию ChromaDB с метаданными `{"chronicle_id": id, "session_id": ..., "type": ...}`.
   - Обновить `embedding_id` в SQL.
4. Проверить, не превышен ли порог числа хроник в сессии без саммаризации. Если да — запустить саммаризацию (см. 3.6).

### 3.4 Алгоритм извлечения контекста для запроса LLM
**Цель**: собрать контекст, влезающий в `max_context_tokens` промта (настраиваемый параметр, по умолчанию ~3000 токенов).

Шаги:
1. Получить последние N сообщений из истории сессии (хранится в оперативной памяти `session_manager`, последние 5-10 обменов).
2. Создать векторный запрос из текста текущего сообщения игрока.
3. Запросить ChromaDB: `collection.query(query_embedding, n_results=5, where={"session_id": session_id})`. Получить top-5 похожих записей.
4. Для каждого результата извлечь из SQL полную запись хроники и её цепочку `parent_ids` (до глубины 2) — это даст связанный контекст.
5. Все отобранные записи упорядочить по времени, удалить дубликаты.
6. Подготовить текстовый блок: для каждой записи использовать `summary`, если доступно, иначе усечённый `content`. Предварять заголовком `(ХРОНИКА: <timestamp>)`.
7. Если общий объём токенов превышает лимит, запустить экстренную саммаризацию этого набора (средствами того же DeepSeek, промт «Summarize the following chronicles concisely...»).

### 3.5 Конфигурация ChromaDB
- **Коллекция**: `world_memory`.
- **Embedding-функция**: по умолчанию `sentence-transformers/all-MiniLM-L6-v2` (размерность 384). Возможна опциональная смена на DeepSeek embedding через конфигурацию.
- **Метаданные документа**:
  ```json
  {
    "chronicle_id": "chr_<uuid>",
    "session_id": "uuid",
    "type": "event",
    "importance": 0.5,
    "timestamp": "ISO8601"
  }
  ```

### 3.6 Механизм саммаризации (Summary Chain)
Фоновая задача, запускаемая двумя триггерами:
- Количество несуммаризированных хроник в сессии ≥ 10.
- Либо запланированная раз в 5 минут.

**Процесс:**
1. Выбрать все хроники текущей сессии, где `summary IS NULL`, упорядоченные по времени.
2. Сформировать список их `content` и передать в DeepSeek API с системным промтом:
   ```
   You are a chronicle summarizer. Given a list of narrative events, produce a single concise paragraph summarizing everything, preserving key characters, locations, and objects.
   ```
3. Получить суммарный текст.
4. Обновить записи: для всех этих хроник установить `summary = <полученный текст>`.
5. Создать новую запись `type = "summary"`, которая ссылается на эти хроники (`parent_ids`), с сгенерированным summary. Это позволит при извлечении будущего контекста ссылаться на один суммарный блок вместо многих мелких.

## 4. Интеграция с DeepSeek API
### 4.1 Конфигурация клиента
- **Base URL**: `https://api.deepseek.com/v1`

- **Параметры по умолчанию**:
  - `temperature`: 0.8 (для творческой магии Мастера)

- **Аутентификация**: `Authorization: Bearer <DEEPSEEK_API_KEY>`, ключ читается из переменных окружения.

### 4.4 Обработка ошибок и Rate Limiting
- **Retry-логика**: exponential backoff с jitter (1s, 2s, 4s, макс. 15s), до 3 попыток.
- **Ошибки сети/сервера**: логировать и возвращать игроку дружественное сообщение «Мастер временно задумался о вечности...».
- **Rate Limits**: отслеживать заголовки ответа `X-RateLimit-Remaining`. При приближении к исчерпанию включать token bucket на стороне приложения (макс. 10 запросов в минуту, ограничение для одного пользователя). Для v0.1 достаточно глобальной очереди с asyncio.Semaphore.

## 8. Схема развёртывания на Hostland
### 8.1 Целевая инфраструктура
- **VPS от Hostland**: минимальная конфигурация 1 vCPU, 2 ГБ RAM, SSD 20 ГБ, ОС Ubuntu 22.04 LTS.
- **ПО**: Python 3.11, Nginx, supervisor или systemd для управления процессами.

### 8.2 Порядок развёртывания
1. **Подготовка сервера**:
   - Подключиться по SSH.
   - Установить зависимости: `sudo apt update && sudo apt install python3.11 python3.11-venv nginx git`.
   - Создать пользователя `exis` и рабочую директорию `/opt/exis/`.
2. **Клонирование репозитория**:
   ```bash
   git clone https://github.com/yourrepo/exis.git /opt/exis/app
   ```
3. **Настройка виртуального окружения**:
   ```bash
   cd /opt/exis/app/backend
   python3.11 -m venv venv
   source venv/bin/activate
   pip install -r requirements.txt
   ```
4. **Переменные окружения**:
   Создать файл `/opt/exis/app/backend/.env`:
   ```
   DEEPSEEK_API_KEY=sk-...
   DATABASE_URL=sqlite:////opt/exis/data/exis_chronicles.db
   CHROMA_PATH=/opt/exis/data/chroma
   SECRET_KEY=случайная_строка
   ```
5. **Настройка Nginx**:
   Конфигурация `/etc/nginx/sites-available/exis`:
   ```nginx
   server {
       listen 80;
       server_name exis.yourdomain.com;

       location / {
           proxy_pass http://127.0.0.1:8000;
           proxy_http_version 1.1;
           proxy_set_header Upgrade $http_upgrade;
           proxy_set_header Connection "upgrade";
           proxy_set_header Host $host;
           proxy_set_header X-Real-IP $remote_addr;
           proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
       }
   }
   ```
   Включить сайт и перезагрузить Nginx.
6. **Systemd-сервис для Uvicorn**:
   Создать `/etc/systemd/system/exis.service`:
   ```ini
   [Unit]
   Description=Exis FastAPI app
   After=network.target

   [Service]
   User=exis
   Group=exis
   WorkingDirectory=/opt/exis/app/backend
   EnvironmentFile=/opt/exis/app/backend/.env
   ExecStart=/opt/exis/app/backend/venv/bin/uvicorn app.main:app --host 127.0.0.1 --port 8000

   [Install]
   WantedBy=multi-user.target
   ```
   Запустить: `sudo systemctl enable --now exis`.
7. **Проверка**:
   - `curl http://localhost:8000/api/health` → ok.
   - Открыть домен в браузере, увидеть чат.

### 8.3 Автоматизация деплоя (CI/CD опционально на будущее)
- GitHub Actions: при пуше в main → SSH на сервер, `git pull`, перезапуск systemd-сервиса. В v0.1 ручной деплой.

**Ответ на вопрос:**

Связка **VSCode + Cline** и производственный сервер (**VPS Hostland + Nginx + приложение**) — это два совершенно разных мира, и они не заменяют друг друга. В нашей архитектуре они разделены намеренно:

*   **VSCode + Cline («Манус»)** – это **инструмент разработчика**, «чертёжный стол» и «бригадир». Здесь Манус вместе с Творцом пишут код, тестируют его, коммитят в Git. Этот контур **не нужен** на боевом сервере – он только создаёт приложение.
*   **VPS Hostland** – это **среда исполнения**, «дом», где живёт готовое приложение. Там работают веб-сервер, базы данных и наше FastAPI-приложение. Игроки подключаются к нему через Nginx, не подозревая о существовании Cline.

Писать на сервере аналог VSCode + Cline, который бы сам себя модифицировал в рантайме, – это **отдельная и значительно более сложная задача**:
*   Потребовался бы браузерный редактор кода, терминал, Git-клиент, менеджер зависимостей.
*   Обеспечение безопасности выполнения сгенерированного кода (изоляция) – отдельная головная боль.
*   Риск «самоампутации»: ошибка в сгенерированном коде могла бы уронить весь мир Экзиса.
*   Для нашего скромного v0.1 это абсолютный overkill.

Даже в долгосрочной перспективе «самопишущийся» ИИ-Мастер скорее будет реализован иначе: не через правку исходного кода в проде, а через управление данными (лор, квесты, диалоги) и, возможно, через вызов изолированных скриптов (песочница). Но это тема для версий v1.x и далее.

**Итог:** Манус (Cline) остаётся на машине Творца. На боевом сервере его нет и не должно быть. Мы не усложняем, а следуем фундаменту: сначала надежное, простое приложение.
