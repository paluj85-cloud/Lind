1. Общее описание и цели системы
1.1. Концепция простым языком
Представьте себе очень умного и трудолюбивого помощника-программиста, который работает на вашем сервере. Вы ставите ему задачу: «Добавь функцию проверки пароля», он идёт в код, всё делает, докладывает. Но есть проблема: на следующий день он забывает всё, что было вчера. Вы снова должны объяснять ему, как устроен ваш проект, какие файлы где лежат, какие решения вы вчера приняли. Это тратит время и нервы.
«Вечный Архив» решает эту проблему. Это — внешняя память для ИИ-помощника. Она работает как блокнот, в который помощник сам записывает самое важное:
•	«Вот здесь лежит главный файл аутентификации»;
•	«Мы решили использовать библиотеку bcrypt для хеширования паролей»;
•	«Вот эту функцию надо будет переписать позже — сейчас сделали временное решение».
Когда вы приходите на следующий день и даёте новое задание, помощник первым делом заглядывает в «Архив», вспоминает контекст и продолжает работу с того же места, не задавая глупых вопросов. «Архив» сам решает, какие именно воспоминания важны для текущей задачи, и подаёт их помощнику в сжатом виде, не забивая ему «голову» (контекстное окно) бесполезной информацией.

3. Архитектура системы и логика взаимодействия
3.1. Компонентная схема (логическая)
•	Точка входа (src/mcp_server.py): Инициализирует FastMCP сервер, подключает все инструменты, запускает фоновый обработчик отложенных задач.
•	Менеджер БД (src/database/): SQLAlchemy модели, движок, CRUD-операции. Весь доступ к БД — только через этот слой.
•	Подсистема индексации (src/indexing/): Обходит файлы в project_root, вычисляет SHA256, парсит код, генерирует описания (с fallback).
•	Ядро памяти (src/memory/):
o	context_manager.py: Реализует гибридный поиск (векторный + FTS5) с RRF-ранжированием.
o	canonizer.py: Взаимодействует с DeepSeek API для извлечения фактов из диалогов.
o	validator.py: Проверяет новые факты на дубликаты и противоречия; при сбоях создаёт отложенные задачи.
o	embedder.py: Обёртка вокруг sentence-transformers для векторизации текста.
•	Подсистема отложенных задач (src/deferred_worker.py): Фоновый asyncio-обработчик, выполняющий задачи из очереди deferred_tasks.
3.2. Жизненный цикл (процесс работы)
1.	Запуск: Разработчик запускает Cline CLI в headless-режиме (cline -y "задача").
2.	Подключение к MCP: CLI читает cline_mcp_settings.json и запускает Eternal Archive как дочерний процесс через stdio.
3.	Handshake: Eternal Archive отдаёт список инструментов (tools/list).
4.	Инициализация проекта и сессии: CLI вызывает create_project (если проект ещё не существует), затем start_session. Сервер запоминает текущий проект и активную сессию в оперативной памяти.
5.	Получение контекста: CLI (по инициативе LLM внутри него) вызывает инструмент get_project_context.
6.	Выполнение задачи: LLM внутри CLI выполняет задачу, попутно вызывая инструменты add_fact, add_seed, index_files.
7.	Микро-канонизация: По достижении порога в ~200 000 токенов (отслеживается клиентом, опционально через get_session_token_count) CLI вызывает инструмент run_micro_canonization.
8.	Завершение: При завершении сессии CLI вызывает summarize_session, запуская макро-канонизацию и закрывая сессию.
3.3. Ключевые проектные решения
•	Транспорт: Только stdio. Никакого HTTP в основном режиме.
•	Инициатива канонизации: Клиент (Cline CLI) считает токены и командует микро-канонизацией. Сервер лишь предоставляет инструмент для запуска.
•	Валидация при добавлении: Каждый факт при вызове add_fact проверяется автоматически. При недоступности DeepSeek операция не прерывается, а создаётся отложенная задача.
•	Управление сессиями: Сервер хранит состояние текущей сессии (current_project_name, current_session_id) в глобальных переменных процесса.
•	Отказоустойчивость: Все критические вызовы к DeepSeek обёрнуты в retry-логику; при исчерпании попыток создаются отложенные задачи.
•	Версионирование эмбеддингов: Каждая запись, содержащая embedding, хранит версию модели (embedding_model_version), что позволяет в будущем плавно мигрировать на новую модель без потери совместимости.

5.2. SQL-схема (дополненная)
sql
-- Проекты
CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    type TEXT NOT NULL CHECK (type IN ('cline', 'exis')),
    root_path TEXT,
    metadata TEXT
);

-- Сессии
CREATE TABLE sessions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    started_at TEXT DEFAULT (datetime('now')),
    ended_at TEXT,
    summary TEXT,
    context_snapshot TEXT,
    last_canonized_message_id INTEGER,  -- НОВОЕ: ID последнего сообщения, обработанного микро-канонизацией
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);

-- Сообщения
CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id INTEGER NOT NULL,
    role TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),
    content TEXT NOT NULL,
    timestamp TEXT DEFAULT (datetime('now')),
    metadata TEXT,
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE
);
CREATE INDEX idx_messages_session ON messages(session_id, timestamp);

-- Сущности (игроки, NPC, функции, классы, модули)
CREATE TABLE entities (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    name TEXT NOT NULL,
    type TEXT NOT NULL,
    description TEXT,
    metadata TEXT,
    embedding BLOB NOT NULL,
    embedding_model_version TEXT NOT NULL DEFAULT 'rubert-tiny2-v1',
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);

-- Факты (элементы памяти)
CREATE TABLE facts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    session_id INTEGER,
    source_message_id INTEGER,   -- автоматически из последнего сообщения user/assistant
    content TEXT NOT NULL,
    type TEXT NOT NULL DEFAULT 'general'
        CHECK (type IN ('decision', 'architecture', 'preference', 'debt', 'world_lore', 'general')),
    status TEXT NOT NULL DEFAULT 'ABSOLUTE_TRUTH'
        CHECK (status IN ('ABSOLUTE_TRUTH', 'RUMOR', 'LIE', 'DEPRECATED')),
    confidence REAL DEFAULT 1.0 CHECK (confidence >= 0.0 AND confidence <= 1.0),
    embedding BLOB NOT NULL,
    embedding_model_version TEXT NOT NULL DEFAULT 'rubert-tiny2-v1',
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE SET NULL,
    FOREIGN KEY (source_message_id) REFERENCES messages(id) ON DELETE SET NULL
);

-- История изменений фактов
CREATE TABLE fact_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    fact_id INTEGER NOT NULL,
    previous_content TEXT,
    new_content TEXT,
    changed_at TEXT DEFAULT (datetime('now')),
    change_reason TEXT,
    FOREIGN KEY (fact_id) REFERENCES facts(id) ON DELETE CASCADE
);

-- Связи фактов с сущностями
CREATE TABLE fact_entity_link (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    fact_id INTEGER NOT NULL,
    entity_id INTEGER NOT NULL,
    link_type TEXT DEFAULT 'related',
    FOREIGN KEY (fact_id) REFERENCES facts(id) ON DELETE CASCADE,
    FOREIGN KEY (entity_id) REFERENCES entities(id) ON DELETE CASCADE,
    UNIQUE(fact_id, entity_id)
);

-- Семена (неразвитые идеи, технический долг)
CREATE TABLE seeds (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    content TEXT NOT NULL,
    type TEXT NOT NULL DEFAULT 'idea'

7. Контекстный менеджер и гибридный поиск
7.1. Алгоритм get_project_context
1.	Получить эмбеддинг запроса от embedder.encode(query).
2.	Для каждого запрошенного типа данных (facts, seeds, files, quests, entities) параллельно:
o	Векторный поиск по соответствующей *_vec таблице (top-50, косинусное сходство).
o	Для facts, seeds, files дополнительно полнотекстовый поиск по *_fts (top-50 BM25). Для quests и entities — только векторный.
3.	Для каждого типа применить Reciprocal Rank Fusion (k=60), объединяя векторные и FTS результаты.
4.	Из каждого типа выбрать топ-кандидатов (не более 5), сохраняя исходное косинусное сходство лучшего результата.
5.	Объединить кандидатов всех типов, отсортировать по RRF-счёту, выбрать общий top-K.
6.	Выполнить confidence_check на основе максимального косинусного сходства среди отобранных кандидатов.
7.	Вернуть результат с метаданными поиска.
7.2. Reciprocal Rank Fusion (RRF)
python
def reciprocal_rank_fusion(
    vector_results: list[tuple[int, float]],  # (id, cosine_sim)
    fts_results: list[tuple[int, float]],     # (id, bm25_score)
    k: int = 60,
) -> list[tuple[int, float, float]]:          # (id, cosine_sim, rrf_score)
    scores = {}
    cosine_map = {doc_id: sim for doc_id, sim in vector_results}
    for rank, (doc_id, _) in enumerate(vector_results, start=1):
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    for rank, (doc_id, _) in enumerate(fts_results, start=1):
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    # возвращаем вместе с исходным косинусным сходством (если был, иначе 0)
    fused = [(doc_id, cosine_map.get(doc_id, 0.0), rrf) for doc_id, rrf in scores.items()]
    return sorted(fused, key=lambda x: x[2], reverse=True)
7.3. Confidence Check
python
def confidence_check(
    top_candidates: list[tuple[int, float, float]],  # (id, cosine_sim, rrf_score)
    min_cosine: float = 0.7,
    min_gap: float = 0.15,
) -> tuple[bool, str]:
    if not top_candidates or top_candidates[0][1] < min_cosine:
        return False, f"Максимальное косинусное сходство ({top_candidates[0][1] if top_candidates else 0:.2f}) ниже порога {min_cosine}"
    if len(top_candidates) >= 2:
        gap = top_candidates[0][1] - top_candidates[1][1]
        if gap < min_gap:
            return False, f"Разрыв между лучшим и вторым результатом ({gap:.2f}) недостаточен"
    return True, "OK"
Пороги (min_cosine, min_gap) настраиваются в .env.

async def validate_fact(content: str, project_id: int, embedder) -> dict:
    emb = await embedder.encode(content)
    # 1. Поиск дубликатов (cosine > 0.9 или расстояние < 0.1)
    closest = await vector_search(emb, project_id, top_k=3)
    if closest and closest[0][1] > 0.9:  # cosine_sim
        return {"status": "duplicate", "existing_fact_id": closest[0][0]}
    # 2. Поиск потенциальных противоречий (0.7 <= cosine < 0.9)
    potential_contradictions = [c for c in closest if 0.7 <= c[1] < 0.9]
    if potential_contradictions:
        try:
            is_contradiction = await deepseek_check_contradiction(content, potential_contradictions[0][1])
        except DeepSeekError:
            # Создать отложенную задачу
            task_id = await create_deferred_task('validate_contradiction', {
                'fact_id': new_fact_id, 'existing_fact_id': potential_contradictions[0][0]
            })
            return {"status": "deferred", "task_id": task_id, "confidence": 0.3}
        if is_contradiction:
            return {"status": "contradiction", "confidence": 0.4}
    # 3. Всё чисто
    return {"status": "ok", "confidence": 0.95}
После выполнения отложенной задачи validate_contradiction факт может быть повышен до ABSOLUTE_TRUTH с confidence=0.9 (если LLM подтвердит отсутствие противоречия).
10. Подсистема отложенных задач и отказоустойчивость
Назначение: Гарантировать, что временная недоступность DeepSeek не приведёт к потере данных или остановке работы.
10.1. Фоновый обработчик (deferred_worker.py)
•	Запускается при старте сервера как asyncio.create_task(worker_loop()).
•	Каждые 60 секунд извлекает из deferred_tasks записи со статусом pending и next_attempt_at <= now.
•	Для каждой задачи вызывает соответствующую функцию (валидацию, микро- или макро-канонизацию).
•	При успехе — статус completed, факт (если был RUMOR) при необходимости обновляется.
•	При неудаче — увеличивает счётчик попыток, назначает следующую попытку через 10 минут. После 3 неудач — статус failed.
10.2. Типы задач и структура payload
Тип задачи	Поля payload (JSON)	Описание
validate_contradiction	{"fact_id": int, "existing_fact_id": int}	Перепроверить, противоречат ли два факта
micro_canonization	{"session_id": int, "project_name": str, "last_message_id": int, "previous_processed_id": int}	Обработать сообщения с ID от previous_processed_id+1 до last_message_id включительно
macro_canonization	{"session_id": int, "project_name": str}	Суммаризировать всю сессию
Механизм last_message_id для микро-канонизации:
•	В момент сбоя сервер фиксирует ID последнего сообщения в сессии (last_message_id) и ID последнего обработанного предыдущей микро-канонизацией (sessions.last_canonized_message_id).
•	Отложенная задача гарантированно обрабатывает только этот фиксированный блок сообщений. Более новые сообщения (если появятся) будут обработаны следующей микро-канонизацией.
•	После успешного выполнения задачи sessions.last_canonized_message_id обновляется до last_message_id.
