### ЭТАП 1: КРИТИЧЕСКИЙ АНАЛИЗ И ВЕРИФИКАЦИЯ
Проанализируй ТЗ на предмет:
1. **Логическая целостность**: нет ли внутренних противоречий (например, требование полной локальности и одновременное использование облачного DeepSeek API описаны как осознанный компромисс — проверь, учтены ли риски); соответствуют ли схемы данных описанным в API сущностям; все ли компоненты, упомянутые в архитектуре, имеют описание реализации.
2. **Полнота**: для каждого функционального блока (индексация, канонизация, валидация, гибридный поиск) определены алгоритмы, входные/выходные данные, обработка ошибок. Нет ли пропущенных ключевых сценариев (например, что делать при недоступности DeepSeek API во время канонизации? или при повреждении индекса FTS?).
3. **Противоречия и нестыковки**: сопоставь SQL-схему, описание API-инструментов и текстовые примеры. Например, есть ли поле `session_id` в таблице `facts` (да) — а передаётся ли оно в `add_fact` (нет в параметрах, но может быть получено из текущей сессии; проясни этот момент). Проверь все ссылки на внешние компоненты (sqlite-vec, FastMCP) и их совместимость.
4. **Риски и «подводные камни»**: дополни список в разделе 14, если найдёшь неучтённые риски (например, деградация производительности гибридного поиска при большом количестве фактов; синхронизация триггеров FTS при массовых вставках; управление версиями эмбеддинг-модели и миграции векторов).
5. **Качество промптов для LLM** внутри ТЗ (микро- и макро-канонизация): оцени, насколько эффективно они составлены для DeepSeek, можно ли улучшить их структуру, чтобы повысить точность извлечения фактов и снизить галлюцинации.

### ЭТАП 2: ПЕРЕРАБОТКА ТЗ В LLM-ОПТИМИЗИРОВАННЫЙ ФОРМАТ
На основе исходного ТЗ создай новую версию документа, которая максимально удобна для последующего прочтения и использования другой LLM (например, в роли AI-разработчика). Требования к переработанному документу:
- **Самодостаточность**: документ должен содержать всю необходимую для реализации информацию, без отсылок «как обсуждалось ранее».
- **Чёткая иерархия**: используй XML-подобные теги или строгую Markdown-структуру с якорями для разделов, чтобы модель могла точно извлекать нужные блоки.
- **Сжатие без потерь**: убери повествовательные отступления, оставив только суть: определения, схемы (в текстовом виде), алгоритмы, форматы данных, конфигурации. Каждый раздел начни с краткого TL;DR (1-2 предложения) для быстрого понимания.
- **Явные связи**: добавь перекрёстные ссылки между разделами (например, «см. алгоритм валидации в 9.3», «формат данных описан в 5.2»). Для таблиц БД укажи, какие поля обязательны для каких API-инструментов.
- **Инструктивный тон**: формулируй так, будто LLM-разработчик будет следовать этому документу как спецификации. Вместо «можно реализовать» пиши «РЕАЛИЗОВАТЬ:». Вместо общих слов давай конкретные шаги.
- **Защита от галлюцинаций**: в неоднозначных местах добавь примечания «Если информации недостаточно, используй fallback: …» и явно пометь статус «НЕОКОНЧАТЕЛЬНОЕ РЕШЕНИЕ (validation required)».

1. Общее описание и цели системы
1.1. Концепция простым языком
Представьте себе очень умного и трудолюбивого помощника-программиста, который работает на вашем сервере. Вы ставите ему задачу: «Добавь функцию проверки пароля», он идёт в код, всё делает, докладывает. Но есть проблема: на следующий день он забывает всё, что было вчера. Вы снова должны объяснять ему, как устроен ваш проект, какие файлы где лежат, какие решения вы вчера приняли. Это тратит время и нервы.
«Вечный Архив» решает эту проблему. Это — внешняя память для ИИ-помощника. Она работает как блокнот, в который помощник сам записывает самое важное:
•	«Вот здесь лежит главный файл аутентификации»;
•	«Мы решили использовать библиотеку bcrypt для хеширования паролей»;
•	«Вот эту функцию надо будет переписать позже — сейчас сделали временное решение».
Когда вы приходите на следующий день и даёте новое задание, помощник первым делом заглядывает в «Архив», вспоминает контекст и продолжает работу с того же места, не задавая глупых вопросов. «Архив» сам решает, какие именно воспоминания важны для текущей задачи, и подаёт их помощнику в сжатом виде, не забивая ему «голову» (контекстное окно) бесполезной информацией.

3.2. Жизненный цикл (процесс работы)
1.	Запуск CLI: Разработчик запускает Cline CLI в headless-режиме (cline -y "задача").
2.	Подключение к MCP: CLI читает cline_mcp_settings.json и запускает Eternal Archive как дочерний процесс через stdio.
3.	Handshake: Eternal Archive отдаёт список инструментов (tools/list).
4.	Получение контекста: CLI (по инициативе LLM внутри него) вызывает инструмент get_project_context.
5.	Выполнение задачи: LLM внутри CLI выполняет задачу, попутно вызывая инструменты add_fact, add_seed, index_files.
6.	Канонизация: По достижении порога (каждые 10 сообщений) CLI или сам Архив инициирует микро-канонизацию.
7.	Завершение: При завершении сессии CLI вызывает summarize_session, запуская макро-канонизацию.
3.3. Ключевые проектные решения
•	Транспорт: Только stdio. Никакого HTTP. Cline CLI общается с Архивом через stdin/stdout в формате JSON-RPC.
•	Канонизация по требованию: CLI сам решает, когда запускать summarise, вызывая инструмент summarize_session.
•	Валидация при добавлении: Каждый новый факт при вызове add_fact проходит автоматическую валидацию перед сохранением.
•	Контекст — в промпт: Результат get_project_context Cline CLI добавляет в промпт задачи.

-- Проекты (разделение контекста: Cline-проект / кампания Экс-Ис)
CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    type TEXT NOT NULL CHECK (type IN ('cline', 'exis')),
    root_path TEXT,                 -- корневая директория (только для 'cline')
    metadata TEXT                   -- JSON: доп. настройки, описание
);

-- Сессии
CREATE TABLE sessions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    started_at TEXT DEFAULT (datetime('now')),
    ended_at TEXT,
    summary TEXT,                   -- результат макро-канонизации
    context_snapshot TEXT,          -- JSON: снимок контекста на конец сессии
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);

-- Сообщения (полный журнал)
CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id INTEGER NOT NULL,
    role TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),
    content TEXT NOT NULL,
    timestamp TEXT DEFAULT (datetime('now')),
    metadata TEXT,                  -- JSON: токены, вызовы инструментов
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE
);
CREATE INDEX idx_messages_session ON messages(session_id, timestamp);

-- Сущности (игроки, NPC, функции, классы, модули)
CREATE TABLE entities (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    name TEXT NOT NULL,
    type TEXT NOT NULL,
    description TEXT,
    metadata TEXT,                  -- JSON: атрибуты, характеристики
    embedding BLOB NOT NULL,        -- float32[768]
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);

-- Факты (элементы памяти)
CREATE TABLE facts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    session_id INTEGER,
    source_message_id INTEGER,
    content TEXT NOT NULL,
    type TEXT NOT NULL DEFAULT 'general'
        CHECK (type IN ('decision', 'architecture', 'preference', 'debt', 'world_lore', 'general')),
    status TEXT NOT NULL DEFAULT 'ABSOLUTE_TRUTH'
        CHECK (status IN ('ABSOLUTE_TRUTH', 'RUMOR', 'LIE', 'DEPRECATED')),
    confidence REAL DEFAULT 1.0 CHECK (confidence >= 0.0 AND confidence <= 1.0),
    embedding BLOB NOT NULL,        -- float32[768]
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE SET NULL,
    FOREIGN KEY (source_message_id) REFERENCES messages(id) ON DELETE SET NULL
);

-- История изменений фактов
CREATE TABLE fact_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    fact_id INTEGER NOT NULL,
    previous_content TEXT,
    new_content TEXT,
    changed_at TEXT DEFAULT (datetime('now')),
    change_reason TEXT,
    FOREIGN KEY (fact_id) REFERENCES facts(id) ON DELETE CASCADE
);

-- Связи фактов с сущностями
CREATE TABLE fact_entity_link (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    fact_id INTEGER NOT NULL,
    entity_id INTEGER NOT NULL,
    link_type TEXT DEFAULT 'related',
    FOREIGN KEY (fact_id) REFERENCES facts(id) ON DELETE CASCADE,
    FOREIGN KEY (entity_id) REFERENCES entities(id) ON DELETE CASCADE,
    UNIQUE(fact_id, entity_id)
);

-- Семена (неразвитые идеи, технический долг)
CREATE TABLE seeds (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    content TEXT NOT NULL,
    type TEXT NOT NULL DEFAULT 'idea'
        CHECK (type IN ('idea', 'debt', 'bug', 'feature')),
    priority INTEGER DEFAULT 0,
    status TEXT NOT NULL DEFAULT 'open'
        CHECK (status IN ('open', 'in_progress', 'resolved')),
    embedding BLOB NOT NULL,        -- float32[768]
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);

-- Квесты / Активные ветки разработки
CREATE TABLE quests (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    title TEXT NOT NULL,
    description TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'active'
        CHECK (status IN ('active', 'completed', 'failed', 'blocked')),
    embedding BLOB NOT NULL,        -- float32[768]

7.2. Пример кода RRF-ранжирования
python
def reciprocal_rank_fusion(
    vector_results: list[tuple[int, float]],  # (id, cosine_sim)
    fts_results: list[tuple[int, float]],     # (id, bm25_score)
    k: int = 60,
) -> list[tuple[int, float]]:
    """Объединяет результаты двух поисков через Reciprocal Rank Fusion."""
    scores = {}
    for rank, (doc_id, _) in enumerate(vector_results, start=1):
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    for rank, (doc_id, _) in enumerate(fts_results, start=1):
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
Этот подход заимствован у Mneme и подтверждён как эффективный для поиска по коду и проектной документации.
7.3. Confidence Check (Проверка уверенности)
Ключевая идея, заимствованная у Mneme: лучше отказаться от ответа, чем дать нерелевантный результат.
python
def confidence_check(
    fused_results: list[tuple[int, float]],
    top_score_threshold: float = 0.25,
    gap_threshold: float = 0.15,
) -> tuple[bool, str]:
    """Проверяет, достаточно ли уверенны результаты поиска."""
    if not fused_results or fused_results[0][1] < top_score_threshold:
        return False, "Нет достаточно релевантных результатов."
    if len(fused_results) >= 2:
        gap = fused_results[0][1] - fused_results[1][1]
        if gap < gap_threshold:
            return False, f"Результаты недостаточно различимы (gap={gap:.3f})."
    return True, "OK"

9.1. Микро-канонизация (Micro-Canonization)
Триггер: Автоматический после накопления 10 новых сообщений в сессии (значение по умолчанию, настраивается через .env).
Процесс:
1.	Берутся последние 15 сообщений (текущий блок + 5 предыдущих для контекста).
2.	Отправляются в DeepSeek со специализированным промптом:
text
Ты — Архивариус. Извлеки из следующей истории диалога все новые факты о проекте.
Для каждого факта укажи:
- content: текст факта;
- type: decision | architecture | preference | debt | general;
- confidence: 0.0 до 1.0;
- related_entities: список имён связанных сущностей.

Если новых фактов нет, верни пустой список.
3.	DeepSeek возвращает JSON-массив фактов.
4.	Каждый факт проходит валидацию (см. раздел 9.3).
5.	Валидные факты сохраняются в facts с session_id и source_message_id.
9.2. Макро-канонизация (Macro-Canonization)
Триггер: Вызов инструмента summarize_session (явный или автоматический при завершении сессии).
Процесс:
1.	Берутся все сообщения сессии.
2.	Отправляются в DeepSeek с промптом:
text
Ты — Архивариус. Проведи полную канонизацию сессии:
1. summary: краткое содержание (3-5 предложений).
2. key_decisions: список ключевых архитектурных решений.
3. new_debt: список нового технического долга.
4. updated_facts: список фактов, которые нужно обновить (с указанием ID).

o	summary → sessions.summary;
o	key_decisions → facts;

10. Безопасность и ограничения
10.1. Модель угроз
Угроза	Уровень риска	Меры защиты
Несанкционированный доступ к БД	Низкий (VPS)	Права ОС (chmod 600), отдельный пользователь eternal-archive
Инъекция в SQL	Средний	Параметризованные запросы SQLAlchemy
Галлюцинации LLM при канонизации	Средний	Валидация фактов, confidence scoring, статус RUMOR
Утечка данных через DeepSeek API	Средний	Минимизация отправляемых данных, только извлечённые факты
Переполнение контекстного окна	Низкий	Динамический top_k, лимит символов
10.2. Ограничения
•	Чтение файлов ограничено project_root (проверка через pathlib).
•	DeepSeek API используется только для канонизации (микро/макро) и валидации.
•	Все данные хранятся локально в SQLite.
10.3. Защита от галлюцинаций
•	Все факты, извлечённые LLM, проходят валидацию (раздел 9.3).
•	Факты имеют статус (ABSOLUTE_TRUTH, RUMOR, LIE, DEPRECATED).
•	Confidence score количественно оценивает надёжность.
•	При неоднозначности контекстный менеджер отказывается от ответа (confidence check, раздел 7.3).

14.2. Архитектурные противоречия и их разрешение
Противоречие 1: Локальность vs Точность
•	Проблема: Локальные модели (rubert-tiny2) менее точны, чем API-сервисы.
•	Решение: Использовать rubert-tiny2 (768-мерный) как компромисс. Для production — миграция на GigaEmbeddings после валидации.
Противоречие 2: Автоматизация vs Контроль
•	Проблема: Автоматическая канонизация может «зашумить» память.
•	Решение: Двухступенчатая валидация (векторная + LLM), confidence scoring, статус RUMOR для неоднозначных фактов.
Противоречие 3: Сложность vs Поддерживаемость
•	Проблема: Гибридная архитектура (HMD, RRF, валидация) сложнее монолита.
•	Решение: Модульная структура, каждый компонент тестируется изолированно. Сложность оправдана качеством памяти.
14.3. План Б (Fallback)

•	Использовать chromadb (легковесная векторная БД, pip install, не требует внешних сервисов).

•	Реализовать тонкую HTTP-прослойку (FastAPI + Uvicorn на localhost) как альтернативный транспорт.

Инициализация проекта должна происходить до использования. *Решение*: либо считать, что проект создаётся при первом обращении с неизвестным `project_name` (реализовать upsert-логику), либо добавить административный инструмент `ensure_project`. Пока помечаем `[ТРЕБУЕТ УТОЧНЕНИЯ]`.

- **Потенциальное дублирование FTS-триггеров при массовых вставках** (раздел 5.2, 8).  
  Триггеры AFTER INSERT настроены на каждую запись, что при массовой индексации файлов (множество `file_index` вставок) может замедлить работу. Отдельно триггеры для `facts_fts` синхронизируются построчно. *Решение*: рассмотреть отключение триггеров на время массовой вставки и последующую полную перестройку FTS, либо использовать внешние content-таблицы с триггерами, но указать потенциальный риск производительности.

- **Отсутствие fallback-стратегии при недоступности DeepSeek API** (раздел 9, 10.2).  
  Канонизация (микро и макро) и LLM-валидация противоречий требуют доступ к API. В разделе 14 обсуждается проблема доступа из РФ, но нет явного поведения при ошибке: должна ли канонизация откладываться, пропускаться, сохраняться в сыром виде? *Решение*: описать fallback-алгоритм: при недоступности API факты всё равно сохраняются с пониженным confidence и статусом `RUMOR`, а канонизация откладывается до восстановления связи.

- **Неопределённость в разделении зон ответственности с CLI** (раздел 3.2).  
  Утверждается, что CLI сам решает, когда вызывать `summarize_session`, но микро-канонизация автоматическая «по достижении порога». Однако у CLI нет информации о количестве сообщений, если Архив ведёт журнал, а CLI только передаёт факты. Это противоречие, указывающее на необходимость либо полной передачи управления сессией Клиенту, либо наличия серверного фонового процесса. *Решение*: рекомендовать синхронизировать пороговые значения и явно описать протокол: CLI отслеживает счётчик и вызывает инструмент микро-канонизации, либо Архив пассивно реагирует на поступление фактов, а канонизация запускается только при вызове `summarize_session` с параметром глубины.

**Дополнительные риски** (дополнение к разделу 14)

- **Риск деградации производительности гибридного поиска при большом объёме фактов и файлов**: RRF-ранжирование требует полного сканирования кандидатов из двух индексов; при десятках тысяч записей может превысить целевые 500 мс. *Митигация*: ввести пороговые значения `top_k` для каждого индекса (например, ограничить候选人 200), кеширование частых запросов.
- **Риск рассинхронизации векторных индексов**: `sqlite-vec` использует отдельные виртуальные таблицы, не синхронизированные автоматически с основными таблицами (в отличие от FTS-триггеров). Не описан механизм обновления/удаления векторов при изменении записей. *Митигация*: явно прописать триггеры/логику для поддержания `*_vec` таблиц в актуальном состоянии.
- **Риск миграции эмбеддинг-модели**: при смене модели (например, на GigaEmbeddings) размерность или семантика векторов изменится, что сделает старые индексы невалидными. *Митигация*: версионировать модель в конфигурации и хранить её идентификатор вместе с записями, предусмотреть скрипт пересчёта векторов.
- **Риск утечки чувствительной информации через промпты канонизации**: в DeepSeek отправляется полная история диалога (до 15 сообщений) для микро-канонизации. Это может включать секреты, если они обсуждались. *Митигация*: перед отправкой фильтровать сообщения по маске (например, исключать строки, похожие на ключи API) или предупреждать пользователя.

**Рекомендации по улучшению промптов канонизации**

- Промпты для микро- и макро-канонизации слишком общие, не задают жёсткий формат ответа, что увеличивает риск галлюцинаций. Рекомендуется:
  - Явно потребовать JSON-объект со строгой схемой (например, `{"facts": [{"content": ..., "type": ..., "confidence": ..., "related_entities": [...]}]}`).
  - Добавить инструкцию: «Если не уверен, лучше пропусти факт, чем придумай».
  - Указать ограничение: «Извлекай факты, только если они явно упомянуты в диалоге, не добавляй предположений».
  - Для макро-канонизации: разделить этапы (сначала summary, потом решения и долг) отдельными запросами, чтобы уменьшить сложность.
  - Использовать пример one-shot, демонстрирующий желаемый результат.

## 3. Архитектура и логика взаимодействия {#section-3}

**TL;DR**: MCP-сервер (только stdio), к которому CLI подключается как к дочернему процессу. Основные потоки: поиск контекста, добавление фактов, канонизация.

### 3.1 Компонентная схема (текстовое описание)
- **MCP Server** (точка входа) – принимает JSON-RPC через stdin/stdout.
- **Context Manager** – гибридный поиск (вектор + FTS5 + RRF).
- **Canonizer** – микро/макро-канонизация с вызовом DeepSeek.
- **Validator** – проверка дубликатов и противоречий.
- **Embedder** – обёртка `sentence-transformers`.
- **File Indexer** – обход файлов, AST-парсинг, хеширование.
- **Database** – SQLite с расширениями, управляется через SQLAlchemy.

### 3.2 Жизненный цикл
1. Cline CLI запускает Eternal Archive как дочерний процесс (stdio).
2. Handshake: сервер отдаёт список инструментов (`tools/list`).
3. Перед задачей CLI вызывает `get_project_context`.
4. В ходе работы используются `add_fact`, `add_seed`, `index_files`.
5. Канонизация: CLI может вызвать `summarize_session` (макро), микро-канонизация запускается автоматически при накоплении 10 сообщений (см. раздел 9.1 `[ТРЕБУЕТ УТОЧНЕНИЯ: механизм триггера]`).
6. Завершение сессии – вызов `summarize_session` или автоматически.

### 3.3 Ключевые решения
- **Транспорт**: исключительно stdio. HTTP (FastAPI) только как fallback (см. раздел 14.3).
- **Канонизация по требованию**: клиент инициирует макро-канонизацию.
- **Валидация при добавлении**: `add_fact` всегда проверяет факт перед сохранением.
- **Контекст в промпт**: результат `get_project_context` вставляется в промпт задачи.

## 7. Контекстный менеджер и гибридный поиск {#section-7}

**TL;DR**: Гибридный поиск = векторное сходство (cosine) + полнотекстовый BM25, объединённые Reciprocal Rank Fusion (k=60). Дополнительная проверка уверенности (confidence check) отсекает нерелевантные результаты.

### 7.1 Алгоритм `get_project_context` (пошагово)

1. Получить `project_id` по `project_name` (если отсутствует – создать? `[ТРЕБУЕТ УТОЧНЕНИЯ: автосоздание проекта]`).
2. Векторизовать `query` с помощью текущей модели эмбеддингов.
3. Для каждого включённого типа (`facts`, `files`, `seeds`, `quests`, `entities`):
   - **Векторный поиск**: выполнить запрос к соответствующей `*_vec` таблице, получить top-N (например, 50) с косинусным расстоянием.
   - **Полнотекстовый поиск** (только для типов, имеющих FTS-индекс: `facts`, `seeds`, `files`): выполнить `SELECT ... FROM *_fts WHERE ... MATCH ...`, получить BM25-ранжированный список (top-N).
   - [Для `entities` FTS нет, поэтому использовать только векторный поиск, либо для их извлечения полагаться на связи с найденными фактами.] `[ТРЕБУЕТ УТОЧНЕНИЯ]`
4. Применить `reciprocal_rank_fusion()` (см. 7.2) к полученным спискам для каждого типа, объединяя векторный и FTS ранги.
5. Выбрать `top_k` лучших результатов каждого типа.
6. Собрать ответ, дополнив факты связанными сущностями через `fact_entity_link`.
7. Выполнить `confidence_check()` (см. 7.3) для общего набора фактов+файлов. Если проверка не пройдена, можно либо вернуть пустой результат, либо пометить флаг `low_confidence` (решение на усмотрение реализации, в ТЗ не уточнено).

### 7.2 RRF-ранжирование

```python
def reciprocal_rank_fusion(
    vector_results: list[tuple[int, float]],  # (id, cosine_sim)
    fts_results: list[tuple[int, float]],     # (id, bm25_score)
    k: int = 60,
) -> list[tuple[int, float]]:
    scores = {}
    for rank, (doc_id, _) in enumerate(vector_results, start=1):
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    for rank, (doc_id, _) in enumerate(fts_results, start=1):
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
```

### 7.3 Проверка уверенности

```python
def confidence_check(
    fused_results: list[tuple[int, float]],
    top_score_threshold: float = 0.25,
    gap_threshold: float = 0.15,
) -> tuple[bool, str]:
    if not fused_results or fused_results[0][1] < top_score_threshold:
        return False, "Нет достаточно релевантных результатов."
    if len(fused_results) >= 2:
        gap = fused_results[0][1] - fused_results[1][1]
        if gap < gap_threshold:
            return False, f"Результаты недостаточно различимы (gap={gap:.3f})."
    return True, "OK"
```

**Связь с валидацией**: этот же confidence check применяется к результатам поиска, чтобы избежать показа слабых результатов. См. раздел 9.3.

## 9. Канонизация и управление памятью {#section-9}

**TL;DR**: Два режима: микро (автоматическая по N сообщениям) и макро (по вызову). Обе используют DeepSeek с промптами для извлечения фактов.

### 9.1 Микро-канонизация

**Триггер**: Автоматический после накопления 10 новых сообщений в сессии. `[ТРЕБУЕТ УТОЧНЕНИЯ: механизм запуска – скорее всего внутри сервера, но инструмент не предусмотрен. Рекомендуется: добавить внутренний хук при добавлении сообщения или инструмент, вызываемый CLI.]`

**Процесс**:
1. Взять последние 15 сообщений (текущий блок + 5 контекстных).
2. Отправить в DeepSeek улучшенный промпт (см. ниже).
3. Распарсить JSON-массив фактов.
4. Каждый факт провалидировать (раздел 9.3).
5. Сохранить валидные факты.

**Улучшенный промпт** (рекомендация):
```text
Ты — Архивариус. Извлеки из истории диалога новые факты о проекте. Верни JSON-объект со строгим форматом:
{
  "facts": [
    {
      "content": "<строка>",
      "type": "decision|architecture|preference|debt|general",
      "confidence": <0.0-1.0>,
      "related_entities": ["<имя>", ...]
    }
  ]
}
Если нет новых фактов, верни {"facts": []}.
Не придумывай факты, которых нет в диалоге. Извлекай только явно упомянутые.
```

### 9.2 Макро-канонизация (`summarize_session`)

**Триггер**: Явный вызов инструмента или при завершении сессии.

**Процесс**:
1. Получить все сообщения сессии.
2. Отправить в DeepSeek с промптом:
```text
Ты — Архивариус. Проведи полную канонизацию сессии. Верни JSON:
{
  "summary": "<3-5 предложений>",
  "key_decisions": [
    {"content": "...", "type": "decision", "confidence": 0.9, "related_entities": ["..."]}
  ],
  "new_debt": [
    {"content": "...", "type": "debt", "priority": 5}
  ],
  "updated_facts": [
    {"fact_id": <id>, "new_content": "..."}
  ]
}
```
3. Сохранить:
   - `summary` → `sessions.summary`.
   - `key_decisions` → новые записи `facts`.
   - `new_debt` → новые записи `seeds`.
   - `updated_facts` → обновить существующие факты, записать изменения в `fact_history`.

### 9.3 Валидация фактов

**Выполняется автоматически при вызове `add_fact` и во время канонизации.**

Алгоритм:
1. **Векторный поиск дубликатов**: найти топ-3 ближайших факта в `facts_vec` (проект).  
   - Если косинусное расстояние < 0.1 → считать дубликатом, отклонить, вернуть существующий ID.
2. **Проверка противоречий**: если расстояние < 0.3 → возможное противоречие.  
   - Отправить оба факта в DeepSeek с вопросом: «Противоречат ли следующие два утверждения?»  
   - Ответ: «да» или «нет».  
   - Если противоречие подтверждено: новый факт сохраняется со статусом `RUMOR` и `confidence` < 0.5. Старый факт не меняется.  
   - Если нет противоречия: статус `ABSOLUTE_TRUTH`.
3. Если расстояние ≥ 0.3 → сохраняется как `ABSOLUTE_TRUTH` с исходным confidence.

**Ссылки**: см. раздел 7.3 для confidence check контекстного поиска.

## 10. Безопасность и ограничения {#section-10}

**TL;DR**: Риски минимизированы локальным хранением, параметризованными запросами, confidence scoring. Ограничение доступа к файловой системе.

### 10.1 Модель угроз

| Угроза | Уровень риска | Меры защиты |
|--------|---------------|-------------|
| Несанкционированный доступ к БД | Низкий | Права `chmod 600`, отдельный пользователь ОС |
| SQL-инъекция | Средний | Параметризованные запросы SQLAlchemy |
| Галлюцинации LLM при канонизации | Средний | Валидация, статусы фактов, confidence scores |
| Утечка данных через DeepSeek API | Средний | Минимизация отправляемых данных (только факты) |
| Переполнение контекстного окна | Низкий | Динамический `top_k`, лимит символов |

### 10.2 Ограничения реализации
- Чтение файлов: строго в пределах `project_root` (проверка `pathlib`).
- DeepSeek API используется **только** для:
  - извлечения фактов (микро/макро),
  - разрешения противоречий (валидация),
  - (опционально) генерации summary для файлов.
- Все данные хранятся локально в SQLite.

### 10.3 Защита от галлюцинаций
- Обязательная валидация каждого извлечённого факта (см. 9.3).
- Статусы: `ABSOLUTE_TRUTH`, `RUMOR`, `LIE`, `DEPRECATED`.
- Количественная оценка уверенности `confidence`.
- Отказ от нерелевантного контекста (confidence check в 7.3).

## 11. Тестирование {#section-11}

**TL;DR**: pytest + pytest-asyncio, изолированные тесты с in-memory SQLite. Ключевые сценарии покрывают добавление фактов, детекцию дубликатов, противоречий, индексацию, гибридный поиск.

- **Модульные**: все компоненты (`canonizer`, `validator`, `embedder`, `context_manager`) тестируются отдельно с замоканными зависимостями.

4. `test_file_indexer` – создать временный Python-файл, проиндексировать, проверить символы и хеш.

| **Деградация производительности гибридного поиска** | Средняя | Среднее | Ограничение кандидатов top-N (200), кеширование частых запросов |
| **Расхождение векторных индексов** | Высокая | Высокое | Явные триггеры или фоновый процесс синхронизации `*_vec` |

### 14.2 Архитектурные противоречия (разрешены)
- **Локальность vs точность**: компромиссная модель rubert-tiny2, в перспективе GigaEmbeddings.
- **Автоматизация vs контроль**: двухступенчатая валидация + confidence.
- **Сложность vs поддерживаемость**: модульная структура, изолированное тестирование.

### 14.3 План Б (Fallback)
- Если sqlite-vec не работает: использовать `chromadb`.
- Если stdio-подключение не удаётся: добавить HTTP-прослойку (FastAPI на localhost).

### 1. Как запускается микро‑канонизация?

**Проблема**  
В ТЗ написано, что микро‑канонизация (превращение последних сообщений в факты) должна происходить сама каждые 10 сообщений. Но не сказано, кто это делает: сервер «Вечного архива» сам следит за количеством или клиент (Cline) должен дать команду.

**Варианты решения**

- **Вариант А. Сервер сам всё считает**  
  Внутри сервера заводится счётчик новых сообщений. Когда CLI отправляет очередное сообщение (через специальный инструмент, который мы добавим), счётчик увеличивается. При достижении 10 сервер сам запускает микро‑канонизацию, не дожидаясь команды извне.  
  *Плюс*: клиенту не нужно помнить о канонизации.  
  *Минус*: сервер становится сложнее.

- **Вариант Б. Клиент командует**  
  Cline CLI сам считает, сколько сообщений он отправил, и когда набирается 10, вызывает отдельный инструмент, например `run_micro_canonization()`. Сервер просто выполняет команду.  
  *Плюс*: логика остаётся снаружи, сервер проще.  
  *Минус*: если клиент забудет вызвать, канонизации не будет.

- **Вариант В. Смешанный**  
  Сервер хранит счётчик и при каждом добавлении факта или сообщения проверяет порог. Если порог достигнут, сервер сам запускает канонизацию, но также разрешает клиенту вызвать её досрочно (например, при завершении небольшой задачи). Так CLI может ускорить процесс, но и автоматика не даст забыть.

**Рекомендация**  
Для MVP проще всего вариант Б: добавить инструмент `micro_canonize`, который CLI дёргает раз в 10 сообщений. Это не требует от сервера хранить состояние, а Cline уже умеет считать свои сообщения.

### 2. Кто обновляет векторные индексы?

**Проблема**  
В базе данных есть отдельные таблицы для векторов (`facts_vec`, `entities_vec` и т.д.). Они не обновляются автоматически, когда меняются обычные записи. В ТЗ сказано только про триггеры для полнотекстового поиска, но не для векторов.

**Варианты решения**

- **Вариант А. Добавить ручную синхронизацию в коде**  
  Каждый раз, когда программа добавляет/изменяет факт, она тут же вставляет или обновляет строчку в векторной таблице. Разработчик пишет в том же методе «сохранить в facts» и сразу «сохранить в facts_vec».  
  *Риск*: легко забыть сделать это в новом месте кода, и векторы начнут расходиться.

- **Вариант Б. Триггеры на уровне приложения (SQLAlchemy events)**  
  Использовать механизм «событий» библиотеки SQLAlchemy: при добавлении записи в таблицу `facts` автоматически вызывается дополнительная функция, которая обновляет `facts_vec`. Это похоже на триггеры, но работает на стороне Python, а не SQLite.  
  *Плюс*: централизованно, один раз настроил – везде работает.  
  *Минус*: зависит от ORM, не сработает, если кто-то будет писать запросы в обход SQLAlchemy.

- **Вариант В. Триггеры SQLite с расширением**  
  Попробовать настроить триггеры в самой базе, которые автоматически копируют данные в векторные таблицы. К сожалению, `sqlite-vec` не поддерживает полноценные триггеры, работающие с виртуальными таблицами, поэтому этот вариант ненадёжен.

**Рекомендация**  
Вариант Б (события SQLAlchemy) — самый чистый и безопасный для проекта, где вся работа с БД идёт через ORM. В документации нужно прописать, что все операции с фактами/сущностями/файлами должны проходить через соответствующие CRUD-функции.

### 3. Откуда берутся session_id и source_message_id у факта?

**Проблема**  
У факта есть поля `session_id` и `source_message_id`, которые связывают его с конкретным разговором и сообщением. Но в инструменте `add_fact` эти поля не передаются. Значит, сервер должен сам их откуда-то взять. Непонятно, откуда.

**Варианты решения**

- **Вариант А. Сервер хранит «текущую сессию»**  
  Когда CLI начинает работу, он вызывает инструмент `start_session`, а когда заканчивает — `end_session`. Между этими вызовами все факты автоматически привязываются к последней открытой сессии.  
  *Нужно*: добавить два новых инструмента и хранить `current_session_id` внутри сервера.

- **Вариант Б. Клиент передаёт идентификатор сессии в каждом вызове**  
  Cline CLI сам помнит номер сессии и добавляет его параметром в `add_fact`. Тогда сервер не хранит состояние.  
  *Плюс*: сервер остаётся «безголовым» (stateless).  
  *Минус*: клиент должен вести учёт сессий и сообщений.

- **Вариант В. Не использовать эти поля для MVP**  
  Для первой версии можно просто не заполнять `session_id` и `source_message_id` (оставить их пустыми). Историю диалогов всё равно временно некуда сохранять, потому что нет инструмента для добавления сообщений.  
  *Самый простой способ*, носящий временный характер.

**Рекомендация**  
Для MVP лучше пойти по пути В (оставить поля пустыми) и добавить инструменты работы с сессией позже. Если нужна связь с сообщениями, выбрать вариант А, потому что он не усложняет жизнь клиенту.

### 4. Должны ли проекты создаваться автоматически?

**Проблема**  
Все инструменты требуют `project_name`, но не сказано, что делать, если такого проекта ещё нет в базе. Создавать ли его тихо или возвращать ошибку?

**Варианты решения**

- **Вариант А. Автоматическое создание**  
  Если проект с таким именем не найден, сервер сам создаёт новый проект типа `cline` с пустыми настройками и продолжает работу.  
  *Удобно*: пользователь просто начинает писать, не думая о создании проекта.  
  *Риск*: опечатка в названии создаст новый пустой проект, а старые данные покажутся потерянными.

- **Вариант Б. Только явное создание**  
  Сервер возвращает ошибку «проект не найден». Администратор (или сам Cline первым делом) должен вызвать специальный инструмент `create_project`.  
  *Надёжнее*: нет случайных созданий.  
  *Минус*: требуется лишнее действие при первом использовании.

- **Вариант В. Создавать по явному запросу с флагом**  
  В инструмент `add_fact` или `get_context` добавить необязательный параметр `create_if_missing=False`. По умолчанию проект не создаётся, но можно явно попросить.  
  *Гибко*, но усложняет интерфейс.

**Рекомендация**  
Для фазы с одним разработчиком и одним проектом вариант А самый простой. Если система будет использоваться с несколькими проектами, лучше вариант Б, чтобы избежать путаницы.

### 5. Как искать по сущностям (entities)?

**Проблема**  
В результатах поиска `get_project_context` есть раздел `entities` (например, «AuthModule»), но непонятно, как они попадают в выдачу. Должны ли они искаться сами по себе или только «прицепляться» к найденным фактам?

**Варианты решения**

- **Вариант А. Искать сущности отдельно по векторному сходству**  
  Для запроса считается вектор, сравнивается с векторами всех сущностей, и наиболее похожие попадают в ответ. Это требует векторного индекса для сущностей (он уже есть, `entities_vec`).  
  *Поведение*: сущности могут находиться, даже если по ним нет ни одного факта.

- **Вариант Б. Показывать только сущности, связанные с найденными фактами**  
  Сначала ищутся факты/файлы как обычно. Потом для каждого найденного факта через таблицу связей (`fact_entity_link`) подтягиваются имена сущностей и выводятся отдельным списком.  
  *Поведение*: сущности появляются, только если есть подходящий факт.

- **Вариант В. Комбинировать**  
  Искать сущности напрямую (векторно), но с меньшим приоритетом, а факты могут дополнительно «подсвечивать» связанные с ними сущности.

**Рекомендация**  
В примере из ТЗ сущности явно выводятся отдельно, причём не только те, что в `related_entities`. Значит, вероятно, подразумевался вариант А. Нужно реализовать прямой векторный поиск по `entities_vec` и добавить его в RRF-объединение (хотя FTS для сущностей нет, только вектор). Либо использовать вариант Б, но тогда в выдаче сущностей будет мало, если фактов не нашлось. Разработчику стоит явно выбрать стратегию и зафиксировать.

### 6. Что делать с описанием файла, если нет комментариев?

**Проблема**  
При индексации файлов нужно создать краткое описание (`summary`). Если в коде есть хорошие комментарии или docstring, их берут. А если нет? В ТЗ написано «опциональный вызов DeepSeek». Но у заказчика DeepSeek может быть платным, и неясно, включена ли эта опция по умолчанию.

**Варианты решения**

- **Вариант А. Ничего не генерировать**  
  Если нет комментариев, `summary` остаётся пустым. Поиск всё равно будет работать по ключевым символам и имени файла.  
  *Просто и дёшево*, но описания не будет.

- **Вариант Б. Всегда вызывать DeepSeek**  
  Для каждого файла без описания отправлять запрос к модели: «Опиши одной фразой, что делает этот код».  
  *Даёт полные описания*, но тратит деньги и время (особенно при первой индексации большого проекта).

- **Вариант В. Сделать настройку**

### 7. Нужен ли сервис systemd, если общение идёт через стандартный ввод/вывод?

**Проблема**  
Cline CLI запускает «Вечный архив» как свою дочернюю программу, общаясь с ней через pipe (stdin/stdout). Значит, сервер не должен висеть в фоне как самостоятельный демон — его жизнь начинается и заканчивается вместе с CLI. Однако в ТЗ приведён пример настройки systemd-сервиса, который постоянно висит в системе и слушает сокет. Это противоречит модели «запуск по требованию».

**Варианты решения**

- **Вариант А. Убрать systemd-сервис**  
  В инструкции по деплою прописать, что автозапуск не нужен, потому что Cline CLI сам управляет процессом. Все настройки окружения передаются через конфиг CLI.  
  *Честно и просто*.

- **Вариант Б. Поддерживать два режима**  
  Сервер может работать в режиме stdio (когда запущен без аргументов) и в режиме HTTP (если передан флаг `--http`). Тогда systemd-сервис используется только для HTTP-резервного варианта.  
  *Избыточно для MVP*, но может пригодиться позже.

- **Вариант В. Использовать systemd для CLI, а не для архива**  
  Вместо сервера в systemd заворачивается сам Cline CLI (в tmux или как сервис), а архив стартует как дочерний процесс внутри. Тогда systemd отвечает только за то, чтобы CLI всегда работал.  
  *Менее путаницы*.

**Рекомендация**  
Пока клиент — только Cline CLI, systemd для «Вечного архива» не нужен. В документации для деплоя достаточно описать ручной запуск или использование tmux. Пункт про systemd стоит удалить или пометить как неактуальный для MVP.

### 3. Архитектура системы и логика взаимодействия

**TL;DR:** Cline CLI запускает дочерний процесс «Вечного Архива» и общается с ним через stdin/stdout в формате JSON-RPC. Основные циклы: поиск контекста, добавление факта, канонизация.

#### 3.1. Компонентная схема (логическая)

1.  **Точка входа (`src/mcp_server.py`):** Инициализирует FastMCP сервер, подключает все инструменты.
2.  **Менеджер БД (`src/database/`):** SQLAlchemy модели, движок, CRUD-операции. **Весь доступ к БД — только через этот слой.**
3.  **Подсистема индексации (`src/indexing/`):** Обходит файлы в `project_root`, вычисляет SHA256, парсит код, генерирует описания.
4.  **Ядро памяти (`src/memory/`):**
    -   `context_manager.py`: Реализует гибридный поиск (векторный + FTS5) с RRF-ранжированием.
    -   `canonizer.py`: Взаимодействует с DeepSeek API для извлечения фактов из диалогов.
    -   `validator.py`: Проверяет новые факты на дубликаты и противоречия.
    -   `embedder.py`: Обёртка вокруг `sentence-transformers` для векторизации текста.

#### 3.2. Жизненный цикл (процесс работы)

1.  **Запуск:** Разработчик запускает Cline CLI в headless-режиме.
2.  **Подключение:** CLI читает свой конфиг и запускает `python -m src.mcp_server` как дочерний процесс.
3.  **Рукопожатие:** Архив отправляет CLI список доступных инструментов (`tools/list`).
4.  **Получение контекста:** Перед началом задачи CLI (по инициативе LLM внутри него) вызывает `get_project_context`.
5.  **Рабочий цикл:** В процессе работы CLI вызывает `add_fact`, `add_seed`, `index_files`.
6.  **Микро-канонизация:** CLI считает израсходованные токены. При превышении порога (например, 200 000) вызывает инструмент `run_micro_canonization`.
7.  **Макро-канонизация:** При завершении сессии CLI вызывает `summarize_session`.

#### 3.3. Ключевые проектные решения

-   **Транспорт:** Только `stdio`. Никакого HTTP в основном режиме.
-   **Инициатива канонизации:** Клиент (Cline CLI) считает токены и командует микро-канонизацией.
-   **Валидация при добавлении:** Каждый факт при вызове `add_fact` проверяется автоматически.
-   **Управление сессиями:** Сервер хранит состояние текущей сессии (`session_id`) и связывает с ней все новые факты.

### 5. Схема данных

**TL;DR:** Реляционная схема SQLite с таблицами для проектов, сессий, сообщений, фактов, сущностей, семян, квестов и индекса файлов. Полнотекстовые и векторные индексы синхронизируются через события SQLAlchemy.

#### 5.1. Полная SQL-схема

```sql
-- Проекты. Создаются ЯВНО инструментом create_project.
CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    type TEXT NOT NULL CHECK (type IN ('cline', 'exis')),
    root_path TEXT,                 -- корневая директория (только для 'cline')
    metadata TEXT                   -- JSON: доп. настройки, описание
);

-- Сессии. Создаются и завершаются инструментами start_session и summarize_session.
CREATE TABLE sessions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    started_at TEXT DEFAULT (datetime('now')),
    ended_at TEXT,
    summary TEXT,                   -- результат макро-канонизации
    context_snapshot TEXT,          -- JSON: снимок контекста на конец сессии
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);

-- Сообщения. Сохраняются через внутренний метод при каждом взаимодействии.
CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id INTEGER NOT NULL,
    role TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),
    content TEXT NOT NULL,
    timestamp TEXT DEFAULT (datetime('now')),
    metadata TEXT,                  -- JSON: токены, вызовы инструментов
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE
);
CREATE INDEX idx_messages_session ON messages(session_id, timestamp);

-- Сущности (модули, классы, функции, игровые персонажи и т.д.).
CREATE TABLE entities (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    name TEXT NOT NULL,
    type TEXT NOT NULL,
    description TEXT,
    metadata TEXT,                  -- JSON: атрибуты, характеристики
    embedding BLOB NOT NULL,        -- float32[768]
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);

-- Факты (элементы памяти).
CREATE TABLE facts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    session_id INTEGER,             -- Автоматически подставляется из активной сессии
    source_message_id INTEGER,      -- Автоматически подставляется из последнего сообщения
    content TEXT NOT NULL,
    type TEXT NOT NULL DEFAULT 'general'
        CHECK (type IN ('decision', 'architecture', 'preference', 'debt', 'world_lore', 'general')),
    status TEXT NOT NULL DEFAULT 'ABSOLUTE_TRUTH'
        CHECK (status IN ('ABSOLUTE_TRUTH', 'RUMOR', 'LIE', 'DEPRECATED')),
    confidence REAL DEFAULT 1.0 CHECK (confidence >= 0.0 AND confidence <= 1.0),
    embedding BLOB NOT NULL,        -- float32[768]
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE SET NULL,
    FOREIGN KEY (source_message_id) REFERENCES messages(id) ON DELETE SET NULL
);

-- История изменений фактов.
CREATE TABLE fact_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    fact_id INTEGER NOT NULL,
    previous_content TEXT,
    new_content TEXT,
    changed_at TEXT DEFAULT (datetime('now')),
    change_reason TEXT,
    FOREIGN KEY (fact_id) REFERENCES facts(id) ON DELETE CASCADE
);

-- Связи фактов с сущностями.
CREATE TABLE fact_entity_link (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    fact_id INTEGER NOT NULL,
    entity_id INTEGER NOT NULL,
    link_type TEXT DEFAULT 'related',
    FOREIGN KEY (fact_id) REFERENCES facts(id) ON DELETE CASCADE,
    FOREIGN KEY (entity_id) REFERENCES entities(id) ON DELETE CASCADE,
    UNIQUE(fact_id, entity_id)
);

-- Семена (неразвитые идеи, технический долг).
CREATE TABLE seeds (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    content TEXT NOT NULL,
    type TEXT NOT NULL DEFAULT 'idea'
        CHECK (type IN ('idea', 'debt', 'bug', 'feature')),
    priority INTEGER DEFAULT 0,
    status TEXT NOT NULL DEFAULT 'open'
        CHECK (status IN ('open', 'in_progress', 'resolved')),
    embedding BLOB NOT NULL,        -- float32[768]
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);

-- Квесты / Активные ветки разработки.
CREATE TABLE quests (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    title TEXT NOT NULL,
    description TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'active'
        CHECK (status IN ('active', 'completed', 'failed', 'blocked')),
    embedding BLOB NOT NULL,        -- float32[768]
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE
);

-- Индексация файлов (только для 'cline'-проектов).
CREATE TABLE file_index (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL,
    file_path TEXT NOT NULL,        -- относительный путь
    file_name TEXT NOT NULL,        -- только имя файла
    extension TEXT,
    summary TEXT,
    key_symbols TEXT,               -- JSON
    dependencies TEXT,              -- JSON
    content_hash TEXT NOT NULL,     -- SHA256
    last_indexed TEXT DEFAULT (datetime('now')),
    embedding BLOB NOT NULL,        -- float32[768]
    FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
    UNIQUE(project_id, file_path)
);

-- Полнотекстовые индексы FTS5
CREATE VIRTUAL TABLE facts_fts USING fts5(content, type, status, content=facts, content_rowid=id);
CREATE VIRTUAL TABLE seeds_fts USING fts5(content, type, content=seeds, content_rowid=id);
CREATE VIRTUAL TABLE file_index_fts USING fts5(file_path, file_name, summary, key_symbols, content=file_index, content_rowid=id);

-- Триггеры для автоматической синхронизации FTS
-- (реализуются через события SQLAlchemy, здесь — для справки)
-- CREATE TRIGGER facts_ai AFTER INSERT ON facts BEGIN
--     INSERT INTO facts_fts(rowid, content, type, status) VALUES (new.id, new.content, new.type, new.status);
-- END;

-- Векторные индексы sqlite-vec
CREATE VIRTUAL TABLE facts_vec USING vec0(embedding FLOAT[768]);
CREATE VIRTUAL TABLE entities_vec USING vec0(embedding FLOAT[768]);
CREATE VIRTUAL TABLE seeds_vec USING vec0(embedding FLOAT[768]);
CREATE VIRTUAL TABLE file_index_vec USING vec0(embedding FLOAT[768]);
```

#### 5.2. Механизм синхронизации векторных индексов

**РЕАЛИЗОВАТЬ:** Все операции записи в таблицы `facts`, `entities`, `seeds`, `file_index` должны проходить **исключительно** через CRUD-функции в `src/database/crud.py`. Внутри этих функций, в рамках той же транзакции, необходимо выполнять соответствующие операции с векторными таблицами (`facts_vec`, `entities_vec` и т.д.) и полнотекстовыми индексами.

Для гарантии атомарности использовать события SQLAlchemy (`after_insert`, `after_update`, `after_delete`), которые слушают изменения в основных моделях и автоматически обновляют виртуальные таблицы. Это исключит человеческий фактор при написании нового кода.

### 7. Контекстный менеджер и гибридный поиск

**TL;DR:** Сердце поиска. Комбинирует результаты двух параллельных поисков (векторного и текстового) с помощью Reciprocal Rank Fusion, а затем проверяет, насколько результат надёжен.

#### 7.1. Алгоритм Reciprocal Rank Fusion (RRF)

```python
def reciprocal_rank_fusion(
    vector_results: list[tuple[int, float]],  # (id, cosine_similarity)
    fts_results: list[tuple[int, float]],     # (id, bm25_score)
    k: int = 60,
) -> list[tuple[int, float]]:
    scores = {}
    for rank, (doc_id, _) in enumerate(vector_results, start=1):
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    for rank, (doc_id, _) in enumerate(fts_results, start=1):
        scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
```

#### 7.2. Проверка уверенности (Confidence Check)

```python
def confidence_check(
    fused_results: list[tuple[int, float]],
    top_score_threshold: float = 0.25,
    gap_threshold: float = 0.15,
) -> tuple[bool, str]:
    if not fused_results or fused_results[0][1] < top_score_threshold:
        return False, "Нет достаточно релевантных результатов."
    if len(fused_results) >= 2:
        gap = fused_results[0][1] - fused_results[1][1]
        if gap < gap_threshold:
            return False, f"Результаты недостаточно различимы (gap={gap:.3f})."
    return True, "OK"
```

### 8. Подсистема индексации файлов

**TL;DR:** Сканирует файлы проекта, читает только разрешённые, вычисляет хеш для инкрементального обновления, парсит код и всегда генерирует описание.

3.  **Парсинг:**
    -   Для `.py` файлов использовать модуль `ast` для извлечения имён классов, функций и импортов.

4.  **Генерация описания (`summary`):**

5.  **Сохранение:**
    -   Векторизовать строку `summary + " " + key_symbols`.
    -   Сохранить/обновить запись в `file_index`.
    -   Обновить FTS и векторный индексы.

### 9. Канонизация и управление памятью

**TL;DR:** Процесс превращения сырых диалогов в структурированные факты. Микро-канонизация вызывается клиентом по токенам, макро-канонизация — в конце сессии.

#### 9.1. Микро-канонизация (`run_micro_canonization`)

**Триггер:** Cline CLI отслеживает использование токенов (счётчик внутри LLM или оценка на стороне клиента). При превышении порога `MICRO_CANONIZATION_TOKEN_THRESHOLD` (значение по умолчанию: `200000`) CLI вызывает `run_micro_canonization`.

**Процесс:**
1.  Сервер берёт последние 15 сообщений из активной сессии.
2.  Формирует промпт и отправляет в DeepSeek API:
    ```text
    Ты — Архивариус. Твоя задача — извлечь новые, явно упомянутые факты о проекте из фрагмента диалога.
    ВЕРНИ СТРОГО JSON-ОБЪЕКТ СЛЕДУЮЩЕГО ФОРМАТА, БЕЗ ЛИШНИХ СЛОВ:
    {
      "facts": [
        {
          "content": "текст факта",
          "type": "decision|architecture|preference|debt|general",
          "confidence": <число от 0.0 до 1.0>,
          "related_entities": ["имя1", "имя2"]
        }
      ]
    }
    ПРАВИЛА:
    - Извлекай факты, только если они ЯВНО упомянуты в диалоге. НЕ ПРИДУМЫВАЙ.
    - Если не можешь определить тип, используй "general".
    - Если новых фактов нет, верни {"facts": []}.
    ```
3.  Парсит JSON-ответ.
4.  Для каждого факта в массиве:
    -   Вызывает процедуру валидации (см. 9.3).
    -   Сохраняет валидные факты в БД.

#### 9.2. Макро-канонизация (`summarize_session`)

**Триггер:** Явный вызов инструмента клиентом при завершении задачи или сессии.

**Процесс:**
1.  Сервер извлекает **все** сообщения сессии.
2.  Формирует промпт для DeepSeek:
    ```text
    Ты — Архивариус. Проведи полную канонизацию сессии. Проанализируй диалог и верни СТРОГО JSON:
    {
      "summary": "Краткое содержание сессии, 3-5 предложений.",
      "key_decisions": [
        {"content": "описание решения", "type": "decision", "confidence": 0.9, "related_entities": ["..."], "priority": 5}
      ],
      "new_debt": [
        {"content": "описание долга", "type": "debt", "priority": 5}
      ],
      "updated_facts": [
        {"fact_id": <id>, "new_content": "новый текст факта"}
      ]
    }
    Если обновлённых фактов нет, оставь массив updated_facts пустым.
    ```
3.  Сохраняет результаты:
    -   `summary` → поле `sessions.summary`.
    -   `key_decisions` → новые записи в `facts` (с валидацией).
    -   `new_debt` → новые записи в `seeds`.
    -   `updated_facts` → обновление существующих фактов и запись в `fact_history`.
4.  Проставляет `ended_at` сессии.

#### 9.3. Валидация фактов

**Выполняется для каждого нового факта из `add_fact` и канонизации.**

**Алгоритм:**
1.  **Поиск дубликата (векторный):**
    -   Найти top-3 ближайших факта в `facts_vec` для текущего проекта.
    -   Если косинусное расстояние до ближайшего < 0.1, факт считается дубликатом.
    -   **Действие:** Отклонить новый факт, вернуть ID существующего.
2.  **Проверка противоречия (векторный + LLM):**
    -   Если расстояние до ближайшего находится в диапазоне `0.1 <= расстояние < 0.3`, существует риск противоречия.
    -   Отправить оба факта в DeepSeek с вопросом: «Противоречат ли эти два утверждения друг другу? Ответь только 'да' или 'нет'.»
    -   Если ответ «да», сохранить новый факт со статусом `RUMOR` и `confidence` < 0.5.
    -   Если ответ «нет», сохранить как `ABSOLUTE_TRUTH`.
3.  **Уверенное сохранение:**
    -   Если расстояние >= 0.3, сохранить факт как `ABSOLUTE_TRUTH` с исходной уверенностью.

### 10. Безопасность и ограничения

**TL;DR:** Локальное хранение, минимальные права, жёсткий контроль доступа к файловой системе, обязательная валидация всего, что приходит от LLM.

| Угроза | Уровень риска | Меры защиты |
| :--- | :---: | :--- |
| Несанкционированный доступ к БД | Низкий | Права файла `0600`, отдельный пользователь ОС. |
| SQL-инъекция | Средний | Параметризованные запросы SQLAlchemy. Разработка строго через CRUD. |
| Галлюцинации LLM при канонизации | Средний | Двухступенчатая валидация, статусы `RUMOR`, `confidence scoring`. |
| Утечка данных через DeepSeek API | Средний | Минимизация отправляемых данных. Отправляются только извлечённые факты/диалоги, но не полный код. |
| Переполнение контекстного окна | Низкий | Динамический `top_k` и лимит символов. |
| Выход за пределы директории (Path Traversal) | Высокий | При индексации **обязательная** проверка: `Path(file_path).resolve()` должен начинаться с `Path(project_root).resolve()`. |

**Жёсткое ограничение на чтение файлов:**
```python
def is_safe_path(project_root: str, file_path: str) -> bool:
    root = Path(project_root).resolve()
    target = root.joinpath(file_path).resolve()
    return target.is_relative_to(root)
```

### 11. Тестирование

**Стратегия:** Полное покрытие компонентов модульными и интеграционными тестами с использованием `pytest` и `pytest-asyncio`.

**Ключевые сценарии:**

5.  `test_hybrid_search_effectiveness`: Сравнить точность и полноту гибридного поиска с чисто векторным и чисто текстовым на наборе данных.

### 12. Деплой и эксплуатация

**TL;DR:** Установка на Ubuntu-сервер, настройка окружения, интеграция с Cline CLI. Systemd-сервис НЕ ИСПОЛЬЗУЕТСЯ.

**Требования к VPS:**
-   **OS:** Ubuntu 22.04/24.04 LTS
-   **RAM:** минимум 4 ГБ (рекомендуется 8 ГБ)
-   **Диск:** 20+ ГБ SSD
-   **CPU:** 2+ ядра

**Процесс установки (кратко):**
1.  Установить системные зависимости: `python3.11`, `git`, `curl`, `nodejs` (для CLI).
2.  Клонировать репозиторий в `/opt/eternal-archive`.
3.  Создать виртуальное окружение, установить зависимости из `requirements.txt`.
4.  Создать и заполнить `.env` файл.
5.  Запустить миграции: `alembic upgrade head`.
6.  Установить и настроить Cline CLI, прописав путь к `src.mcp_server` в его конфиге.

**Запуск:** Процесс «Вечного Архива» запускается и останавливается исключительно как дочерний процесс Cline CLI. Никакого systemd-сервиса для сервера не создаётся.

**Проверка работоспособности:**
```bash
cd /opt/eternal-archive
source venv/bin/activate
echo '{"jsonrpc":"2.0","id":1,"method":"tools/list"}' | python -m src.mcp_server
```

### 14. Риски, подводные камни и разрешённые противоречия

В этом разделе собраны все выявленные в ходе анализа проблемы и **финальные решения по ним**.

#### 14.1. Разрешённые архитектурные противоречия

| Противоречие | Решение |
| :--- | :--- |
| **Автоматизация канонизации vs Контроль** | Микро-канонизация запускается **клиентом** по порогу токенов. Это даёт контроль и предсказуемость. |
| **Локальность vs Точность эмбеддингов** | Использование `rubert-tiny2` как стартового компромисса. Архитектура предусматривает лёгкую замену модели в будущем. |
| **Сложность архитектуры vs Поддерживаемость** | Оправдана качеством памяти. Модульность и обязательное тестирование компонентов нивелируют риск. |

#### 14.2. Карта рисков и их митигация

| Риск | Влияние | Вероятность | Митигация |
| :--- | :---: | :---: | :--- |
| Нестабильность `sqlite-vec` на целевом Linux | Высокое | Средняя | Предварительное тестирование. Fallback: замена на `chromadb`. |
| Недоступность DeepSeek API (сетевые проблемы, блокировки) | Высокое | Высокая | **Fallback-стратегия при ошибке API:** факты сохраняются со статусом `RUMOR` и `confidence=0.4`. Канонизация откладывается до восстановления доступа. Резервный прокси-провайдер. |
| Деградация производительности при большом числе фактов (>10 000) | Среднее | Средняя | Ограничение кандидатов для RRF (top-100). Кеширование частых запросов. Регулярный `VACUUM`. |
| Расхождение векторных индексов с основными данными | Высокое | Средняя | Исключительное использование CRUD-функций и событий SQLAlchemy для всех операций записи. Добавление теста, проверяющего консистентность данных. |
| Смена модели эмбеддингов в будущем | Высокое | Низкая | Поле `embedding` должно быть не `BLOB`, а `BLOB + version`. Добавление скрипта миграции для пересчёта векторов. |

-   **MCP (Model Context Protocol):** Протокол для подключения AI-ассистентов к внешним инструментам.
-   **FTS5:** Модуль полнотекстового поиска в SQLite.
-   **sqlite-vec:** Расширение SQLite для векторного поиска.
-   **RRF (Reciprocal Rank Fusion):** Алгоритм комбинирования ранжированных списков.
-   **Канонизация:** Процесс извлечения фактов из истории диалогов.
-   **Эмбеддинг:** Векторное представление текста.
-   **Сессия:** Один непрерывный сеанс работы с проектом.
