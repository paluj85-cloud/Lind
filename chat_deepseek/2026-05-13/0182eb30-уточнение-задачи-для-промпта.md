Идея: «Вечный Архив» как система долговременной памяти для Cline CLI

1. Истоки: проект Экс-Ис и рождение «Вечного Архива»
В рамках создания ролевой игры «Экс-Ис: Архитектор Миров» была разработана архитектура System Prompt v5.0 и системы внешней памяти «Вечный Архив». Ключевая концепция: LLM (языковая модель) выступает как гениальный рассказчик и процессор, а внешняя база данных — как нерушимая, вечная память, хранящая абсолютно все факты, события и «семена будущего». Специальный менеджер контекста перед каждым запросом к LLM извлекает из базы только релевантную информацию, не перегружая контекстное окно.

Игровая система включает:

Тайный и Явленный Свитки — структурированные факты с доказательной базой.

Семена будущего (seeds) — микро-детали, способные повлиять на мир спустя долгое время.

Нити судьбы (threads) — незакрытые сюжетные линии.

Микро- и макро-канонизацию — регулярную структуризацию накопленной истории.

Агента валидации — механизм защиты от противоречий.

Эта архитектура оказалась универсальной: она решает фундаментальную проблему «слепоты» LLM к долгой истории и может быть применена далеко за пределами игр.

2. Cline CLI: что это и в чём его проблема
Cline (https://github.com/cline/cline) — это AI-ассистент для разработки, работающий как расширение VS Code. Он способен:

Самостоятельно исследовать кодовую базу,

Создавать и редактировать файлы,

Выполнять команды в терминале,

Пользоваться браузером,

Работать с MCP-серверами (Model Context Protocol).

Cline по сути является автономным агентом-кодировщиком, который решает комплексные задачи разработки. Однако его главное ограничение — контекстное окно. В длинных сессиях или при возвращении к проекту через несколько дней он теряет контекст и может:

Забывать принятые архитектурные решения,

Повторно задавать одни и те же вопросы,

Предлагать противоречащие уже сделанному изменения,

Терять «долгоиграющие» TODO и идеи.

Разработчики Cline сознательно не применяют RAG для индексации кода, полагая, что прямой взгляд на код по мере роста контекстных окон эффективнее. Однако это не отменяет потребности во внешней мета-памяти для решений, причин, намерений и планов — того, что не хранится в файлах, но составляет «коллективный разум» проекта.

3. Как архитектура «Вечного Архива» решает проблемы Cline
Использование системы, аналогичной игровому «Вечному Архиву», может дать Cline следующие возможности:

3.1. Долговременная память между сессиями
Все принятые решения, найденные баги, одобренные паттерны, обсуждения и планы сохраняются в структурированном виде во внешней БД. При старте новой сессии Cline получает сжатое, но содержательное резюме прошлой работы.

3.2. Умная загрузка релевантного контекста
Перед началом задачи менеджер контекста извлекает только то, что относится к текущему запросу. Например:

Если задача связана с аутентификацией, подгружаются факты об используемом провайдере, причины выбора JWT, известные уязвимости.

Если задача касается рефакторинга UI, поднимаются соглашения по компонентам и стилям.

Это предотвращает раздувание промпта и улучшает качество ответа LLM.

3.3. «Семена будущего» как менеджер технического долга
Идеи, которые не были реализованы, найденные, но не исправленные баги, запланированные оптимизации хранятся как «seeds». При каждой новой сессии Cline видит их и может предложить закрыть «хвосты».

3.4. Защита от противоречий (агент валидации)
Новые факты, предлагаемые LLM, проверяются на совместимость с уже утверждёнными решениями. Это предотвращает ситуацию, когда ассистент начинает рекомендовать смену библиотеки или архитектуры, противоречащую ранее сделанному и документированному выбору.

3.5. Непрерывность при работе в команде
Если над проектом работает несколько разработчиков (или один, но с разных устройств), «Вечный Архив» способен хранить единый источник истины, с которым может взаимодействовать каждый экземпляр Cline.

4. Техническое предложение: MCP-сервер «Eternal Archive»
Наиболее естественная интеграция с Cline — через протокол MCP (Model Context Protocol), который уже поддерживается Cline и позволяет расширять его функционал подключаемыми серверами.

4.1. Компоненты MCP-сервера
Компонент	Описание
База данных	PostgreSQL (или SQLite для простоты) с pgvector для векторных индексов. Хранит факты, семена, нити, сессии и логи.
Контекстный менеджер	Сервис, который по запросу Cline (до вызова LLM) формирует сжатый, релевантный контекст из БД.
Инструменты канонизации	Cline или внешний планировщик могут запускать микро-канонизацию каждые N сообщений и полную канонизацию при завершении сессии.
Агент валидации	Проверяет новые факты на непротиворечивость перед записью.
REST API / MCP Tools	Набор инструментов, доступных Cline: get_relevant_facts, add_fact, add_seed, start_session, summarize_session и т.д.
4.2. Адаптированная модель данных
Терминология «Экс-Ис» заменяется на понятную разработчикам:

Оригинал (Экс-Ис)	Аналог для Cline	Описание
Явленный Свиток (Manifest Truth)	Проектные факты (Project Facts)	Подтверждённые решения, архитектурные правила, документированные соглашения.
Тайный Свиток (Hidden Truth)	Контекст сессии (Session Context)	Промежуточные выводы, ещё не утверждённые гипотезы, текущие намерения.
Семена (Seeds)	Технический долг и идеи (Debt & Ideas)	Найденные проблемы, предложения по улучшению, запланированные задачи.
Нити судьбы (Threads)	Активные ветки разработки (Active Threads)	Незавершённые фичи, открытые вопросы, незакрытые ревью.
Канонизация (Canonization)	Суммаризация сессии (Session Summarization)	Сжатие истории сессии до ключевых фактов, решений и планов.
4.3. Схема работы
Начало сессии: Cline через MCP вызывает start_session, получает summary предыдущих сессий, активные threads и релевантные seeds.

Рабочий цикл: В ходе диалога Cline может запрашивать get_relevant_facts(query) для уточнения контекста.

Фиксация решений: При принятии важного решения Cline (или разработчик через команду) вызывает add_fact(fact), add_seed(seed).

Микро-канонизация: Каждые ~10 сообщений асинхронно запускается «Archivist» (отдельный LLM-вызов), который извлекает из последнего лога ключевые факты и обновляет БД.

Завершение сессии: summarize_session создаёт полное резюме, которое будет подано при следующем старте.

6. Ожидаемые выгоды
Cline перестаёт «забывать» о принятых ранее решениях и планах.

Повышается качество последовательной работы над долгоживущими проектами.

Уменьшается время на повторное введение ассистента в контекст.

Снижается риск противоречивых или разрушительных изменений.

Открывается путь к «коллективной памяти» команды разработчиков, использующих Cline.

7. Потенциальные риски и контрольные точки
Конфликт с философией Cline: Cline предпочитает прямое чтение кода, поэтому «Вечный Архив» следует позиционировать только как дополнение для мета-информации, а не как замену чтения файлов.

Качество семантического поиска: может потребоваться тонкая настройка, чтобы избежать потери релевантных фактов.

Накладные расходы: дополнительные вызовы LLM для канонизации увеличат потребление токенов. Необходимо оценить соотношение цена/качество.

Сложность внедрения: MCP-сервер и БД добавляют ещё один компонент в инфраструктуру разработчика; важно обеспечить лёгкую установку и конфигурацию.

8. Заключение
Идея перенести архитектуру «Вечного Архива» из игровой вселенной Экс-Ис на рабочий инструмент Cline CLI выглядит чрезвычайно многообещающей. Она решает фундаментальную проблему памяти AI-ассистентов, сохраняя при этом их сильные стороны. Реализация в виде MCP-сервера позволяет интегрироваться с Cline естественным образом, не ломая его основной цикл работы с кодом.

Такой «внешний мозг» способен превратить Cline из полезного, но «забывчивого» помощника в полноценного члена команды, который помнит всё. Это шаг к тому, чтобы AI-инструменты действительно заслужили звание «коллеги», а не просто «одноразового скрипта».

- при ответе использует эту карту, чтобы давать точные советы, не требуя от тебя объяснять структуру проекта.

Ты можешь спросить: «Как у нас реализована аутентификация?» — и он ответит, опираясь на реальный код, а не только на память разговоров.

2. Как он работает с файлами и БД
2.1. Индексация при старте и обновлениях
При первом запуске Архивариус сканирует указанную корневую директорию проекта и подключается к БД.
Для каждого файла и таблицы БД он:
- читает содержимое,
- генерирует краткое описание (summary) с помощью LLM (или извлекает docstrings/комментарии),
- создаёт векторное представление (embedding) этого описания,
- сохраняет в индекс.

При изменении файлов (можно отслеживать через inotify/git hooks) или структуры БД происходит переиндексация только изменённых объектов.

2.2. Что хранится в индексе (метаданные)
Для файлов:
- путь, имя, расширение,
- краткое описание (что делает модуль/скрипт),
- список ключевых функций/классов/переменных (сигнатуры и короткие пояснения),
- связи: какие файлы импортирует, какие используют его,
- эмбеддинг описания (для поиска).

Для таблиц БД:
- название, список колонок с типами,
- описание назначения таблицы,
- связи с другими таблицами (внешние ключи),
- пример запроса (если есть),
- эмбеддинг описания.

Для ключевых сущностей (классов, функций, SQL-запросов) могут создаваться отдельные записи с ссылками на родительский файл/таблицу.

2.3. Использование в диалоге
Контекстный менеджер при поиске релевантной информации учитывает не только факты из диалогов, но и записи индекса. При запросе пользователя:
- Вычисляется эмбеддинг запроса.
- Ищутся топ-K релевантных записей из file_index и db_index.
- В промпт добавляется краткая выжимка (названия, пути, ключевые описания).
- Если нужно углубиться, Архивариус может запросить полное содержимое файла или результат SQL-запроса (через специальные инструменты).

4. Архитектура системы (расширенная)
4.1. Компоненты
- Чат-интерфейс (Frontend): веб-страница, похожая на chat.deepseek.com, с сайдбаром для переключения тредов.
- API-сервер (Backend): FastAPI, обрабатывает сообщения, управляет тредами, вызывает LLM.
- Менеджер памяти (Core):
  - Контекстный менеджер (тот же, что и раньше),
  - Индексатор файлов,
  - Индексатор БД,
  - Экстрактор фактов.
- Индексы:
  - PostgreSQL с pgvector для хранения фактов, записей о файлах и таблицах БД.
- Инструменты для LLM:
  - read_file(path) – возвращает содержимое файла,
  - run_query(sql) – выполняет SQL-запрос (только SELECT) к подключенной БД,
  - list_directory(path) – список файлов в директории,
  - search_code(term) – поиск по индексу.

4.2. Процесс индексации
При старте системы или по команде пользователя запускается полное сканирование.
Для каждого файла и таблицы генерируется summary (с помощью LLM или регулярных выражений для кода/схемы).
Summary сохраняется с путём, связями и эмбеддингом.
При изменении файлов (слушатель файловой системы или git diff) обновляются только изменённые записи.
Аналогично для БД можно периодически опрашивать information_schema.

4.3. Поток обработки сообщения (обновлённый)
Пользователь отправляет сообщение в текущем треде.
Контекстный менеджер:
- получает из БД последние N сообщений треда (локальный контекст),
- вычисляет эмбеддинг запроса,
- ищет релевантные факты из facts,
- ищет релевантные записи из file_index и db_index,
- формирует блоки knowledge_base и project_context.

Формируется промпт с этими данными, отправляется LLM.
LLM может вызвать инструменты (read_file, run_query) — тогда сервер выполняет их и результат добавляется в контекст для повторного вызова LLM.
Итоговый ответ показывается пользователю.
После ответа запускается микро-канонизация: из последних сообщений извлекаются новые факты, они сохраняются в общую память.

5. Безопасность и ограничения
- Все файлы и БД находятся на одном сервере, доступ контролируется правами ОС.
- Инструмент run_query разрешает только SELECT (или настраиваемый список безопасных команд).
- read_file может быть ограничен корневой директорией проекта.
- Пользователь может явно исключить конфиденциальные файлы из индексации (через .archivarius_ignore).

Таким образом, Архивариус становится мощным ассистентом, который понимает проект на глубоком уровне и может вести параллельные обсуждения, не теряя нить и не требуя повторных объяснений. Это идеальный инструмент для разработки сложных систем вроде «Экс-Ис».

- Язык: Python 3.11+ (строго)
- Бэкенд-фреймворк: FastAPI + Uvicorn (асинхронный REST API)
