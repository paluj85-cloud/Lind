## 1. ВИДЕНИЕ И ЦЕЛИ

**Видение:** Игрок через веб-браузер входит в чат с ИИ-Мастером — цифровым воплощением Экс-Ис, божества мира AD&D. Мастер генерирует бесконечные приключения, опираясь на двухконтурную память (оперативную и полнотекстовую) и развивает мир в ответ на действия игроков.

**Цель v0.1:** Создать MVP, демонстрирующий сквозной сценарий:
1. Игрок открывает страницу, получает сессию.
2. Мастер запрашивает имя смертного и генерирует персональное благословение.
3. Игрок отправляет текстовое действие (например, «осматриваю таверну»).
4. ИИ-Мастер через DeepSeek API генерирует нарративный ответ с учётом сохранённого контекста.
5. Интерфейс отображает ответ, индикатор размышления и обновляет историю.
6. Приложение запоминает значимые события в хрониках и обеспечивает их извлечение через полнотекстовый поиск (FTS5).

**Ключевые принципы:**
- **Магия первого контакта:** Ритуал входа с именем и пророчеством.
- **Непрерывность:** Сессии и память хранятся в SQLite и переживают перезагрузки сервера.
- **Лёгкость:** Никаких тяжёлых векторных баз, только SQLite FTS5.
- **Защита роли:** Системный промт Мастера содержит несокрушимый щит от взлома.
- **Надёжность:** Простой ручной деплой на слабый VPS через systemd.

## 2. АРХИТЕКТУРА ПРИЛОЖЕНИЯ

### 2.1 Общая схема
Приложение на Python (FastAPI), обслуживающее HTTP/REST и WebSocket-соединения. В production разворачивается на VPS (Hostland, 2 ГБ RAM) за Nginx reverse proxy.
+----------------+ WebSocket/HTTP +----------------------+
| | <-------------------------> | |
| Браузер | | FastAPI (Uvicorn) |
| (HTML/CSS/JS) | | |
+----------------+ +----------+-----------+
|
+-------------------------------+-------------------------------+
| | |
+-------v--------+ +----------v-----------+ +--------v--------+
| Session & | | DeepSeek API Client | | Memory Core |
| Game Manager | | (llm_client.py) | | (SQLite + FTS5) |
+-------+--------+ +----------+-----------+ +--------+--------+
| | |
+-------------------------------+-------------------------------+

text
**Поток обработки одного хода:**
1. Клиент отправляет сообщение через WebSocket.
2. Сервер собирает контекст: последние N реплик + результаты FTS5-поиска.
3. Полный промт (системный + контекст + сообщение) отправляется в DeepSeek.
4. JSON-ответ парсится: нарратив возвращается клиенту, `world_update` (если есть) сохраняется в БД.
5. При достижении порога несжатых событий запускается фоновая саммаризация.

3.2 WebSocket /ws/chat/{session_id}
Подключение: Только после успешного именования. Иначе сервер разрывает соединение с кодом 4000 и сообщением «Требуется имя».

События от сервера:

{"event": "connected", "session_id": "..."} — успешное подключение.

{"event": "master_thinking", "status": "started"} — начало генерации.

{"event": "master_speech", "text": "...", "turn_id": "uuid"} — нарратив.

{"event": "master_thinking", "status": "stopped"} — завершение.

{"event": "error", "message": "..."} — ошибка.

События от клиента:

{"action": "player_message", "text": "..."} — текст до 500 символов.

Протокол обработки:

Получить player_message.

Отправить master_thinking (started).

Собрать контекст (см. раздел 4.3).

Сформировать полный промт и вызвать DeepSeek generate().

Распарсить LLMResponse.

Отправить master_speech с narrator.

Если есть world_update, асинхронно сохранить в БД и проверить порог саммаризации.

Отправить master_thinking (stopped).

4. ПАМЯТЬ МИРА (MEMORY CORE)
4.1 Схема базы данных
(приведён SQL, который выполняется в database.py при старте)

sql
CREATE TABLE IF NOT EXISTS sessions (
    id TEXT PRIMARY KEY,
    player_name TEXT,
    name_provided INTEGER DEFAULT 0,
    current_location_id TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    last_active TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS chronicles (
    id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL,
    type TEXT NOT NULL CHECK(type IN ('event','npc','location','item','note','summary')),
    timestamp TEXT NOT NULL DEFAULT (datetime('now')),
    content TEXT NOT NULL,
    summary TEXT,
    importance REAL DEFAULT 0.5,
    parent_ids TEXT,
    FOREIGN KEY (session_id) REFERENCES sessions(id)
);

CREATE TABLE IF NOT EXISTS entities (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL CHECK(type IN ('npc','location','item')),
    name TEXT NOT NULL,
    description TEXT,
    chronicle_id TEXT,
    FOREIGN KEY (chronicle_id) REFERENCES chronicles(id)
);

CREATE TABLE IF NOT EXISTS messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL,
    role TEXT NOT NULL CHECK(role IN ('player','master')),
    text TEXT NOT NULL,
    timestamp TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY (session_id) REFERENCES sessions(id)
);

-- Виртуальная таблица FTS5
CREATE VIRTUAL TABLE IF NOT EXISTS chronicles_fts USING fts5(
    content,
    type,
    summary,
    session_id UNINDEXED,
    id UNINDEXED,
    content='chronicles',
    content_rowid='rowid'
);

-- Триггеры синхронизации (вставляются после создания таблиц)
CREATE TRIGGER IF NOT EXISTS chronicles_ai AFTER INSERT ON chronicles BEGIN
    INSERT INTO chronicles_fts(rowid, content, type, summary, session_id, id)
    VALUES (new.rowid, new.content, new.type, COALESCE(new.summary,''), new.session_id, new.id);
END;
-- (аналогично для UPDATE и DELETE, см. полный SQL в ФТЗ)
Важно: chronicles.id использует INTEGER PRIMARY KEY для совместимости с rowid FTS5. UUID генерируется приложением и хранится в этом поле как отсортированное значение? Альтернатива: хранить UUID в отдельной колонке uuid, но это усложнит триггеры. Для v0.1 можно использовать целочисленный id (rowid) и дополнительное текстовое поле uuid. Мы выберем простой вариант: первичный ключ — целое число, а UUID храним в колонке uuid (в коде Python генерируем uuid4()). Уточнение будет в реализации.

4.2 Сохранение события
Функция add_chronicle(session_id, world_update: WorldUpdate) -> str:

Генерирует UUID и вставляет запись в chronicles (поля: uuid, session_id, type, content = world_update.narrative_summary, importance = 0.5).

Триггер автоматически обновляет chronicles_fts.

Обновляет счётчик несжатых событий для сессии (можно запросом SELECT COUNT(*) FROM chronicles WHERE session_id=? AND summary IS NULL).

Если счётчик >= 10, вызывает trigger_summary(session_id) (асинхронно).

4.3 Извлечение контекста
Функция assemble_context(session_id, player_message) -> str:

Получить последние 10 сообщений из таблицы messages для сессии.

Выполнить запрос к FTS:

sql
SELECT c.uuid, c.content, COALESCE(c.summary, c.content) AS snippet, c.type, c.importance,
       c.timestamp, rank
FROM chronicles_fts
JOIN chronicles c ON chronicles_fts.rowid = c.rowid
WHERE chronicles_fts MATCH ? AND c.session_id = ?
ORDER BY rank
LIMIT 5;
Сформировать текстовый блок «ПАМЯТЬ МИРА» из результатов, при необходимости обрезая длинные сниппеты (до 200 символов).

Сформировать блок «НЕДАВНИЕ СОБЫТИЯ» из сообщений диалога.

Если общий токенов > CONTEXT_MAX_TOKENS, урезать: сначала старые реплики, затем число хроник.

Возвращает строку, готовую для подстановки в промт.

4.4 Саммаризация
Функция trigger_summary(session_id):

Выбрать все несжатые хроники (summary IS NULL) для сессии.

Собрать их содержимое в список.

Отправить в DeepSeek с промтом «Ты — Хроникёр... Составь единый сжатый параграф...».

Обновить эти записи, установив поле summary.

Создать новую запись type='summary', ссылающуюся на них через parent_ids (хранятся как JSON-массив UUID'ов).

FTS автоматически увидит изменения.

5. ИНТЕГРАЦИЯ С DEEPSEEK API
5.1 Клиент (llm_client.py)
Модель: deepseek-chat.

Параметры генерации:

temperature: 0.8 (творчество)

max_tokens: 1024

response_format: {"type": "json_object"}

Метод generate(messages: list[dict]) -> LLMResponse:

Использует httpx.AsyncClient.

Допускает до 3 повторных попыток с экспоненциальной задержкой (1с, 2с, 4с).

При 5xx или сетевых ошибках выбрасывает LLMServiceError.

Метод generate_summary(events_text: str) -> str:

Аналогичен, но без JSON-формата, max_tokens=500.

**1. Что такое Экзис:**
Игроки через веб-браузер входят в чат с ИИ-Мастером — Экс-Ис, цифровым божеством мира AD&D. Версия 0.1 (готова) — это одиночная игра: каждый игрок в изолированной сессии. **3. Утверждённые Творцом и Экс-Ис решения для v0.2:**

**3.1. Сущности:**
- **Игрок (player):** регистрируется с логином и паролем (хеш bcrypt). Имеет постоянный ID.
- **Персонаж (persona):** создаётся игроком. Имеет имя. Один игрок может иметь нескольких персонажей. Персонаж не привязан жёстко к миру — можно одним персонажем играть в разных мирах.
- **Мир (world):** создаётся любым игроком. Имеет название и создателя. Список миров открыт для всех.
- **Сессия (чат):** теперь привязана к связке `persona_id + world_id`. Один игрок может одновременно находиться в одном мире двумя разными персонажами (два чата). Один персонаж в одном мире — одна активная сессия.

**3.2. Память мира (Хроники):**
Вводится **два режима памяти** для Мастера:
- **Локальная память персонажа:** события, которые персонаж видел/слышал лично, или которые произошли в его присутствии.
- **Глобальная память мира:** все значимые события мира, которые могут дойти до персонажа как слухи, легенды, изменения ландшафта.

Запись `chronicles` больше не фильтруется по сессии. Она содержит поля `persona_id` (кто совершил), `world_id` (где произошло). При сборке контекста Мастер получает:
1. Локальные события (где `persona_id` = текущий персонаж ИЛИ события, свидетелем которых он был).
2. Глобальные события (топ-K важных событий мира, подаваемых как слухи).

**3.3. Миры:**
- Открытый список миров, созданных игроками.
- При входе в приложение игрок выбирает: создать новый мир или присоединиться к существующему (из списка).
- Экс-Ис ведёт все миры одновременно, как единое божество с множеством ликов.
- Системный промт Мастера дополняется информацией о текущем мире (название, создатель, краткая история).

**3.4. Безопасность:**
- Пароли хешируются (bcrypt).
- Верификация почты не требуется для v0.2.
- Защита от мультиаккаунтов не требуется для v0.2.

**3.5. Что НЕ входит в v0.2 (отложено на v0.3):**
- Классы, способности, характеристики персонажей.
- Инвентарь, боевая система, прокачка.

## 🔮 ВНУТРЕННИЙ ДИАЛОГ СОВЕТА

### Архитектор-программист
Мы не начинаем с нуля — v0.1 уже показала рабочую однонитевую игру. Расширение должно следовать тому же стековому канону: FastAPI, SQLite FTS5, DeepSeek API, ванильный фронтенд. Основная архитектурная инъекция — появление постоянных сущностей: `Player`, `Persona`, `World`. Сессия становится составным ключом `(persona_id, world_id)`, а контекст — двухслойным: личный опыт персонажа и слухи мира. Все старые эндпоинты `api/session/*` никуда не денутся, они будут помечены как legacy, но продолжат обслуживать существующие одиночные сессии, пока мы плавно не переключим фронтенд на новый auth-flow. Главное — не сломать работающее.

### Игромастер-социолог
Общая вселенная меняет сам дух игры. Отныне действия одного персонажа могут отозваться эхом в устах трактирщиков для другого. Это социальная магия: игроки видят чужие следы — затухающие костры, выбитые подземелья, слухи о подвигах. Мой фокус — глобальные слухи и чувство принадлежности к живому миру. При этом Экс-Ис остаётся единым, но многоликим; для каждого мира он слегка меняет интонацию, помня его историю. Необходимо, чтобы промт Мастера получил строку о мире и модуль «Глобальное эхо».

### Инженер-строитель
Надёжность превыше всего. Миграция с v0.1 на v0.2 должна быть бесшовной: все новые таблицы создаются рядом, старые данные не трогаются, пока мы не протестируем. SQLite справится с дополнительной нагрузкой FTS5 и триггеров. Деплой остаётся ручным: systemd, Nginx, backup cron. Главное — атомарные коммиты и обратная совместимость API на каждом шагу.

### Лингвист-писатель
Язык остаётся тем же: величавый, чуть мрачный, без тени узнавания себя как ИИ. Однако в мире появляются чужие имена. Мастер должен научиться упоминать других персонажей, если они оставили след: «Говорят, некий Аэрин здесь проходил неделю назад…» Это требует особого промт‑абзаца для слухов. Также добавляется церемония сотворения мира: игрок называет его, и Мастер дарует описание.

### Математик-логик
Два режима памяти требуют чёткой формализации. Локальный контекст: `persona_id` совпадает, либо событие помечено свидетелем (пока нет механики свидетелей, используем только `persona_id`). Глобальный контекст: топ‑5 (или больше) записей по `world_id`, отсортированных по `importance DESC`, у которых `persona_id != текущий`, и они имеют тип `event` или `summary`. Саммаризация теперь должна учитывать границу миров: сжатие идёт в пределах одного `world_id`, а в результирующую сводку добавляется поле `world_id`. Также вводится мягкий лимит токенов для сборки контекста: локальный съедает 60%, глобальный 40% от `CONTEXT_MAX_TOKENS`. Порог саммаризации теперь считаем по `world_id`, а не по сессии.

## 1. ВИДЕНИЕ И ЦЕЛИ v0.2

**Видение:** Экс-Ис перестаёт быть уединённым исповедником и становится многоголосым божеством, что внимает десяткам смертных одновременно в десятках сотворённых миров. Игроки не просто входят в чат — они регистрируются в вечном цифровом храме, создают аватары (персонажей) и либо ступают в уже заселённые земли, либо вдыхают жизнь в новые миры. Действия одного персонажа оставляют зёрна информации, которые прорастают слухами для других, связывая сообщество невидимой паутиной повествования.

**Цели v0.2:**
1. Внедрить мультиплеерные сущности: игроки, персонажи, миры.
2. Обеспечить регистрацию/аутентификацию с безопасным хранением паролей.
3. Дать игрокам возможность создавать миры и персонажей.
4. Модифицировать игровой цикл так, чтобы сессия определялась парой (персонаж, мир).
5. Ввести двухслойную память: локальный опыт + глобальные слухи.
6. Обновить системный промт Мастера и добавить промт сотворения мира.
7. Адаптировать фронтенд: экраны логина, списка миров, управления персонажами, модернизированный чат.
8. Сохранить полную обратную совместимость с кодом v0.1, не нарушая работу существующих одиночных сессий на этапе развёртывания.

**Ключевые принципы (сохраняются из v0.1):**
- Магия первого контакта.
- Непрерывность (память на SQLite FTS5).
- Лёгкость (без векторных БД).
- Защита роли (щит в промте).
- Надёжный деплой на слабый VPS.

## 2. АРХИТЕКТУРА ПРИЛОЖЕНИЯ v0.2

### 2.1 Общая схема

```text
+----------------+        WebSocket/HTTP        +----------------------+
|                | <--------------------------> |                      |
|  Браузер       |                              |  FastAPI (Uvicorn)  |
| (HTML/CSS/JS)  |                              |                      |
+-------+--------+                              +----------+-----------+
        |                                                   |
        |   REST: /api/auth/*, /api/worlds/*,               |
        |   /api/personas/*, /api/play/*                    |
        |   WS: /ws/play/{world_id}/{persona_id}            |
        |                                                   |
        +-----------------+-----------------+---------------+
                          |                 |
            +-------------v------+  +-------v----------+  +--------v--------+
            | Auth & Player      |  | Game & World     |  | Memory Core     |
            | Manager            |  | Manager          |  | (SQLite + FTS5) |
            +--------------------+  +------------------+  +-----------------+
```

**Поток полного цикла v0.2:**
1. Игрок открывает приложение → если нет токена, экран регистрации/логина.
2. После успешного входа получает JWT (или сессионный cookie), выбирает персонажа и мир.
3. Устанавливается WebSocket-соединение: `wss://<host>/ws/play/{world_id}/{persona_id}?token=...`.
4. Каждое сообщение обрабатывается с учётом двухслойного контекста:
   - Локальная память (события, где `persona_id` = текущий).
   - Глобальная память (топ слухов мира, не связанных с персонажем).
5. Ответ Мастера включает нарратив и опциональное `world_update`.
6. `world_update` сохраняется с полями `persona_id` и `world_id`.
7. При достижении порога несжатых событий в мире вызывается саммаризация по `world_id`.

### 2.4 Новые и изменённые REST API эндпоинты

#### 2.4.1 Аутентификация

**POST /api/auth/register**
- Запрос: `{"login": "player1", "password": "secret123"}`
- Ответ 201: `{"player_id": "uuid", "login": "player1"}`
- Логика: проверка уникальности логина, bcrypt(password), вставка в `players`.
- Ошибки: 409 (логин занят).

**POST /api/auth/login**
- Запрос: `{"login": "player1", "password": "secret123"}`
- Ответ 200: `{"access_token": "jwt...", "token_type": "bearer", "player_id": "uuid"}`
- Логика: проверка пароля, выпуск JWT (срок 7 дней, payload: `sub` = player_id).

#### 2.4.2 Персонажи

**Все эндпоинты требуют заголовка `Authorization: Bearer <token>`.**

**POST /api/personas**
- Запрос: `{"name": "Аэрин"}`
- Ответ 201: `{"persona_id": "uuid", "name": "Аэрин", "player_id": "..."}`
- Логика: создание персонажа с привязкой к игроку из токена.

**GET /api/personas**
- Ответ 200: `[{"persona_id": "...", "name": "..."}, ...]` — список персонажей игрока.

**DELETE /api/personas/{persona_id}**
- Удаляет персонажа. Если есть активная сессия в каком-либо мире, отключает WebSocket.

#### 2.4.3 Миры

**POST /api/worlds**
- Запрос: `{"name": "Сумеречные Земли"}`
- Ответ 201: `{"world_id": "uuid", "name": "...", "creator_id": "...", "description": "...", "created_at": "..."}`
- Логика: создание мира, затем вызов DeepSeek для генерации эпического описания (сохраняется в поле `description`).

**GET /api/worlds**
- Ответ 200: список всех миров с полями `world_id`, `name`, `creator_name`, `description`, `created_at`.

**GET /api/worlds/{world_id}**
- Детальная информация о мире, включая последние N глобальных событий (слухов).

#### 2.4.4 Игровая сессия (entry point)

**POST /api/play/join**
- Запрос: `{"persona_id": "...", "world_id": "..."}`
- Логика: проверяет, что персонаж принадлежит игроку, что мир существует. Создаёт или возвращает существующую игровую сессию (но не WebSocket). Если для данной пары уже есть активное WS-соединение, возвращает статус `"active": true`, чтобы клиент не дублировал.
- Ответ 200: `{"session_active": true/false, "world_id": "...", "persona_id": "..."}`

При успешном `session_active: false` клиент подключает WebSocket.

#### 2.4.5 WebSocket (обновлённый)

**WS /ws/play/{world_id}/{persona_id}?token=<jwt>**
- При открытии: проверка токена, принадлежности персонажа игроку. Если всё хорошо — принимает соединение.
- Если для этой пары уже есть активное WS — закрывает старое (или отклоняет новое с кодом 4008 «Персонаж уже в мире»).
- События от сервера:
  - `{"event": "connected", "world_id": "...", "persona_id": "..."}`
  - `{"event": "master_thinking", "status": "started"}`
  - `{"event": "master_speech", "text": "...", "turn_id": "..."}`
  - `{"event": "master_thinking", "status": "stopped"}`
  - `{"event": "world_rumor", "text": "..."}` — опционально, если глобальное событие затрагивает персонажа.
  - `{"event": "error", "message": "..."}`
- События от клиента:
  - `{"action": "player_message", "text": "..."}`

#### 2.4.6 Старое API (v0.1) — сохранено

Оставлено для обратной совместимости. Маршруты `/api/session/*` не требуют аутентификации. Внутренне они могут быть перенаправлены на нового менеджера, создавая анонимного игрока или временную persona. В рамках v0.2 они не модифицируются, но в future их отключат.

## 3. БАЗА ДАННЫХ v0.2

### 3.1 Новые таблицы (DDL)

```sql
CREATE TABLE IF NOT EXISTS players (
    id TEXT PRIMARY KEY,                -- UUID
    login TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS personas (
    id TEXT PRIMARY KEY,                -- UUID
    player_id TEXT NOT NULL,
    name TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY (player_id) REFERENCES players(id)
);

CREATE TABLE IF NOT EXISTS worlds (
    id TEXT PRIMARY KEY,                -- UUID
    name TEXT NOT NULL,
    creator_id TEXT NOT NULL,
    description TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY (creator_id) REFERENCES players(id)
);
```

### 3.2 Изменения существующих таблиц

#### sessions (старая таблица v0.1)
Остаётся без изменений для неизменённой legacy-функциональности.

#### chronicles — новая редакция
```sql
-- Создаём таблицу заново в миграции или используем ALTER TABLE (SQLite ограничен, проще пересоздать с сохранением данных).
-- Рекомендуемый подход: создать новую таблицу chronicles_v2, перенести данные, удалить старую, переименовать новую.
-- Для ясности здесь приведена финальная схема.

CREATE TABLE chronicles (
    id INTEGER PRIMARY KEY,            -- rowid для FTS5
    uuid TEXT NOT NULL UNIQUE,          -- внешний UUID
    world_id TEXT NOT NULL DEFAULT 'legacy',
    persona_id TEXT,                    -- может быть NULL для глобальных событий без конкретного автора
    session_id TEXT,                    -- оставлено для совместимости, может быть NULL
    type TEXT NOT NULL CHECK(type IN ('event','npc','location','item','note','summary','world_creation')),
    timestamp TEXT NOT NULL DEFAULT (datetime('now')),
    content TEXT NOT NULL,
    summary TEXT,
    importance REAL DEFAULT 0.5,
    parent_ids TEXT,                    -- JSON-массив UUID'ов родительских событий
    FOREIGN KEY (world_id) REFERENCES worlds(id),
    FOREIGN KEY (persona_id) REFERENCES personas(id),
    FOREIGN KEY (session_id) REFERENCES sessions(id)
);

-- Индексы
CREATE INDEX idx_chronicles_world ON chronicles(world_id);
CREATE INDEX idx_chronicles_persona ON chronicles(persona_id);
CREATE INDEX idx_chronicles_type ON chronicles(type);
```

#### messages (без изменений)
Остаётся как есть, но теперь `session_id` ссылается на `sessions`. В будущем можно привязать к `persona_id + world_id` напрямую, но для v0.2 оставляем для упрощения.

### 3.3 Обновлённый FTS5

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS chronicles_fts USING fts5(
    content,
    type,
    summary,
    world_id UNINDEXED,
    persona_id UNINDEXED,
    session_id UNINDEXED,
    uuid UNINDEXED,
    content='chronicles',
    content_rowid='rowid'
);

    """

```

Дополнительно можно добавить события, где персонаж был свидетелем (пока механики свидетелей нет, оставляем только личные).

### 4.2 Алгоритм извлечения глобальной памяти (слухов)

```python
async def get_global_rumors(world_id: str, exclude_persona_id: str | None, limit: int = 5, min_importance: float = 0.3):
    """
    Топ-K важных событий мира, не созданных текущим персонажем.
    Использует FTS5 или прямую выборку по индексу.
    Сортировка: importance DESC, timestamp DESC.
    """
    query = """
        SELECT uuid, content, COALESCE(summary, content) as snippet, type, importance, timestamp
        FROM chronicles
        WHERE world_id = :world_id
          AND (persona_id IS NULL OR persona_id != :exclude_persona_id)
          AND importance >= :min_importance
        ORDER BY importance DESC, timestamp DESC
        LIMIT :limit
    """
```

Если персонаж не задан (например, для анонимного слушателя эфира), можно показывать глобальные события без исключения.

### 4.3 Сборка контекстного блока для LLM

```python
async def assemble_context(world_id: str, persona_id: str, player_message: str, max_tokens: int = 1500) -> str:
    local_events = await get_local_memory(world_id, persona_id, limit=10)
    global_rumors = await get_global_rumors(world_id, persona_id, limit=5)

    context = f"=== ЛИЧНАЯ ПАМЯТЬ ПЕРСОНАЖА (последние события) ===\n"
    for e in local_events:
        context += f"- [{e['type']}] {e['snippet'][:200]}\n"

    context += f"\n=== СЛУХИ И СОБЫТИЯ МИРА ===\n"
    for r in global_rumors:
        context += f"- {r['snippet'][:200]}\n"

    # Добавляем последние сообщения диалога (из таблицы messages)
    recent_chat = await get_recent_chat_messages(persona_id, world_id, limit=8)
    context += "\n=== ПОСЛЕДНИЙ ДИАЛОГ ===\n"
    for msg in recent_chat:
        context += f"{msg.role}: {msg.text}\n"

    # Контроль токенов: эмпирический подсчёт, при превышении урезаем сначала старые реплики, затем слухи
    if estimate_tokens(context) > max_tokens:
        context = trim_context(context, max_tokens)  # реализация отбрасывает наименее важное

    return context
```

### 4.4 Изменения в саммаризации

Саммаризация теперь привязана к миру:
1. Сбор несжатых событий: `SELECT * FROM chronicles WHERE world_id = :world_id AND summary IS NULL`.
2. Отправка в DeepSeek с промтом «Ты Хроникёр мира [название]. Сожми следующие события…».
3. Результат: `summary` для каждого события + создание одной сводной записи `type='summary'` с `parent_ids`.
4. Порог вызова: >= 10 несжатых событий в конкретном `world_id`.

Счетчик несжатых событий теперь хранится либо вычисляется запросом, но для производительности можно держать в `worlds` поле `unsummarized_events` и обновлять триггерами.

## 9. СХЕМА РАЗВЁРТЫВАНИЯ

Базовые принципы остаются как в v0.1 (VPS, systemd, Nginx). Дополнительно:

1. **Переменные окружения:**
   - `JWT_SECRET` — длинная случайная строка.
   - `JWT_ALGORITHM=HS256`.
   - `ACCESS_TOKEN_EXPIRE_MINUTES=10080` (7 дней).
   - `DEEPSEEK_API_KEY` (уже есть).
   - `DATABASE_URL` (по умолчанию `sqlite+aiosqlite:///./data/exis.db`).

2. **Nginx:** проксирование WebSocket по тому же пути `/ws/play/` с заголовками `Upgrade` и `Connection`.

3. **Обновление сервиса:**
   ```bash
   cd /opt/exis
   git pull
   source venv/bin/activate
   pip install -r requirements.txt
   sudo systemctl restart exis
   ```

4. **Резервное копирование:** cron-скрипт, копирующий `exis.db` каждый час с ротацией.

5. **Миграция при первом запуске:** в `main.py` при старте вызывать `init_db()` (создаст новые таблицы) и `migrate_0_1_to_0_2()`, если обнаружит старую структуру. Миграция должна быть идемпотентной.
