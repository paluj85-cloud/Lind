Привет! Если задача — быстро отдавать конкретные главы по ссылкам и книг всего ~100, то **база данных MySQL** подходит гораздо лучше, чем текстовые файлы. Вот почему и как это можно организовать.

### Почему MySQL, а не txt

1. **Быстрый доступ по ссылке на главу**  
   В БД вы можете хранить каждую главу отдельной строкой с полями `book_id`, `chapter_number` (или `slug`/`chapter_id`), `title`, `content`. Тогда запрос типа  
   ```sql
   SELECT content FROM chapters WHERE book_id = ? AND chapter_number = ?;
   ```  
   мгновенно вернёт текст ровно той главы, которая нужна ИИ. Ничего лишнего читать не придётся.  
   С текстовыми файлами пришлось бы либо парсить один большой файл и искать границы главы, либо создавать сотни маленьких файлов — и то, и другое менее удобно и медленнее при частых обращениях.

2. **Структура и метаданные**  
   Помимо текста, в БД легко хранить метаданные: названия, даты, статус обработки, связи с авторами и т.п. Позже это пригодится, если ИИ начнёт использовать, например, фильтры по темам или языку.

3. **Целостность и одновременный доступ**  
   FastAPI с несколькими воркерами может одновременно запрашивать главы. БД корректно обрабатывает параллельные чтения без блокировок (на чтение). С файлами пришлось бы думать о возможных конфликтах при обновлениях.

4. **Готовность к более сложным сценариям**  
   Со временем может появиться потребность в полнотекстовом поиске или семантическом поиске (векторные эмбеддинги). В MySQL можно добавить `FULLTEXT` индекс или хранить рядом вектор (хотя тогда лучше специализированные решения, но структура уже будет готова). С txt-файлами это реализовать сложнее.

### Пример схемы таблиц

```sql
CREATE TABLE books (
    id INT AUTO_INCREMENT PRIMARY KEY,
    title VARCHAR(500) NOT NULL,
    author VARCHAR(255),
    language VARCHAR(50),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE chapters (
    id INT AUTO_INCREMENT PRIMARY KEY,
    book_id INT NOT NULL,
    chapter_number INT NOT NULL,          -- или slug VARCHAR(100)
    title VARCHAR(500),
    content MEDIUMTEXT NOT NULL,          -- до 16 МБ, для главы хватит
    UNIQUE KEY unique_chapter (book_id, chapter_number),
    FOREIGN KEY (book_id) REFERENCES books(id) ON DELETE CASCADE
);
```

Если ссылка на главу выглядит как `/books/42/chapters/5`, FastAPI получит параметры, сделает запрос к БД и вернёт текст именно этой главы.

### А что с файлами?

Файловый подход имеет право на жизнь, если вы храните каждую главу в отдельном `.txt`-файле с именем, например, `book_42/chapter_5.txt`. Тогда:
- Путь к файлу вычисляется по параметрам.
- Чтение файла целиком через `async read` – операция I/O, но для ~100 книг это тоже быстро.
- Однако: нет встроенной проверки ссылочной целостности (существует ли книга с таким id), сложнее делать резервные копии «глава за главой», и если главы начнут править люди/ИИ, возникает вопрос блокировок при редактировании.

Но главное преимущество БД именно в вашем кейсе – **запрос только нужной главы без лишней логики разбора** и простота связывания с другими данными.

### Оптимизация для частых обращений

Если ИИ обращается к одним и тем же главам многократно (например, многократное цитирование), можно добавить кэширование:
- В памяти приложения (простой `lru_cache` на функции получения главы) – для горячих глав.
- Или Redis – если сервис распределённый.

При всего 100 книгах такой кэш будет очень лёгким, а основное хранилище останется в MySQL.

### Итог

**Рекомендую MySQL** – это даст ясную структуру, мгновенный доступ по идентификатору главы, простоту расширения и полную совместимость с FastAPI (через SQLAlchemy/asyncpg/aiomysql/Tortoise ORM). Для текстов большого объема MEDIUMTEXT более чем достаточно. Если захотите потом быстро сменить движок (например, на PostgreSQL), перенос будет простым.

### Почему PostgreSQL может быть предпочтительнее для AI-проекта

1. **Векторный поиск (в перспективе)**  
   Если ИИ будет не просто читать главы, но и искать релевантные фрагменты по смыслу, вам понадобятся эмбеддинги. PostgreSQL + `pgvector` позволяют хранить векторы и делать быстрый ANN-поиск прямо в той же БД без отдельных сервисов типа Pinecone/Weaviate.

2. **Продвинутый полнотекстовый поиск**  
   Даже сейчас, без векторов, может понадобиться поиск по фразам, синонимам, с учётом языка. `tsvector` в PostgreSQL даёт существенно более гибкое и точное решение.

3. **Гибкость расширений**  
   Postgres легко расширяется под специфические задачи (семантический поиск, работа с PDF, UUID, работа с массивами, hstore). Вероятность, что при росте системы вам что-то понадобится, что в MySQL сделать сложнее, выше.

4. **Лучшая защита от случайных ошибок**  
   PostgreSQL строже к типам данных и синтаксису, что снижает риск незаметной порчи данных, когда логика приложения усложнится.

### Но MySQL тоже не плох

- Если вы уверены, что дальше пары сотен книг и простого чтения глав дело не пойдёт, MySQL абсолютно справится.
- MySQL проще в настройке для начинающих (хотя и Postgres сейчас не сложен).
- Если у вас уже поднят сервер с MySQL и есть опыт — переход на Postgres ради 100 книг не обязателен.

### Почему одна таблица — это единственно верный путь

| Критерий | Одна таблица `chapters` | Отдельные таблицы на каждую книгу |
|----------|-------------------------|-----------------------------------|
| **Структура БД** | Простая, предсказуемая, легко расширяется | Сотня таблиц, которые нужно помнить и обслуживать |
| **Запрос главы** | `SELECT ... WHERE book_id=? AND chapter_num=?` — один универсальный запрос | Придётся динамически подставлять имя таблицы, что опасно, неудобно и ломает ORM |
| **Изменение структуры** | `ALTER TABLE chapters ADD COLUMN language VARCHAR(10);` — один раз | Придётся обновить 100 таблиц, если добавите новое поле |
| **Целостность данных** | Внешние ключи, уникальные индексы работают естественно | Ссылочную целостность между книгами и «таблицами глав» организовать крайне сложно |
| **ORM и миграции** | SQLAlchemy/Tortoise ORM работают «из коробки» | Модели придётся плодить динамически, либо эмулировать, либо отказаться от ORM |
| **Производительность** | 100 книг × ~сотни глав = десятки тысяч записей. С индексом `(book_id, chapter_num)` доступ молниеносный | Производительность на том же уровне, но с дикой сложностью поддержки |
| **Резервное копирование** | Одна команда `pg_dump` — всё сохранилось | 100 отдельных таблиц, схему которых не запомнить |

Даже если книг станет 10 000 или 1 000 000, реляционные базы данных прекрасно справляются с одной таблицей, если настроены правильные индексы и при необходимости партицирование (но это отдельная тема и для вашего объёма не требуется).

---

### Как это должно выглядеть

Конкретно для вашего проекта идеальная схема:

```sql
CREATE TABLE books (
    id SERIAL PRIMARY KEY,
    title TEXT NOT NULL,
    author TEXT,
    language VARCHAR(50)
);

CREATE TABLE chapters (
    id SERIAL PRIMARY KEY,
    book_id INTEGER NOT NULL REFERENCES books(id) ON DELETE CASCADE,
    chapter_number INTEGER NOT NULL,   -- или slug, если нужно
    title TEXT,
    content TEXT NOT NULL,
    UNIQUE (book_id, chapter_number)
);

CREATE INDEX idx_chapters_book_chapter ON chapters(book_id, chapter_number);
```

FastAPI-роут просто принимает `book_id` и `chapter_number`, делает один запрос — и возвращает текст ИИ. Никакой магии, полная типобезопасность, лёгкая поддержка.

---

### А что насчёт «книги отдельно»?

Есть ситуации, когда данные **разных типов** лучше выносить в разные таблицы (например, отдельная таблица для изображений, другая — для аудио). Но у вас все главы — это однородные текстовые сущности, отличающиеся только `book_id`. Поэтому их место строго в одном месте.

---

### Итог

**Храните все книги в единой структуре «одна таблица книг + одна таблица глав».** Это стандарт индустрии, так работают все CMS, блоги, библиотеки. Отдельная таблица на каждую книгу только создаст хаос и не даст ни единого преимущества при ваших объёмах данных.

Если появятся вопросы по коду FastAPI + SQLAlchemy для такой схемы, спрашивайте — разберём подробнее.

## 1. Структура базы данных (PostgreSQL)

### Книги и главы

```sql
CREATE TABLE books (
    id SERIAL PRIMARY KEY,
    title TEXT NOT NULL,
    author TEXT,
    annotation TEXT,          -- краткая аннотация всей книги
    summary TEXT,             -- краткое изложение всей книги (2-3 абзаца)
    created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE chapters (
    id SERIAL PRIMARY KEY,
    book_id INTEGER NOT NULL REFERENCES books(id) ON DELETE CASCADE,
    chapter_number INTEGER NOT NULL,
    title TEXT,
    content TEXT NOT NULL,            -- полный текст главы
    summary TEXT,                     -- краткое изложение главы (создаёт ИИ)
    UNIQUE (book_id, chapter_number)
);
CREATE INDEX idx_chapters_book ON chapters(book_id, chapter_number);
```

### Персонажи

```sql
CREATE TABLE characters (
    id SERIAL PRIMARY KEY,
    book_id INTEGER NOT NULL REFERENCES books(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    aliases TEXT[],                   -- массив альтернативных имён
    description_appearance TEXT,      -- описание внешности
    description_personality TEXT,     -- характер, черты
    abilities TEXT[],                 -- умения, навыки
    background TEXT,                  -- предыстория
    summary TEXT,                     -- общее краткое описание для быстрого поиска
    UNIQUE (book_id, name)
);
CREATE INDEX idx_characters_book ON characters(book_id);
```

### Локации

```sql
CREATE TABLE locations (
    id SERIAL PRIMARY KEY,
    book_id INTEGER NOT NULL REFERENCES books(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    description TEXT,                 -- описание места
    features TEXT[],                  -- особенности (массив)
    summary TEXT,
    UNIQUE (book_id, name)
);
CREATE INDEX idx_locations_book ON locations(book_id);
```

### Арки приключений (сюжетные линии)

```sql
CREATE TABLE story_arcs (
    id SERIAL PRIMARY KEY,
    book_id INTEGER NOT NULL REFERENCES books(id) ON DELETE CASCADE,
    title TEXT NOT NULL,
    description TEXT,                 -- развёрнутое описание арки
    summary TEXT,                     -- краткое изложение арки
    sequence_order INTEGER            -- порядок арок в книге
);
CREATE INDEX idx_arcs_book ON story_arcs(book_id);
```

### Связующие таблицы (принадлежность к главам/аркам)

```sql
-- В каких главах появляется персонаж
CREATE TABLE chapter_characters (
    chapter_id INTEGER NOT NULL REFERENCES chapters(id) ON DELETE CASCADE,
    character_id INTEGER NOT NULL REFERENCES characters(id) ON DELETE CASCADE,
    role_description TEXT,            -- что делал персонаж именно в этой главе (опционально)
    PRIMARY KEY (chapter_id, character_id)
);

-- Локации по главам
CREATE TABLE chapter_locations (
    chapter_id INTEGER NOT NULL REFERENCES chapters(id) ON DELETE CASCADE,
    location_id INTEGER NOT NULL REFERENCES locations(id) ON DELETE CASCADE,
    PRIMARY KEY (chapter_id, location_id)
);

-- Главы, входящие в арку
CREATE TABLE arc_chapters (
    arc_id INTEGER NOT NULL REFERENCES story_arcs(id) ON DELETE CASCADE,
    chapter_id INTEGER NOT NULL REFERENCES chapters(id) ON DELETE CASCADE,
    chapter_order INTEGER,            -- порядок главы внутри арки
    PRIMARY KEY (arc_id, chapter_id)
);

-- Персонажи, ключевые для арки
CREATE TABLE arc_characters (
    arc_id INTEGER NOT NULL REFERENCES story_arcs(id) ON DELETE CASCADE,
    character_id INTEGER NOT NULL REFERENCES characters(id) ON DELETE CASCADE,
    PRIMARY KEY (arc_id, character_id)
);
```

Такая реляционная схема позволяет задавать любые вопросы:  
«Дай описание Гарри Поттера», «В какой главе Хогвартс появляется впервые?», «Найди все арки с участием Волан-де-Морта», «Покажи похожую ситуацию на битву в Запретном лесу».

## 2. Как ИИ будет «читать» книгу и заполнять базу

Типичный pipeline обработки одной книги:

1. **Разбить текст на главы** (вручную или автоматически по шаблонам «Глава N»).  
2. Для каждой главы отправить её полный текст LLM (одним запросом) с заданием:
   - Написать краткое изложение (`summary`) длиной 2–5 предложений.
   - Выделить персонажей (имя, краткое описание внешности/характера, умения, что делают в данной главе).
   - Выделить локации (название, что происходит).
   - Определить, к какой арке относится глава, или предложить новую арку.
3. Объединить данные: новых персонажей/локации добавить в соответствующие таблицы, заполнить связи.
4. Сгенерировать аннотацию и общее краткое изложение книги, агрегируя summary глав (ещё один вызов LLM).
5. Сохранить всё в БД.

На практике это можно автоматизировать в FastAPI-воркере при загрузке книги.

## 3. Базовые индексы и полнотекстовый поиск

### Полнотекстовый поиск по контенту и summary

Чтобы поддержать запросы типа «найди главы, где упоминается философский камень», добавим `tsvector`:

```sql
ALTER TABLE chapters ADD COLUMN content_tsv tsvector
    GENERATED ALWAYS AS (to_tsvector('russian', content)) STORED;
CREATE INDEX idx_chapters_tsv ON chapters USING GIN(content_tsv);

ALTER TABLE chapters ADD COLUMN summary_tsv tsvector
    GENERATED ALWAYS AS (to_tsvector('russian', summary)) STORED;
CREATE INDEX idx_chapters_summary_tsv ON chapters USING GIN(summary_tsv);

-- Аналогично для персонажей и локаций
ALTER TABLE characters ADD COLUMN description_tsv tsvector
    GENERATED ALWAYS AS (to_tsvector('russian', summary || ' ' || description_appearance || ' ' || description_personality)) STORED;
CREATE INDEX idx_characters_tsv ON characters USING GIN(description_tsv);
```

Быстрый поиск:
```sql
SELECT id, title, ts_rank(content_tsv, query) AS rank
FROM chapters
WHERE content_tsv @@ plainto_tsquery('russian', 'философский камень')
ORDER BY rank DESC
LIMIT 5;
```

## 4. Семантический поиск «похожих ситуаций» с pgvector

Чтобы отвечать на запросы вроде `"Я опишу ситуацию, а ты найди похожую"`, нужно сравнивать смысл описания со смыслом контента глав или арок. Для этого — эмбеддинги.

Добавим колонки для векторного представления:

```sql
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE chapters ADD COLUMN embedding vector(1536);
ALTER TABLE story_arcs ADD COLUMN embedding vector(1536);
ALTER TABLE characters ADD COLUMN embedding vector(1536);
-- размерность зависит от модели (1536 для text-embedding-ada-002, 768 для многих open-source)
```

При заполнении: после генерации `summary` главы/арки/персонажа через API эмбеддингов получаем вектор и сохраняем.

Индексация:
```sql
CREATE INDEX ON chapters USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
-- или hnsw
CREATE INDEX ON chapters USING hnsw (embedding vector_cosine_ops);
```

Пример поиска похожих арок по описанию пользователя:
```python
from pgvector.sqlalchemy import Vector
import openai

# Получить embedding запроса
query_embedding = openai.Embedding.create(input=user_query, model="text-embedding-ada-002")['data'][0]['embedding']

# Искать в БД
similar_arcs = await session.execute(
    select(story_arcs).order_by(story_arcs.embedding.cosine_distance(query_embedding)).limit(5)
)
```

Таким образом, первый этап поиска — по векторам кратких изложений. Если релевантность высокая, уже загружаем полный контент из `chapters.content`.

## 5. Сценарий ответа ИИ (FastAPI)

Пользователь спрашивает: «Расскажи о характере Гермионы».

1. Система ищет по точному совпадению имени в таблице `characters` (или с использованием синонимов).
2. Если персонаж найден, возвращает поля `description_personality`, `summary`, дополняя их выдержками из `chapter_characters`.
3. Если нужен «разговор» с контекстом, можно через LLM-запрос собрать эти данные и попросить модель сформулировать ответ.

Спрашивают: «Найди момент, где команда сражается с троллем».

1. Генерируется embedding запроса.
2. Ищутся самые близкие главы по `chapters.embedding` (на основе summary).
3. Берутся top-3 главы, их краткие содержания сравниваются более точно LLM, затем выдаётся релевантный отрывок из полного текста.

## 6. Зачем слои кратких изложений?

Вы правильно заметили: сначала ИИ читает `summary` (аннотацию, краткое изложение главы/арки). Это даёт выигрыш по скорости и стоимости:
- Вектор `summary` (например, 200 токенов) дешевле хранить и быстрее сравнивать, чем эмбеддинг всей главы (тысячи токенов).
- При полнотекстовом поиске `tsvector` по summary мгновенно отсекает нерелевантные главы.
- LLM при генерации ответа можно передавать только summary глав, а полный текст загружать лишь при явном запросе «покажи оригинал».

Рекомендуется генерировать summary при добавлении книги и хранить их, тогда поиск будет очень быстрым.

### 1. Чего не хватает в базовой схеме?

Сейчас у вас есть:
- `chapters.content` – полный текст главы.
- `chapter_characters` – факт появления персонажа в главе.

Но нет информации, **какие именно фрагменты текста соответствуют диалогам** конкретного героя. Искать упоминания «Гермиона сказала» через `LIKE` или `tsvector` ненадёжно: реплики часто оформляются без явного указания имени, да и порядок слов может быть любым.

Поэтому лучший путь – **распарсить текст главы на реплики** и сохранить их в отдельной таблице.

### 2. Новая таблица: `utterances` (реплики)

Эта таблица будет содержать каждую отдельную реплику любого персонажа:

```sql
CREATE TABLE utterances (
    id SERIAL PRIMARY KEY,
    chapter_id INTEGER NOT NULL REFERENCES chapters(id) ON DELETE CASCADE,
    character_id INTEGER NOT NULL REFERENCES characters(id) ON DELETE CASCADE,
    content TEXT NOT NULL,               -- текст реплики
    position_in_chapter INTEGER,         -- порядковый номер реплики в главе (для сортировки)
    -- опционально: можно добавить локацию, если диалог привязан к месту
    location_id INTEGER REFERENCES locations(id) ON DELETE SET NULL
);

CREATE INDEX idx_utterances_chapter ON utterances(chapter_id);
CREATE INDEX idx_utterances_character ON utterances(character_id);
CREATE INDEX idx_utterances_chapter_order ON utterances(chapter_id, position_in_chapter);
```

Теперь каждая реплика жёстко привязана к персонажу и главе.

### 3. Как заполнить таблицу реплик (при «чтении» книги ИИ)

Это ключевой момент: нужна обработка текста с помощью LLM или специализированного парсера диалогов. Можно отправить главу модели с инструкцией:

> «Выдели все диалоги из этого текста. Для каждого диалога укажи:  
> - говорящего персонажа (должен совпадать с именем из списка персонажей),  
> - точный текст реплики,  
> - порядковый номер в главе.  
> Ответ верни строго в формате JSON.»

На выходе получим массив объектов, которые легко вставить в `utterances`. При вставке нужно сопоставить имена с `character_id` (по таблице `characters` для данной книги).

Этот шаг делается один раз при импорте книги, и дальше вся структура уже готова к запросам.

### 4. Запрос «все диалоги Гермионы»

Теперь он элементарен:

```sql
SELECT u.content, u.position_in_chapter, ch.chapter_number, b.title
FROM utterances u
JOIN characters c ON u.character_id = c.id
JOIN chapters ch ON u.chapter_id = ch.id
JOIN books b ON ch.book_id = b.id
WHERE c.name = 'Гермиона' AND b.id = 1  -- конкретная книга, если нужно
ORDER BY ch.chapter_number, u.position_in_chapter;
```

Можно ещё добавить контекст: показать соседние реплики, чтобы восстановить диалог полностью (кто что сказал до и после). Для этого, зная `chapter_id` и `position_in_chapter`, легко подгрузить несколько соседних записей:

```sql
SELECT u2.content, c2.name AS speaker
FROM utterances u2
JOIN characters c2 ON u2.character_id = c2.id
WHERE u2.chapter_id = ? AND u2.position_in_chapter BETWEEN ?-2 AND ?+2
ORDER BY u2.position_in_chapter;
```

Например, PostgreSQL позволяет искать фразы, но такой подход будет пропускать диалоги, где имя не упоминается, или путать повествование с речью. Поэтому для качественного решения **я однозначно рекомендую таблицу реплик**.

### 6. Связь с другими объектами

Таблицу `utterances` можно расширить:
- `location_id` – если диалог происходит в конкретном месте.
- `arc_id` – если нужно прямо привязать реплику к сюжетной арке (хотя обычно это решается через связь главы с аркой).
- `embedding` (pgvector) для семантического поиска по содержанию реплик.

Таким образом, ответ на ваш вопрос: **в таблице `characters` нет прямых ссылок на текст**. Ссылка на оригинал идёт через `chapter_characters` (появление в главе) и, главное, через `utterances` (конкретные реплики). Последняя и даёт возможность мгновенно найти все диалоги персонажа.

Если нужно, могу помочь с кодом на Python для парсинга диалогов или с эндпоинтом FastAPI, который обрабатывает запрос «покажи диалоги X».

### Подход 1: Хранение ссылок (индексов) на текст главы

Таблица выглядела бы примерно так:

```sql
CREATE TABLE utterance_refs (
    id SERIAL PRIMARY KEY,
    chapter_id INTEGER NOT NULL REFERENCES chapters(id),
    character_id INTEGER NOT NULL REFERENCES characters(id),
    start_char INTEGER NOT NULL,   -- позиция первого символа в content главы
    end_char INTEGER NOT NULL      -- позиция последнего символа
);
```

**Плюсы:**
- Экономия места (текст не дублируется).
- При изменении главы (если такое вообще возможно) реплики автоматически обновляются.

**Минусы (и их много):**

1. **Хрупкость**  
   Если по какой-то причине текст главы изменится (например, вы исправите опечатку, переформатируете абзацы), все индексы станут невалидными. Придётся пересчитывать заново. При импорте книг это вряд ли случится, но в процессе жизни базы правки могут быть.

2. **Сложные запросы**  
   Чтобы получить все диалоги Гермионы, нужно:
   - Найти все записи в `utterance_refs` для персонажа.
   - Для каждой записи извлечь подстроку из `chapters.content` с помощью `SUBSTRING(content FROM start_char FOR end_char - start_char)`.
   Это более громоздко и медленнее, чем просто `SELECT content FROM utterances WHERE character_id = ?`.

3. **Невозможность индексации реплик**  
   Вы не сможете создать полнотекстовый индекс (GIN) по содержимому реплик, потому что сами реплики хранятся размазанными внутри большой колонки `chapters.content`.  
   Чтобы искать по словам внутри реплик («найди реплику, где Гермиона говорит о философском камне»), придётся всё равно извлекать подстроки и потом фильтровать на лету — это крайне медленно.

4. **Векторный поиск (эмбеддинги) реплик**  
   Для семантического поиска по репликам («найди похожие по смыслу диалоги») нужно вычислить embedding для каждой реплики. Если реплика не вынесена отдельно, вам придётся каждый раз вырезать её из главы, вычислять вектор на лету или хранить вектор в `utterance_refs` (что уже странно: храним вектор, но не сам текст).  
   А если текст реплики поменяется, вектор станет неактуальным. Лучше хранить и текст, и его вектор в одной строке — тогда связка остаётся целостной.

5. **Кэширование и микросервисы**  
   Если ИИ-сервер будет отделён от БД, гораздо эффективнее передавать готовый массив реплик, чем заставлять его дёргать полный текст глав и резать строки.

6. **Удобство анализа**  
   Храня реплики отдельно, вы можете легко подсчитать количество реплик персонажа, среднюю длину, распределение по главам и т.п. простыми агрегациями. С индексами это сделать сложнее.

### Подход 2: Хранение самих реплик (предложенный ранее)

Таблица `utterances` с колонкой `content TEXT`.  
**Плюсы:**
- Мгновенная выдача всех реплик персонажа без дёрганья глав.
- Возможность построить полнотекстовый индекс (`GIN`) по `content` для быстрого поиска слов.
- Возможность добавить колонку `embedding` и искать семантически подобные диалоги.
- Независимость от форматирования текста глав.
- Простота в коде: один SELECT — и всё готово.

**Минусы:**
- Дублирование данных (текст реплики хранится и в `chapters.content`, и в `utterances.content`).  
  Но в вашем случае это не проблема: объём 100 книг × ~200 реплик на главу — это порядка 200–500 МБ, что ничтожно для современных дисков. Дублирование здесь оправдано ради скорости и гибкости.

---

### Гибридный вариант (компромисс)

Можно хранить и то, и другое:

```sql
CREATE TABLE utterances (
    id SERIAL PRIMARY KEY,
    chapter_id INTEGER NOT NULL REFERENCES chapters(id),
    character_id INTEGER NOT NULL REFERENCES characters(id),
    content TEXT NOT NULL,
    start_char INTEGER,          -- опционально, для обратной ссылки
    end_char INTEGER,            -- опционально
    embedding vector(1536)       -- если нужно
);
```

Тогда у вас будет и полный текст для быстрых запросов, и возможность привязаться к позиции в главе, если вдруг понадобится показать контекст вокруг реплики. Но для контекста проще сделать запрос с `position_in_chapter` (порядковым номером) и подгрузить соседние реплики из той же таблицы — это ещё быстрее, чем лазить в оригинальный текст.

---

### Что случится, если всё-таки хранить только ссылки?

Представьте, что ИИ спрашивает: «Покажи все диалоги Гермионы и найди среди них те, где говорится о зельях».  
Ваш код должен:
1. Вытащить ID персонажа.
2. Выбрать все `utterance_refs` для него, получить список `(chapter_id, start, end)`.
3. Для каждого — прочитать `SUBSTRING(content FROM start FOR end - start)`.
4. Отфильтровать те, которые содержат слово «зелья» (или пройтись по ним с помощью LLM).

Это получается N+1 запросов (или один запрос с генерацией всех подстрок через `LATERAL`, но он всё равно не сможет использовать индексы по содержимому реплик). При большом количестве книг и частых запросах это будет тормозить.

Если же вы храните `utterances.content`, то задача сводится к:
```sql
SELECT content FROM utterances
WHERE character_id = ? AND content_tsv @@ plainto_tsquery('russian', 'зелья');
```
— мгновенный поиск по индексу.

### Итог

Хранение самих реплик — это не просто копирование данных, а **денализация ради производительности и удобства запросов**, типичная для поисковых систем. В вашем проекте с ИИ это полностью окупается. Ссылки на позиции можно хранить как дополнительную информацию, но основным источником для ответов должны быть готовые тексты реплик.

Так что мой совет остаётся: смело храните `content` в `utterances`. Это даст максимальную гибкость при разработке вопросно-ответной системы.

---

## Почему ссылки на позиции в тексте не спасают

Если вы храните только `start_char` / `end_char`, то при **любом** изменении текста главы (даже исправлении опечатки в другом месте) все смещения могут «поехать», и ссылки станут указывать не туда. Их всё равно придётся пересчитывать заново, как и перегенерировать реплики. Так что это не избавляет от переобработки, а лишь добавляет неудобства.

## Как эффективно обновлять `utterances` при изменении главы

### 1. Храните версию или хеш контента главы

Добавьте в таблицу `chapters` поле:

```sql
ALTER TABLE chapters ADD COLUMN content_hash TEXT;
```

При сохранении главы вычисляйте, например, `hashlib.sha256(content.encode()).hexdigest()`.

### 2. Обнаружение изменений

Когда администратор (или ИИ) выгружает обновлённую книгу:
- Сравните хеши новых глав с сохранёнными.
- Для всех глав, где хеш изменился, запустите повторный парсинг диалогов и обновите `utterances`.

### 3. Стратегия обновления реплик

Лучший способ — **удалить все старые реплики изменённой главы и вставить новые**:

```sql
DELETE FROM utterances WHERE chapter_id = ?;
-- Затем вставляем заново распарсенные реплики
```

Это атомарно в рамках транзакции, и вы не получите рассогласованных данных.

**Почему не мержить?**  
Сопоставить старые реплики с новыми — сложно, ведь они могут измениться по длине, содержанию и даже количеству. Проще пересоздать, тем более для 100 книг это лёгкая операция.

### 4. Автоматизация через триггеры (для версионности)

Если книга может меняться часто, и нужно сохранять историю реплик, можно ввести версионирование:

```sql
ALTER TABLE utterances ADD COLUMN version INTEGER DEFAULT 1;
ALTER TABLE chapters ADD COLUMN current_version INTEGER DEFAULT 1;
```

При обновлении главы:
- Увеличить `chapters.current_version`.
- Не удалять старые реплики, а вставлять новые с увеличенной `version`.
- В запросах всегда фильтровать по последней версии (`WHERE version = (SELECT current_version FROM chapters WHERE id = chapter_id)`).

Так сохранится история, но для вашего проекта это избыточно.

---

## Что делать, если изменения затрагивают персонажей?

Может быть, в новой редакции персонажа переименовали, изменили характер и т.д. Тогда нужно обновлять и таблицы `characters`, `locations`, `story_arcs`. Здесь уже лучше не полагаться на автоматический парсинг, а предоставить интерфейс для ручного подтверждения изменений или полуавтоматического с контролем ИИ.

Типичный процесс:
1. ИИ-парсер выдаёт предложения по изменённым сущностям.
2. Редактор (человек или доверенный скрипт) подтверждает слияние.
3. Для неизменившихся сущностей ничего не трогается.

---

## Итог

- **Обновление реплик при изменении книги — стандартная операция**, которая решается повторным прогоном парсера только для изменившихся глав.
- Даже если бы вы хранили ссылки на позиции, их пришлось бы пересчитывать точно так же. Хранение полного текста реплик не усложняет, а упрощает обновление (не надо возиться с пересчётом смещений).
- При небольших объёмах (100 книг) переобработка занимает секунды и не заметна для системы.

Таким образом, выбранный подход с `utterances` остаётся и масштабируемым, и удобным даже при периодических правках книг.

### Как это могло бы работать

Гарри вздохнул.

### Плюсы использования якорей

1. **Точная привязка без перепарсинга**  
   Если текст главы изменился, но якоря остались на месте (или были перенесены редактором), можно обновить только те реплики, чей контент изменился, без перегенерации всей главы.

2. **Устойчивость к сдвигам текста**  
   В отличие от числовых позиций (`start_char`, `end_char`), якоря не ломаются при вставке/удалении текста до них, пока сами якоря не задеты.

3. **Соавторство с человеком**  
   Если вы предполагаете ручную доразметку или правки, редактор может видеть якоря в специальном режиме и корректировать их.

4. **История изменений**  
   Можно хранить версии реплик: при изменении текста внутри `<utterance>` создаётся новая версия записи в `utterances`, а старые сохраняются.

---

### Минусы и подводные камни

1. **Загрязнение исходного текста**  
   Исходный текст книги перестаёт быть «чистым». Это может быть проблемой, если вы захотите показывать текст читателям без обработки, или передавать его в LLM для других целей (придётся вырезать якоря).

2. **Работа с существующими книгами**  
   Если книги приходят извне (сканы, EPUB, TXT), в них изначально нет якорей. Вам всё равно придётся **один раз пройти парсером**, который расставит якоря. То есть якоря не избавляют от первоначальной обработки — они лишь упрощают последующие обновления.

3. **Сложность хранения**  
   Текст главы с якорями — это уже не plain text, а структурированный. Хранить его в `TEXT`-колонке можно, но для извлечения «чистого» текста нужно будет либо вырезать теги на лету, либо хранить два варианта: `content_raw` с якорями и `content_plain` для чтения. Это усложняет схему.

4. **Поведение ИИ-моделей**  
   Если вы подаёте LLM текст с якорями для поиска или генерации ответа, это может сбивать модель, добавляя шум. Вам придётся предварительно очищать текст.

5. **Синхронизация якорей при редактировании**  
   Представьте, что человек (или другой ИИ) редактирует главу в текстовом редакторе и случайно удаляет якорь или дублирует его. Тогда связь разрушится. Нужна система валидации при сохранении, что усложняет код.

---

### Альтернатива: мягкие якоря через идентификацию контента

Вместо вставки маркеров в сам текст можно использовать «логические якоря»: хранить в `utterances` **контрольную сумму содержимого реплики** (например, первые 50 символов + длина). При обновлении главы:
- Вычислить хеш каждой реплики, предложенной новым парсером.
- Сопоставить с существующими записями по близости текста.
- Если реплика не изменилась, оставить её; если изменилась или добавилась — обновить/вставить.

Это позволяет автоматически синхронизировать без внедрения в оригинальный текст.

### Что выбрать для вашего проекта?

Поскольку книг ~100 и они, вероятно, статичны (загружены один раз), якори могут быть **избыточным усложнением**. Вот сценарий без якорей:

1. При импорте книги парсер (LLM) разбивает на главы и выделяет реплики, создавая записи в `utterances`.
2. Если позже вы получаете обновлённую версию книги, вы заново прогоняете парсер для изменённых глав и перезаписываете соответствующие `utterances` (или делаете версионирование, если нужна история).
3. Для быстрого сравнения используете хеши глав (`content_hash`).

Это просто, надёжно и не требует менять исходный текст.

**Якори стоит применять, если:**
- Вы планируете часто редактировать книги вручную в среде, которая понимает разметку.
- У вас огромный объём книг, и повторный полный парсинг — дорогая операция.
- Вы хотите хранить текст с семантической разметкой как единый источник правды (например, XML/TEI Shakespeare).

Но для 100 книг и работы с ИИ я бы остался на варианте с хранением чистого текста и отдельной таблицей `utterances`, с обновлением через перепарсинг по хешам. Это даст наилучший баланс простоты и функциональности.

Если будут конкретные примеры, где якоря кажутся необходимыми, обсудим детальнее.
