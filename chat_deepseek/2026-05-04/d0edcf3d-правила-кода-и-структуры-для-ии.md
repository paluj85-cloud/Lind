### RESPONSE (ИИ)

## 1. Философия проекта: явность и предсказуемость

ИИ отлично угадывает паттерны. Если весь код написан в едином стиле и следует очевидным конвенциям, модель будет генерировать продолжение без ошибок.

*   **Явное лучше неявного.** Не используй магию, метапрограммирование или хитрые трюки без крайней нужды.
*   **Одна сущность — одна ответственность.** Каждый модуль, класс, функция делают что-то одно. Если не можешь описать назначение файла одним коротким предложением, его нужно разбить.
*   **Принцип минимального удивления.** Код должен вести себя так, как читается по названиям. Функция `getUserName` не должна менять аватар.

## 2. Именование: контекст без чтения тела

Хорошее имя избавляет ИИ от необходимости анализировать всю реализацию — он понимает смысл по сигнатуре.

*   **Переменные:** `userName`, `totalPrice`, `isActive` (булевы только через `is`, `has`, `can`).
*   **Функции:** глагол + существительное. `fetchUserData`, `calculateTax`, `validateForm`.
*   **Классы/компоненты:** существительное. `UserProfile`, `PaymentGateway`, `Button`.
*   **Файлы:** имя должно полностью совпадать с экспортом по умолчанию или главной сущностью. Модуль с классом `UserService` → файл `UserService.ts`. Компонент `UserCard` → `UserCard.tsx`. Избегай `index.ts`-прокладок глубоко в папках (только на верхнем уровне для публичного API).

## 3. Структура каталогов: карта для ИИ

## 4. Правила написания кода под ИИ

### 4.1. Чистые функции и иммутабельность
Избегай побочных эффектов внутри функций, не обозначенных как мутирующие. Функции должны возвращать новое значение, а не менять переданное. ИИ часто ошибается именно на сайд-эффектах.
```typescript
// ПЛОХО: ИИ может не заметить мутацию внутри цепочки
function applyDiscount(cart) {
  cart.items.forEach(item => item.price *= 0.9);
  return cart;
}

// ХОРОШО: Понятная сигнатура и предсказуемое поведение
function applyDiscount(cart: Cart): Cart {
  const items = cart.items.map(item => ({ ...item, price: item.price * 0.9 }));
  return { ...cart, items };
}
```

### 4.2. Строгая работа с типами
Используй TypeScript или хотя бы JSDoc-аннотации в JavaScript. Типы — это **наиболее сжатая подсказка** для нейросети о допустимых значениях и структуре данных.
```typescript
// ПЛОХО: ИИ не знает, что свойства опциональны, а id — строка
function createUser(data) { ... }

// ОТЛИЧНО: Модель точно знает форму объекта
interface UserDTO {
  id: string;
  name: string;
  email: string;
  age?: number;
}
function createUser(data: UserDTO): Promise<User> { ... }
```

### 4.3. Ограничение размера и сложности
1.  **Функция ≤ 20 строк.** Если больше — декомпозируй. ИИ хорошо генерирует маленькие части.
2.  **Файл ≤ 150 строк.** Это гарантирует, что весь файл поместится в контекстное окно без необходимости жертвовать соседними файлами.
3.  **Модуль ≤ 7 внутренних элементов.** Если у объекта/модуля больше 7 публичных методов/полей, вероятно, нарушен Single Responsibility Principle.

### 4.4. Комментарии для ИИ
Пиши комментарии там, где контекст не очевиден. Хороший комментарий объясняет **«почему»**, а не «что».
```typescript
// ПЛОХО: повтор кода
// Считаем общую цену
const total = cart.items.reduce((sum, i) => sum + i.price, 0);

// ХОРОШО: пояснение бизнес-логики
// Согласно требованию ISSUE-4512, стоимость доставки не входит в корзину,
// пока пользователь не выберет регион. Сумма на этом шаге — чистая.
const pureProductCost = cart.items.reduce(...);
```

## 5. Явный контекст для ИИ: папка `.ai`

Создай в корне папку `.ai` (или встрой в `docs/`), которую ты будешь явно передавать в промпт при старте новой сессии или крупных задач.

*   **`architecture.md`**: схема приложения (слои, потоки данных), глоссарий терминов.
*   **`code-style.md`**: ссылки на эту памятку или чек-лист.
*   **`tech-stack.md`**: версии фреймворков, ключевых библиотек и причины их выбора.
*   **`prompts/`**: готовые сценарии для типовых задач («Добавление нового модуля», «Написание миграции БД»).

При обращении к ИИ просто укажи: *«Перед выполнением задачи изучи `./ai/architecture.md` и `./ai/code-style.md`»*.

## 6. Инструменты, насаждающие дисциплину

Не полагайся только на человеческую (или ИИ) память. Заставь код автоматически форматироваться и проверяться.

*   **Prettier**: единый стиль отступов, кавычек, точек с запятой.
*   **ESLint**: правила, запрещающие `var`, `any`, неиспользуемые переменные, сложные конструкции.
*   **Husky + lint-staged**: запуск линтеров на pre-commit, чтобы «нечистый» код физически не попадал в репозиторий.
*   **SonarLint** (опционально): IDE-плагин для обнаружения багов и code smells прямо при написании.

## Итоговый чек-лист для внесения правок с ИИ

1.  Я пишу промпт, прикрепив нужный файл и файлы его типов (интерфейсов).
2.  Я ожидаю, что сгенерированный код уже отформатирован.
3.  Я прошу ИИ «объясни, какие части структуры затронуты» перед тем как дать код.
4.  Сгенерированный код я вставляю в проект, и линтер сразу подсвечивает потенциальные проблемы.
5.  Код нигде не дублируется — утилита вынесена в общий `lib`, компонент — в `shared/ui`.

Такой подход позволит рассматривать ИИ не как «генератор каши», а как усиление твоей же архитектурной мысли. Кодовая база останется живой, расширяемой и понятной. Удачи в разработке!

## 4. Правила именования
- **Переменные:** существительные, точно описывающие хранимые данные (`userId`, `totalAmount`, `isLoading`, `hasError`).
- **Функции:** глагол + существительное (`fetchUser`, `calculateTax`, `sendNotification`). Булевы функции должны начинаться с `is`, `has`, `can`.
- **Классы / компоненты:** существительное в PascalCase (`UserProfile`, `PaymentForm`).
- **Файлы:** имя файла полностью совпадает с именем экспортируемой по умолчанию сущности (модуль `UserService` → файл `UserService.ts`, компонент `LoginForm` → `LoginForm.tsx`). Файлы `index.ts` допускаются только на уровне публичного API домена (экспорт) и в корне `src`.

## 5. Строгость типов и сигнатур
- Все входные параметры функций и возвращаемые значения обязаны иметь явные типы (никаких `any`, кроме крайне обоснованных случаев).
- Типы должны быть самодокументированными: опциональные поля обозначены `?`, read-only помечены `readonly`.
- Предпочтительны `interface` для описания объектов, `type` для объединений и утилитарных типов.

## 6. Размеры и сложность
- **Функция** — не более 20 строк тела. Если больше, декомпозируй на вспомогательные функции.
- **Файл** — не более 150 строк. Достижение лимита — сигнал к выделению подмодуля.
- **Модуль / класс** — не более 7 публичных методов/полей.

## 7. Комментарии
- Комментарий объясняет **«почему»**, а не «что делает код».

- Перед каждой функцией, которая экспортируется, допустим краткий JSDoc с описанием назначения и параметров.

## 8. Работа с зависимостями и сайд-эффектами
- Запросы к API, работа с таймерами, localStorage и другими браузерными API должны быть изолированы в соответствующих `api/` или `lib/` модулях.
- В компонентах используй хуки (`useEffect`, `useQuery`) только для явной синхронизации, без скрытого изменения стейта.
- Избегай прямого импорта реализации, используй `index.ts` модуля как публичный контракт.

## 9. Требования к генерируемому коду

### 🧠 Универсальные принципы проектирования

Классические принципы проектирования созданы для того, чтобы код было легко читать, тестировать и изменять. Это именно то, что нужно для эффективной работы с ИИ, так как они помогают писать код, в котором сложнее допустить ошибку. Два столпа, на которых всё держится:

*   **SOLID**: Это аббревиатура из пяти принципов объектно-ориентированного программирования, сформулированных Робертом Мартином[reference:0][reference:1]. Для нас с вами самый важный — **S (Single Responsibility Principle)**. Он гласит, что каждый класс или модуль должен иметь одну и только одну причину для изменения[reference:2][reference:3]. Это напрямую перекликается с моим советом про «одну сущность — одну ответственность».
*   **KISS (Keep It Simple, Stupid)**: Пожалуй, самый важный принцип при работе с ИИ. Код должен быть настолько простым, насколько это возможно. Сложные, запутанные конструкции — главный источник ошибок для нейросети. Этому принципу вторят практически все найденные руководства, формулируя его как «Simplicity First»[reference:4][reference:5][reference:6].
*   **DRY (Don't Repeat Yourself)**: Не дублируйте код. Любое повторение — это риск, что при исправлении ошибки или добавлении функционала вы забудете обновить одну из копий. ИИ это тоже касается: он может найти и изменить только первый попавшийся фрагмент[reference:7][reference:8].
*   **YAGNI (You Aren't Gonna Need It)**: Не пишите код «на будущее», если в нём нет прямой необходимости сейчас. Это правило защищает от переусложнения и появления мёртвого кода, который будет сбивать ИИ с толку[reference:9][reference:10].

### 📜 Готовые наборы правил для ИИ

Если вы хотите не просто познакомиться с принципами, а сразу дать готовую инструкцию своему ИИ-помощнику, можно взять за основу один из этих файлов. Они написаны специально для «агентного» кодинга:

Он покрывает все аспекты: от фундаментальных принципов и правил рабочего процесса (например, «Атомарные изменения») до конкретных гарантий качества («Функции > 30 строк — повод для рефакторинга»). ### 🛠️ Инструменты автоматического контроля

Это «must-have». Лучшие правила — те, которые исполняются автоматически. Они не дадут ни вам, ни ИИ сохранить «плохой» код в репозиторий.

*   **ESLint** — это анализатор кода, который находит потенциальные ошибки и несоответствия стилю, заданному в его конфигурации[reference:13].
*   **Prettier** — это форматер кода, который автоматически приводит его к единому стилю (отступы, кавычки, точки с запятой)[reference:14]. Он решает все споры о форматировании.
*   **Husky + lint-staged** — связка инструментов, которая позволяет запустить проверку (ESLint) и форматирование (Prettier) для изменённых файлов прямо перед созданием коммита[reference:15]. Это последний рубеж обороны, который гарантирует, что в репозиторий не попадёт «грязный» код.

### 📜 AGENTS.md: Новый универсальный стандарт

Это, пожалуй, самое важное изменение. По аналогии с `README.md`, был предложен файл `AGENTS.md` как единый стандарт для инструкций AI-агентам. Эту инициативу поддерживают такие гиганты, как OpenAI, Google, Anthropic и Cursor[reference:0][reference:1]. Смысл в том, чтобы иметь один файл, который понимают все инструменты, вместо того чтобы плодить отдельные конфигурации под каждую IDE. Многие современные инструменты, включая Cline, уже начинают его поддерживать.

### 🏆 Топ-3 альтернатив и дополнений

Помимо уже известного вам репозитория `obviousworks`, вот несколько проектов с сильными наборами правил:

1.  **Agentic Coding Rulebook (от `obviousworks`)**:
    Это прямой наследник и расширение того `global_rules.md`, который вы уже видели. По сути, это целый **комплект стандартов для профессиональной AI-разработки**.
    *   **Сильные стороны**: Очень структурированный, "боевой" набор правил, с фокусом на безопасность. Включает отдельные файлы по безопасности (`security_rules.md`), производительности (`performance_rules.md`), тестированию (`testing_rules.md`) и шаблоны для миграции с других инструментов[reference:2].
    *   **Итог**: Это выбор №1, если вам нужен максимально полный и серьезный набор правил, который можно внедрить в профессиональный проект.

2.  **Vibe Coding AI Rules (от `obviousworks`, специально для вас)**:
    Этот проект — "золотая жила" для пользователя Cline. Он содержит **уже готовые конфигурации для конкретных IDE**, включая и `.clinerules`[reference:3]!
    *   **Сильные стороны**: Практичность. Вы можете пойти в папку `ide-specific/cline/` и сразу увидеть, как правильно структурировать ваш `.clinerules` для проекта. Это самый быстрый способ получить работающий, а не просто теоретический, набор правил[reference:4].
    *   **Итог**: Это лучший выбор для быстрого старта с Cline, так как избавляет от догадок о формате файла.

3.  **Cursor Prompts (`e-gov/cursor-prompts`)**:
    Очень качественный и структурированный набор от команды, которая использует его для корпоративной разработки.
    *   **Сильные стороны**: Правила разбиты по категориям (безопасность, логирование, код-стандарты) и по языкам (Java, TypeScript, Python/FastAPI, Nuxt/Vue)[reference:5]. Особенно полезна папка `commands/` с готовыми промптами для конкретных задач, например, `security-check.md`[reference:6].
    *   **Итог**: Прекрасный пример того, как зрелая команда организует правила для AI-разработки. Можно смело копировать структуру и отдельные файлы, особенно если вы работаете с указанными технологиями.

### 💎 Итог: что делать?

Все эти находки не заменяют, а скорее **дополняют** ваш текущий `.clinerules`. Чтобы получить максимальную пользу:

1.  **Возьмите эталонную структуру**: Загляните в папку `ide-specific/cline/` проекта **Vibe Coding AI Rules**[reference:10]. Посмотрите, как там организован `.clinerules` и какие инструкции Cline понимает лучше всего.
2.  **Используйте лучшие своды правил**: Проекты **Agentic Coding Rulebook** и **Cursor Prompts** — это кладезь готовых, мощных и безопасных правил. Выберите оттуда те, что подходят под ваш стек (например, правила безопасности или код-стиля) и добавьте их в свой `.clinerules`[reference:11][reference:12].
3.  **Следите за стандартом**: Помните про `AGENTS.md`. Это будущее, и многие инструменты к нему придут. В идеале ваш проект должен быть готов к его использованию.

Такая стратегия — использовать готовые мощные блоки и собирать из них свой идеальный набор правил — является самым гибким и эффективным подходом на сегодняшний день.

Если захотите обсудить какой-то из этих проектов подробнее или понадобится помощь с интеграцией их правил в ваш `.clinerules`, дайте знать.
