Техническое задание: модуль извлечения нарративных атомов (Admin Book Importer)
1. Общее описание
Модуль предоставляет интерфейс (бэкенд + веб-админка) для загрузки художественных книг, их автоматического анализа языковой моделью (ИИ) и формирования набора нарративных атомов семи типов: конфликты, твисты, дилеммы, отношения, локации, сенсорные крючки, детали NPC. Администратор просматривает извлечённые атомы, редактирует их, подтверждает или удаляет, после чего одним действием импортирует их в основную библиотеку системы генерации арок.

Бэкенд реализуется на FastAPI (Python 3.11+), работает асинхронно. Длительная обработка ведётся в фоне с обновлением состояния в БД; фронтенд опрашивает прогресс.

5. API-методы
Все URL начинаются с /api/admin.

5.1 Загрузка и управление книгой
POST /books/upload
Принимает multipart/form-data с полем file. Валидирует расширение (.txt, .epub, .fb2, .md) и размер. Преобразует в plain text (для fb2 извлекается текст из <body> по абзацам с вставкой маркеров глав, если обнаружены секции). Создаёт запись в arc_book, сохраняет plain text на диске. Возвращает { "book_id": 1 }.

GET /books/current
Возвращает информацию о текущей загруженной книге (последней по времени) или 404, если ни одной книги не загружено. Ответ:

json
{
  "id": 1,
  "original_filename": "...",
  "file_size": 12345,
  "status": "awaiting_review",
  "settings": {...},
  "created_at": "..."
}
(Списка книг как такового нет, всегда работаем с одной.)

POST /books/current/start
Запускает или возобновляет анализ для текущей книги. Тело:

json
{
  "fragment_size": 8000,
  "split_by_chapters": true,
  "extraction_prompt": "..." (опционально),
  "postprocessing_prompt": "..." (опционально),
  "restart": false
}
Если restart=true, удаляет все существующие фрагменты и черновики, разбивает книгу заново. Если false и есть фрагменты, продолжает обработку с незавершённых. Промпты сохраняются в arc_prompt (при отличии). Книга переводится в status='processing', создаётся/обновляется arc_processing_state. Запускается фоновая задача process_book(book_id).

POST /books/current/abort
Устанавливает статус книги 'aborted', оставляя все промежуточные данные.

DELETE /books/current
Полностью удаляет книгу, фрагменты, черновики и состояние. После этого можно загружать новую.

6. Процессы обработки
6.1 Загрузка файла и конвертация
Поддерживаются форматы: .txt, .epub, .fb2, .md.

FB2: извлекается содержимое всех <body>, текст структурируется по абзацам <p>. Для каждой секции верхнего уровня (или <section>) вставляется строка-маркер вида [Глава: <название>] для облегчения последующего разбиения.

epub: через ebooklib извлекается текст глав; аналогично вставляются маркеры.

md: убирается минимальная разметка, заголовки # обрабатываются как потенциальные границы глав.

Итоговый plain text сохраняется на диск, размер записывается в arc_book.file_size.

6.2 Разбиение на фрагменты
Если split_by_chapters=true, текст сканируется на предмет шаблонов глав (Глава, Chapter, #, маркер [Глава: ...]). Текст разделяется по этим точкам.

Каждый полученный блок (глава) проверяется на длину. Если длина превышает fragment_size, блок режется по границам абзацев (двойной перевод строки) так, чтобы каждый фрагмент был ≤ fragment_size, но максимально близок к нему.

Фрагменты записываются в arc_fragment с порядковым номером, статусом 'pending'.

6.3 Фоновая обработка книги
Функция process_book(book_id) реализует конечный автомат:

Стадия фрагментации
Устанавливает arc_processing_state.stage = 'fragmentation', выполняет разбиение, записывает фрагменты.

Стадия экстракции
stage = 'extraction'. Запускает параллельную обработку фрагментов с ограничением семафором (PARALLEL_FRAGMENTS из конфига, по умолчанию 1).

Для каждого фрагмента:

Установить статус 'in_progress'.

Отправить запрос к ИИ (системный промпт из arc_prompt с типом extraction, пользовательское сообщение – текст фрагмента).

Получить ответ JSON. Сервер ожидает JSON-объект с ключами по типам атомов. Для каждого типа создаются записи в arc_draft_*. Названия масок преобразуются в коды (транслитерация, UPPER_CASE, обрезка до 50 символов). Генерируются временные ID: TEMP-{book_id}-{PREFIX}-{NNNN}, где PREFIX: CONF, TWIS, DILE, REL, LOC, SENS, NPC.

В случае ошибки (сеть, таймаут, невалидный JSON) фрагмент снова становится 'pending', увеличивается retry_count. При retry_count >= 2 фрагмент помечается 'error', вся обработка останавливается: книга получает status='error', в arc_processing_state.error_message заносится описание. Успешно обработанные фрагменты остаются.

Обновление счётчика processed_fragments происходит атомарно после каждого успешного фрагмента.

Стадия постобработки
После успешной экстракции всех фрагментов:

stage = 'postprocessing'.

Собираются все черновые атомы из всех таблиц arc_draft_*.

Формируется JSON-массив и отправляется ИИ со вторым промптом (postprocessing).

Ожидаемый ответ содержит:

deduplicated: массив итоговых атомов с полями data (полный объект атома), merge_of (список временных ID, которые были объединены), weak_candidate (bool).

compatibility_suggestions: объект, сопоставляющий temp_id → обновлённые поля совместимостей внутри data.

tags_suggestions: объект, сопоставляющий temp_id → массив строк тегов (которые должны соответствовать справочнику arc_tag; если тег новый, он будет создан при импорте).

Старые черновики удаляются. Создаются новые записи в arc_draft_* на основе deduplicated, заполняются merge_of, weak_candidate. Поля совместимостей и тегов внутри data обновляются согласно предложениям. Статус новых черновиков – 'draft'.

Книга переходит в status='awaiting_review', arc_processing_state.stage = 'awaiting_review'.

6.4 Редактирование и импорт
Администратор редактирует черновики (меняет data, ставит статус confirmed или deleted). Все правки сохраняются с is_manual_edit = true.

При вызове импорта:

Для каждого подтверждённого атома генерируется постоянный ID, используя автоинкремент (можно блокировкой таблицы) в соответствии с форматом: CONF-{NNNN}, TWIST-{NNNN} и т.д.

В JSON-полях data временные ID, ссылающиеся на другие атомы этой книги, заменяются на постоянные (по карте map_temp_to_perm). Внешние ID (из основной библиотеки) остаются.

Данные записываются в основные таблицы. Связи маски-конфликт – в arc_conflict_mask; теги – в arc_atom_tag (предварительно проверяется наличие тега в arc_tag, новые добавляются).

После импорта генерируется ответ с количеством импортированных атомов по типам.

Книга переводится в status='done'.

7. Обработка ошибок и повторные попытки
Запрос к ИИ обёрнут в логику повторных попыток (2 повтора). Задержка между повторами 2 с.

Исключения: httpx.HTTPError, asyncio.TimeoutError, ошибки парсинга JSON.

При достижении лимита попыток фрагмент останавливается, весь процесс прерывается.

Администратор может перезапустить анализ с настройкой restart=true для полного сброса, либо restart=false, чтобы продолжить с упавшего фрагмента.

FOREIGN KEY на fragment_id REFERENCES arc_fragment(id) ON DELETE SET NULL — ок.

Поле merge_of JSONB: хранит список temp_id, которые были объединены. В описании API GET /books/current/drafts показывает merge_of. Отлично.

Но важно: при постобработке ИИ может предложить объединить атомы в один, merge_of будет содержать старые temp_id. Новый temp_id будет сгенерирован заново? При этом merge_of содержит старые temp_id, которые были удалены. Но при последующем импорте, когда постоянные ID заменяются, карта map_temp_to_perm будет включать старые temp_id? Импорт берёт черновики со статусом confirmed, у которых есть temp_id. Но merge_of ссылается на старые temp_id, которые больше не существуют в черновиках (удалены). Нужно сохранить карту старых temp_id на новые, чтобы при импорте разрешить ссылки в merge_of. В ТЗ об этом не сказано. Это серьёзная логическая ошибка. Нужно либо сохранять старые черновики как-то, либо передавать карту. В deduplicated ИИ может возвращать атом с merge_of, содержащим старые temp_id. При создании нового черновика merge_of будет содержать старые temp_id. После удаления старых записей эти ID больше нигде не хранятся. При импорте эти старые ID не имеют отображения в постоянные, потому что они не являются атомами для импорта (они не confirmed). Но merge_of в новом черновике — это ссылка на исходные атомы, которые были объединены. Это метаинформация, может быть просто списком временных ID для истории, не требующая замены на постоянные ID. В процессе импорта заменяются временные ID на постоянные в полях совместимости. merge_of — это не поле совместимости, это история объединения, она может остаться с temp_id. Это не вызовет ошибок, если эти temp_id не используются для ссылок целостности. Но если в compatible_conflicts внутри data нового атома есть ссылки на старые temp_id, которые были объединены и больше не существуют как отдельные атомы, то при постобработке они должны быть обновлены на новые temp_id. ТЗ говорит: "Поля совместимостей и тегов внутри data обновляются согласно предложениям." compatibility_suggestions: "объект, сопоставляющий temp_id → обновлённые поля совместимостей внутри data." То есть для каждого нового temp_id (или старого?) даются обновленные совместимости. Скорее всего, ИИ возвращает предложения для итоговых атомов (новые temp_id), и они уже содержат корректные ссылки на другие итоговые атомы (новые temp_id). Тогда merge_of просто информационен. Это приемлемо, но нужно явно указать, что merge_of не участвует в маппинге при импорте, и поля совместимостей в data уже ссылаются на новые temp_id. Проверим.

В описании ответа постобработки: "compatibility_suggestions: объект, сопоставляющий temp_id → обновлённые поля совместимостей внутри data". Ожидается, что в deduplicated уже содержатся атомы с data, но совместимости могут быть неполны или требуют корректировки. Поэтому применяются compatibility_suggestions к этим атомам. Атомы в deduplicated имеют temp_id, которые сервер присваивает. Значит, compatibility_suggestions ссылаются на эти новые temp_id. Тогда старые temp_id не используются в совместимостях. merge_of хранит старые temp_id справочно. Это не ломает связи. Но при импорте, когда мы заменяем временные ID внутри data на постоянные, нужно пройти по всем полям-спискам (compatible_conflicts, compatible_twists и т.д.), которые содержат temp_id, и заменить на соответствующие постоянные. Эти temp_id — новые, присутствующие в черновиках. Поэтому маппинг будет строиться по новым temp_id → постоянные. Старые temp_id, которые только в merge_of, не нужно мапить. Это логично, но надо быть внимательным, что в merge_of не будет использоваться для поиска. Ошибкой это не является, но требует уточнения. В ТЗ можно добавить примечание.

Это допустимо, но требует явного указания, что для агрегированных атомов `fragment_id` может быть `NULL`, чтобы разработчик не пытался во что бы то ни стало найти соответствие.

**4.2. Ссылки на старые temp_id в merge_of**
После замены старых черновиков новыми поле `merge_of` новых записей содержит старые `temp_id`. При импорте эти старые ID **не участвуют в замене на постоянные**, потому что соответствующие записи уже удалены. Это не ломает логику (если `merge_of` используется только как история), но может сбить с толку. Желательно добавить примечание: «Поле `merge_of` носит информационный характер и не мапится при импорте; все ссылки внутри `data` уже указывают на новые `temp_id` итоговых атомов».

**4.3. Валидация ссылок при ручном редактировании и импорте**
Когда администратор вручную редактирует `data` черновика (PUT), он может прописать в списках совместимости несуществующие `temp_id`. При импорте это приведёт к «битым» ссылкам или ошибкам. Также не описано поведение, если подтверждённый атом ссылается на другой неподтверждённый (deleted / draft).

**Рекомендация:** добавить на этапе импорта очистку ссылок (удаление ID, не найденных в карте подтверждённых атомов) или выбрасывать предупреждение.

**4.4. Стадия в прогрессе**
`GET /books/current/progress` в примере ответа возвращает `"stage": "processing"`, но `arc_processing_state.stage` содержит более детальные значения: `'fragmentation'`, `'extraction'`, `'postprocessing'`, `'awaiting_review'`. Лучше синхронизировать, чтобы фронтенд мог показывать точный этап.

**4.5. Отсутствие эндпоинта получения одного черновика**
Есть только массовый `GET /books/current/drafts`. При большом количестве атомов это неудобно, но для MVP допустимо. Стоит хотя бы зафиксировать это как сознательное решение.

**4.6. Блокировка таблиц при импорте**
Использование блокировки всей основной таблицы для генерации ID (`CONF-0001`, …) избыточно. Лучше предложить таблицу счётчиков с атомарным `UPDATE … SET next_id = next_id + 1`, что позволит обойтись без блокировок. Пункт можно оставить как рекомендацию.

**4.7. Удаление файла на диске**
В `DELETE /books/current` не упомянуто удаление plain-text файла с диска. Очевидно, нужно, но стоит прописать явно.
