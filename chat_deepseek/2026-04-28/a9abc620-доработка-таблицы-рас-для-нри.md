- Языки (родные и дополнительные).
- Особенности зрения (инфравидение, световая слепота и т.п.).
- Расовые способности/умения (например, устойчивость к магии, заклинания, дыхание).
- Типичное мировоззрение (помогает задать характер).
- Диапазоны роста/веса (для физического описания).
- Расширяемый механизм для уникальных особенностей (драконы, оборотни).

### 3. Формат `age_categories` – стандартизация категорий

Текущий JSON-формат позволяет любую структуру, но для единообразия и удобства генерации возраста NPC рекомендую фиксированный набор ключей (не все расы обязаны иметь каждую категорию):

```json
{
  "infant":   {"min": 0,   "max": 2},
  "child":    {"min": 3,   "max": 15},
  "adolescent":{"min": 16,  "max": 24},
  "young_adult":{"min": 25,  "max": 39},
  "adult":    {"min": 40,  "max": 149},
  "middle_aged":{"min":150,  "max": 239},
  "old":      {"min":240,  "max": 319},
  "venerable":{"min":320,  "max": 450}

```

Для людей категории будут сдвинуты (взрослый с 18 лет, старый с 60 и т.д.). При генерации возраста мы случайно выбираем одну из категорий (с учётом весов, если нужно) и берём случайное значение из диапазона. Для рас с очень долгой жизнью (эльфы, драконы) границы расширяются.

### 4. Моделирование смешанных рас (полуэльфы, полуорки) – роль `parent_race_ids` и `character_races`

Ваша связка `parent_race_ids` + `character_races` даёт два уровня:

- **`parent_race_ids`** – описывает происхождение расы-гибрида. Например, запись `races` для полуэльфа будет содержать `parent_race_ids = [id_эльфа, id_человека]`. Это позволяет в будущем наследовать некоторые черты (например, усреднять возрастные категории) или просто служит мета-информацией.
- **`character_races`** – определяет фактический расовый состав конкретного персонажа. Для чистокровного эльфа – одна запись (`proportion=1.0`, `is_primary=true`). Для полуэльфа – тоже одна запись (ссылка на расу «полуэльф»). Если вы хотите поддержать более сложное смешение (1/4 орка, 3/4 человека), можно добавить несколько записей с `proportion`.

**Рекомендация:** для простоты используйте стандартные расы-гибриды (полуэльф, полуорк и т.д.) как отдельные записи в `races`. `parent_race_ids` будет указывать на исходные расы. А несколько записей в `character_races` оставьте для нестандартных случаев (мастер захочет создать уникального NPC с необычным наследием). В генераторе NPC по умолчанию вы будете брать `is_primary = true` и соответствующую расу.

**Дополнительное соображение:** если вы всё же решите комбинировать на лету (без явной расы-гибрида), потребуется сложная логика усреднения параметров. В AD&D это редкость, поэтому явные расы предпочтительнее.

### 5. Расширяемый JSON для расовых способностей – `racial_abilities`

Поле `racial_abilities` (JSONB) позволяет хранить любые уникальные особенности, включая те, которых нет в стандартных колонках.

**Примеры:**

```json

    "damage": "2d10",
```

### 6. Связи с другими таблицами (влияние на генерацию NPC)

**Как расовые данные будут использоваться:**

- **Атрибуты (`attributes`):** При генерации базовых значений (например, roll 4d6) накладываются модификаторы из `stat_modifiers`.
- **Возраст:** Выбирается категория из `age_categories` (случайно или согласно распределению для NPC), затем точный возраст из диапазона.
- **Речь и черты характера:** Можно добавить неявную связь – таблицы `speech_patterns` и `traits` могут иметь поле `race_id` (или `allowed_races`), чтобы расы задавали предпочтительные паттерны. Например, для эльфов – напевная речь, высокомерие.
- **Физические особенности:** `typical_height_weight` и `size` используются для генерации внешности. Также `racial_abilities` может добавлять особые черты (чешуя, крылья).
- **Имена:** Из `common_names` – выбор случайного имени с учётом пола (если добавить разделение по полу, можно расширить).
- **Мировоззрение:** `alignment_tendencies` сужает круг возможных мировоззрений при генерации (если не задано иное).

**Рекомендация по дополнительным связям:**
- Создайте таблицу `racial_languages` (многие-ко-многим), если у вас есть отдельный справочник `languages`. Тогда вместо `races.languages` (массив строк) будет `racial_languages(race_id, language_id, is_native)`.
- Для черт характера и речевых паттернов можно сделать таблицы `racial_traits` и `racial_speech_patterns` с внешними ключами. Но для первого этапа достаточно логики на уровне приложения.

### Итоговая схема таблицы `races`

```sql
CREATE TABLE races (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL UNIQUE,
    description TEXT,
    parent_race_ids INTEGER[], -- массив ID родительских рас (NULL если нет)
    size VARCHAR(20) CHECK (size IN ('Tiny', 'Small', 'Medium', 'Large', 'Huge')),
    speed INTEGER, -- футов за раунд
    languages TEXT[], -- массив имён языков (или внешний ключ)
    vision TEXT[], -- массив, например {'Darkvision 60ft', 'Low-light vision'}
    stat_modifiers JSONB, -- {"str":0, "dex":2, "con":-1, "int":0, "wis":1, "cha":0}
    racial_abilities JSONB, -- произвольные способности
    alignment_tendencies TEXT[], -- массив мировоззрений (коды: LG, LN, LE, NG, N, NE, CG, CN, CE)
    typical_height_weight JSONB, -- {"height":{"min":150,"max":190},"weight":{"min":50,"max":80}}
    reproduction_mode VARCHAR(50),
    average_lifespan INTEGER,
    age_categories JSONB, -- структурированный словарь категорий (infant, child, adolescent, young_adult, adult, middle_aged, old, venerable)
    common_names TEXT[], -- массив типичных имён
    base_traits JSONB, -- описательные черты (например, {"ears":"pointed", "build":"slender"})
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_races_name ON races(name);
CREATE INDEX idx_races_parents ON races USING GIN(parent_race_ids);
```

**Таблица `character_races` (уже существует) – оставляем без изменений:**

```sql
CREATE TABLE character_races (
    character_id INTEGER REFERENCES characters(id) ON DELETE CASCADE,
    race_id INTEGER REFERENCES races(id),
    proportion NUMERIC(3,2) CHECK (proportion BETWEEN 0 AND 1),
    is_primary BOOLEAN DEFAULT FALSE,
    PRIMARY KEY (character_id, race_id)
);
```

## 2. Формат `age_categories`

Текущий вариант – JSON. Нужна ли стандартизация категорий? **Да, чтобы генератор возраста работал предсказуемо.** Предлагаю фиксированный список ключей для всех рас (но раса может не указывать некоторые категории, если они не применимы):

```json
{
  "infant": {"min": 0, "max": 2},
  "child": {"min": 3, "max": 12},
  "adolescent": {"min": 13, "max": 17},
  "young_adult": {"min": 18, "max": 30},
  "adult": {"min": 31, "max": 80},
  "middle_aged": {"min": 81, "max": 120},
  "old": {"min": 121, "max": 160},
  "venerable": {"min": 161, "max": 200}
}
```

Для эльфов диапазоны будут сдвинуты (например, `"adult": {"min": 80, "max": 300}`). Для драконов – свои огромные числа. Это единый формат, который легко парсить. При генерации возраста для NPC выбирается случайная категория (можно с весами) и внутри неё случайное число.

**Вопрос:** нужны ли веса для категорий (например, молодых взрослых больше)? Можно добавить в каждую категорию поле `weight`, но это усложнение. Пока оставим только `min`/`max`.

## 3. Моделирование смешанных рас

Связка `parent_race_ids` (в таблице `races` для записи о полукровке) + `character_races` (доли крови у персонажа) – **вполне достаточно**.

- **Явная смешанная раса** (полуэльф, полуорк) создаётся как отдельная запись в `races`, где `parent_race_ids = [id_эльфа, id_человека]`. Тогда все её описательные параметры (имена, возрастные категории, средняя продолжительность жизни, рост и т.д.) задаются индивидуально, наследуя логически, но без автоматического усреднения. Это просто и прозрачно.
- **Нестандартное смешение** (1/4 гнома + 3/4 человека) – через несколько записей в `character_races` с `proportion`. Тогда для генерации возраста, внешности и т.п. приложение должно усреднять или выбирать доминантную расу (`is_primary`). Однако таких кейсов будет мало, и их можно обрабатывать через логику на уровне кода, а не через схему БД.

**Вывод:** оставляем `parent_race_ids` как чисто информационное поле (для справки, кто от кого произошёл) и не делаем автоматического наследования параметров. Тогда структура остаётся простой.

## 4. Уникальные для отдельных рас поля (драконы, оборотни)

Поскольку мы вынесли все механики в отдельные таблицы, **специфические поля не нужны**. Вместо этого:

- У вас будет таблица `racial_abilities` (или `race_special_features`), где поле `ability_data` хранит JSON для любых нестандартных способностей (дыхание, смена формы, устойчивости).
- Для дракона создаются записи: `breath_weapon`, `flying_speed`, `natural_armor` и т.п.
- Для оборотня – `shapeshift`, `silver_vulnerability` и пр.

Таким образом, `races` остаётся чистой, а расширяемость обеспечивается через связанные таблицы.

## 5. Связи с другими таблицами и влияние на генерацию NPC

Даже если игровые параметры не хранятся в `races`, расы всё равно влияют на генерацию **неигровых аспектов**:

- **Имя**: из `common_names`.
- **Физическое описание**: из `typical_height_weight`, `base_traits` (например, «серебристые волосы»).
- **Возраст**: из `age_categories` и `average_lifespan`.
- **Черты характера** и **речевые паттерны** – можно создать таблицы `racial_trait_preferences` и `racial_speech_preferences`, где для каждой расы будут заданы вероятностные распределения. Но это уже за рамками текущего обсуждения `races`. Пока достаточно помнить, что при генерации NPC вы можете использовать `race_id` как фильтр для выбора подходящих черт из справочников.

**Никаких дополнительных полей в `races` для этих связей не требуется.**

### Влияние на генерацию NPC (только лорная часть)

- **Имя** – будет браться из культуры, а не из расы. Связь: персонаж → культура → список имён. Раса может влиять на доступные культуры (например, эльф не может принадлежать оркской культуре).
- **Возраст** – генерируется из `age_categories` и `average_lifespan`.
- **Внешность** – из `physical_traits`.
- **Физические особенности** – из `physical_traits` (например, «чешуя зелёного цвета»).
- **Черты характера, речь** – будут зависеть от культуры и воспитания, а не от расы напрямую (хотя можно добавить предпочтения, но это позже).

### Итог по обсуждению

Предлагаю остановиться на этой структуре как на финальной для таблицы `races` в рамках текущего этапа. Дополнительные таблицы (`reproduction_modes`, `sex_types`) – простые справочники, создаются без сложностей. `physical_traits` – гибкий JSON, позволяющий добавлять любые признаки без изменения схемы.
